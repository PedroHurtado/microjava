## B. El núcleo ISP: jerarquía de interfaces

```
   ACCIONES DE DOMINIO              IMPLEMENTACIÓN JPA (default methods)
   (qué se puede hacer)            (cómo se hace sobre Spring Data JPA)

   IAdd<T>                          IAddJpa<T,ID,J>      ── add
                                       extiende IRepositoryJpa + IAdd

   IGet<T,ID>                       IGetJpa<T,ID,J>      ── get
     ▲    ▲                            extiende IRepositoryJpa + IGet
     │    │
     │    └── IUpdate<T,ID>         IUpdateJpa  extiende IGetJpa + IUpdate ── update
     │                                (necesita leer antes de actualizar)
     │
     └─────── IRemove<T,ID>         IRemoveJpa  extiende IGetJpa + IRemove ── remove
                                      (necesita leer antes de borrar)

   IRepositoryJpa<T,ID,J>  →  jpa()  ·  mapper()  ·  domainType()
                               (los puntos variables que cablea el repo)
```

```
   ¿Por qué IRemove e IUpdate extienden IGet, pero IAdd no?

      Para BORRAR    → primero hay que comprobar que EXISTE  → necesita Get
      Para ACTUALIZAR → primero hay que comprobar que EXISTE  → necesita Get
      Para AÑADIR     → es algo NUEVO, no hay nada que leer    → NO necesita Get

   La herencia de interfaces ESCRIBE esa regla en el sistema de tipos.
```

## C. Cómo se compone un repositorio concreto

```
   PizzaRepository
   │
   ├── implementa las VISTAS (solo dominio, lo que expone a la app):
   │       IAddPizza · IGetPizza · IUpdatePizza · IRemovePizza
   │
   ├── implementa las *Jpa (de donde HEREDA la implementación):
   │       IAddJpa · IUpdateJpa · IRemoveJpa
   │
   ├── cablea los puntos variables:
   │       jpa()        → PizzaJpaRepository
   │       mapper()     → PizzaMapper
   │       domainType() → Pizza.class
   │
   └── SOBRESCRIBE solo get:
           findByIdWithIngredients (JOIN FETCH)   ← carga el agregado completo

   add / update / remove  →  NO se escriben: vienen del default method.
```

## E. El problema de datos distribuidos y sus soluciones

```
   database per service  ──▶  NO hay transacción ACID entre servicios
                                        │
                  ┌─────────────────────┴─────────────────────┐
                  ▼                                            ▼
            2PC (clásico)                          consistencia eventual
            ✗ bloqueante                           ✓ estados intermedios
            ✗ punto único de fallo                   que CONVERGEN
            ✗ acopla servicios                       │
            → se EVITA                  ┌────────────┴────────────┐
                                        ▼                         ▼
                                   SAGA (6.7)               OUTBOX (6.8)
                                   flujo + compensaciones   no perder eventos
```

## F. Saga por coreografía: flujo feliz y compensación

```
   FLUJO FELIZ
   Ordering ──OrderCreated──▶ Catalog ──StockReserved──▶ Payments ──PaymentCompleted──▶ Ordering
   (Pendiente)                (reserva)                  (cobra)                        (Confirmado)

   FALLO + COMPENSACIÓN (orden inverso)
   Payments ──PaymentFailed──┬──▶ Catalog:  libera stock      (compensa paso 2)
                             └──▶ Ordering: Order = Cancelado (compensa paso 1)

   Regla: cada paso es una transacción LOCAL + un EVENTO; cada paso debe ser IDEMPOTENTE.
```

---

## G. Outbox: una sola transacción, publicación reintentable

```
   ┌──── misma transacción ACID local (@Transactional) ────┐
   │  INSERT orders        (el dato)                         │   →  COMMIT (ambos o ninguno)
   │  INSERT outbox        (el evento)                       │
   └─────────────────────────────────────────────────────────┘
                      │
                      ▼  (worker @Scheduled, reintentable)
              OutboxPublisher  ── lee filas no publicadas ──▶  Event Bus  ──▶  Saga

   Sin Outbox:   add(); publish();   ← si cae en medio, evento perdido o inventado
   Con Outbox:   todo en una transacción;   ← el evento no se puede perder
```

## H. Las piezas de Spring usadas en el módulo

| Concepto | Pieza de Spring / JPA |
|---|---|
| Acción atómica | `IAdd`, `IGet`, `IRemove`, `IUpdate` (una interfaz por operación) |
| Implementación reutilizable | `default methods` en las interfaces `*Jpa` |
| Punto variable de datos | `JpaRepository<J, ID>` |
| Leer por id | `findById` (`Optional`) |
| Guardar | `save` |
| Borrar | `deleteById` |
| Cargar agregado completo | `JOIN FETCH` en una `@Query` (`findByIdWithIngredients`) |
| Relación N:M | `@ManyToMany` + `@JoinTable` |
| Hidratar sin eventos | subclase `Hydration` (ctor protected) |
| No encontrado | lanza `EntityNotFoundException` → 404 vía `@RestControllerAdvice` |
| Registro de beans | `@Component` / `@Service` / `@Repository` (component scanning) |
| Bus de eventos en memoria | `ApplicationEventPublisher` + `@EventListener` |
| Worker de Outbox | `@Scheduled` + `@Transactional` (`@EnableScheduling`) |

## Puntos clave del módulo (síntesis)

- **Cada servicio, dueño de sus datos.** El acceso pasa por su API; nunca por su BD directamente.
- **ISP = una interfaz por operación**, compuesta a demanda. La herencia entre interfaces
  **codifica precondiciones** de negocio.
- **Default methods** dan reutilización sin herencia de clases: el repositorio **compone** las
  acciones que necesita.
- **Dominio puro:** clase de persistencia separada + mapper manual + `Hydration` para reconstruir
  sin emitir eventos.
- Sin transacción ACID global: **consistencia eventual** con **Saga** (flujo + compensaciones) y
  **Outbox** (entrega fiable), reforzada con **idempotencia**.
