# Módulo 3.4 — Validación de entrada (FluentValidation)

> **Duración:** ~60 minutos  
> **Contexto:** Implementar validaciones robustas en catalog-api  
> **Hilo conductor:** ShopMicro - Garantizar integridad de datos

---

## ¿Por qué FluentValidation?

### El problema: Validación con anotaciones (Jakarta Bean Validation)

```java
public record CreateProductRequest(
    @NotBlank(message = "El nombre es obligatorio")
    @Size(min = 3, max = 100, message = "El nombre debe tener entre 3 y 100 caracteres")
    String name,

    @NotNull(message = "El precio es obligatorio")
    @DecimalMin(value = "0.01", message = "El precio debe ser mayor a 0")
    BigDecimal price,

    @PositiveOrZero(message = "El stock no puede ser negativo")
    int stock
) { }
```

**Problemas:**
1. Lógica de validación acoplada al DTO
2. Difícil de reutilizar
3. No permite validaciones complejas (cross-field) sin anotaciones custom
4. Difícil de testear de forma aislada
5. Mensajes de error en anotaciones (mezcla de responsabilidades)

> Jakarta Bean Validation (Hibernate Validator) es la opción **nativa** de Spring. Es
> válida y muy usada, pero en este módulo aplicamos **FluentValidation** por su API
> declarativa desacoplada, equivalente a la del material original.

### Ventajas de FluentValidation

✅ Validación desacoplada del modelo  
✅ API fluida y legible  
✅ Soporta validaciones que consultan dependencias (BD)  
✅ Validaciones complejas y cross-field  
✅ Reutilizable en múltiples contextos  
✅ Fácil de testear  
✅ Mensajes de error localizables  

---

## Setup

```xml
<dependency>
    <groupId>com.github.fluentvalidation</groupId>
    <artifactId>fluentvalidation</artifactId>
    <version>1.x</version>
</dependency>
```

> Usamos el estilo `AbstractValidator<T>` con `ruleFor(...)`, idéntico al de .NET. Cada
> validador es un bean `@Component` para poder inyectar dependencias (repositorios).

## Validador básico

### CreateProductValidator.java

```java
package com.shopmicro.catalog.features.products.create;

import com.shopmicro.catalog.core.fluentvalidation.AbstractValidator;
import org.springframework.stereotype.Component;

import java.math.BigDecimal;

@Component
public class CreateProductValidator extends AbstractValidator<CreateProduct.Command> {

    public CreateProductValidator() {
        ruleFor(CreateProduct.Command::name)
            .notEmpty().withMessage("El nombre es obligatorio")
            .maximumLength(100).withMessage("El nombre no puede exceder 100 caracteres")
            .minimumLength(3).withMessage("El nombre debe tener al menos 3 caracteres");

        ruleFor(CreateProduct.Command::description)
            .notEmpty().withMessage("La descripción es obligatoria")
            .minimumLength(10).withMessage("La descripción debe tener al menos 10 caracteres");

        ruleFor(CreateProduct.Command::price)
            .greaterThan(BigDecimal.ZERO).withMessage("El precio debe ser mayor a 0")
            .lessThanOrEqualTo(new BigDecimal("999999.99"))
                .withMessage("El precio no puede exceder 999999.99");

        ruleFor(CreateProduct.Command::stock)
            .greaterThanOrEqualTo(0).withMessage("El stock no puede ser negativo")
            .lessThanOrEqualTo(999999).withMessage("El stock no puede exceder 999999");
    }
}
```

> En Java pasamos una **referencia a método** (`CreateProduct.Command::name`) como
> selector de la propiedad, en lugar de la expresión lambda `x => x.Name` de C#.

## Validaciones complejas

### Validaciones condicionales

```java
@Component
public class UpdateProductValidator extends AbstractValidator<UpdateProduct.Command> {

    public UpdateProductValidator() {
        ruleFor(UpdateProduct.Command::name)
            .notEmpty()
            .when(cmd -> cmd.name() != null && !cmd.name().isBlank())
            .withMessage("Si proporciona un nombre, no puede estar vacío");

        ruleFor(UpdateProduct.Command::price)
            .greaterThan(BigDecimal.ZERO)
            .when(cmd -> cmd.price() != null)
            .withMessage("Si actualiza el precio, debe ser mayor a 0");
    }
}
```

### Validaciones cross-field

```java
@Component
public class CreateOrderValidator extends AbstractValidator<CreateOrder.Command> {

    public CreateOrderValidator() {
        ruleFor(cmd -> cmd)   // valida el objeto completo
            .must(CreateOrderValidator::beValidOrderDates)
            .withMessage("La fecha de entrega debe ser posterior a la fecha de creación");

        ruleFor(CreateOrder.Command::items)
            .notEmpty().withMessage("La orden debe contener al menos un artículo")
            .must(CreateOrderValidator::haveValidQuantities)
                .withMessage("Todos los artículos deben tener cantidad > 0");
    }

    private static boolean beValidOrderDates(CreateOrder.Command command) {
        return command.deliveryDate().isAfter(Instant.now().plus(1, ChronoUnit.DAYS));
    }

    private static boolean haveValidQuantities(List<OrderItem> items) {
        return items.stream().allMatch(i -> i.quantity() > 0);
    }
}
```

### Validaciones contra la base de datos (unicidad)

```java
@Component
public class CreateProductValidator extends AbstractValidator<CreateProduct.Command> {

    private final ProductRepository repository;

    public CreateProductValidator(ProductRepository repository) {
        this.repository = repository;

        ruleFor(CreateProduct.Command::name)
            .notEmpty()
            .must(this::beUniqueProductName)
            .withMessage("Ya existe un producto con ese nombre");
    }

    private boolean beUniqueProductName(String name) {
        return !repository.existsByName(name);
    }
}
```

> En .NET FluentValidation ofrece `MustAsync` con `CancellationToken`. En Spring Boot el
> acceso a datos suele ser **síncrono y bloqueante** (JDBC/JPA), por lo que basta con
> `must(...)` consultando el repositorio. Si trabajas con stack reactivo (WebFlux + R2DBC),
> existen variantes asíncronas.

## Integración con PipelinR

### Middleware para validación automática

```java
package com.shopmicro.catalog.core.behaviors;

import an.awesome.pipelinr.Command;
import com.shopmicro.catalog.core.fluentvalidation.AbstractValidator;
import com.shopmicro.catalog.core.fluentvalidation.ValidationException;
import com.shopmicro.catalog.core.fluentvalidation.ValidationFailure;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import org.springframework.beans.factory.ObjectProvider;
import org.springframework.core.ResolvableType;
import org.springframework.stereotype.Component;

import java.util.List;

@Component
public class ValidationMiddleware implements Command.Middleware {

    private static final Logger log = LoggerFactory.getLogger(ValidationMiddleware.class);

    private final ObjectProvider<AbstractValidator<?>> validators;

    public ValidationMiddleware(ObjectProvider<AbstractValidator<?>> validators) {
        this.validators = validators;
    }

    @Override
    @SuppressWarnings("unchecked")
    public <R, C extends Command<R>> R invoke(C command, Next<R> next) {
        List<ValidationFailure> failures = validators.stream()
            .filter(v -> supports(v, command))
            .map(v -> (AbstractValidator<C>) v)
            .flatMap(v -> v.validate(command).stream())
            .toList();

        if (!failures.isEmpty()) {
            log.warn("Validation failed for {} with {} errors",
                command.getClass().getSimpleName(), failures.size());
            throw new ValidationException(failures);
        }

        return next.invoke();
    }

    private boolean supports(AbstractValidator<?> validator, Object command) {
        Class<?> validated = ResolvableType
            .forClass(AbstractValidator.class, validator.getClass())
            .getGeneric(0).resolve();
        return validated != null && validated.isInstance(command);
    }
}
```

> **Registro:** en Spring no hace falta `AddTransient(typeof(IPipelineBehavior<,>)...)`.
> Basta con que `ValidationMiddleware` y los validadores sean beans `@Component`;
> PipelinR los descubre a través del `ObjectProvider` configurado en el `Pipeline`
> (ver Módulo 3.2).

## Manejo de excepciones de validación

La `ValidationException` se transforma en `ProblemDetail` (422) en el manejador global
con `@RestControllerAdvice`.

### GlobalExceptionHandler.java (fragmento)

```java
package com.shopmicro.catalog.infrastructure.web;

import com.shopmicro.catalog.core.fluentvalidation.ValidationException;
import com.shopmicro.catalog.core.fluentvalidation.ValidationFailure;
import jakarta.servlet.http.HttpServletRequest;
import org.springframework.http.HttpStatus;
import org.springframework.http.ProblemDetail;
import org.springframework.web.bind.annotation.ExceptionHandler;
import org.springframework.web.bind.annotation.RestControllerAdvice;

import java.net.URI;
import java.util.List;
import java.util.Map;
import java.util.stream.Collectors;

@RestControllerAdvice
public class ValidationExceptionHandler {

    @ExceptionHandler(ValidationException.class)
    public ProblemDetail handle(ValidationException ex, HttpServletRequest request) {
        ProblemDetail problem = ProblemDetail.forStatusAndDetail(
            HttpStatus.UNPROCESSABLE_ENTITY, "One or more validation errors occurred.");
        problem.setType(URI.create("https://api.shopmicro.local/errors/validation-error"));
        problem.setTitle("Validation Error");
        problem.setInstance(URI.create(request.getRequestURI()));

        Map<String, List<String>> errors = ex.getFailures().stream()
            .collect(Collectors.groupingBy(
                ValidationFailure::propertyName,
                Collectors.mapping(ValidationFailure::errorMessage, Collectors.toList())
            ));

        problem.setProperty("errors", errors);
        return problem;
    }
}
```

> `problem.setProperty("errors", errors)` añade un campo de extensión al JSON RFC 7807,
> equivalente a `problemDetails.Extensions["errors"]` en .NET.

## Validadores reutilizables

### Custom Validators

En .NET se escriben como métodos de extensión sobre `IRuleBuilder`. En Java los
exponemos como métodos sobre nuestro builder de reglas (o helpers estáticos que aplican
varias reglas encadenadas).

```java
package com.shopmicro.catalog.core.validators;

import com.shopmicro.catalog.core.fluentvalidation.RuleBuilder;

import java.math.BigDecimal;

public final class CustomValidators {

    private CustomValidators() { }

    public static RuleBuilder<?, String> beValidSkuFormat(RuleBuilder<?, String> rule) {
        return rule
            .matches("^[A-Z]{3}-\\d{4}$")
            .withMessage("El SKU debe tener formato: ABC-1234 "
                + "(3 letras mayúsculas, guión, 4 dígitos)");
    }

    public static RuleBuilder<?, BigDecimal> beValidPrice(RuleBuilder<?, BigDecimal> rule) {
        return rule
            .greaterThan(BigDecimal.ZERO)
                .withMessage("El precio debe ser mayor a 0")
            .lessThanOrEqualTo(new BigDecimal("999999.99"))
                .withMessage("El precio no puede exceder 999999.99")
            .scalePrecision(2, 8)
                .withMessage("El precio no puede tener más de 2 decimales");
    }
}
```

**Usar:**

```java
@Component
public class CreateProductValidator extends AbstractValidator<CreateProduct.Command> {

    public CreateProductValidator() {
        CustomValidators.beValidSkuFormat(ruleFor(CreateProduct.Command::sku));
        CustomValidators.beValidPrice(ruleFor(CreateProduct.Command::price));
    }
}
```

## Ejemplo completo: Validador con reglas complejas

```java
package com.shopmicro.catalog.features.products.create;

import com.shopmicro.catalog.core.fluentvalidation.AbstractValidator;
import com.shopmicro.catalog.core.fluentvalidation.CascadeMode;
import com.shopmicro.catalog.features.products.domain.ProductRepository;
import org.springframework.stereotype.Component;

import java.math.BigDecimal;
import java.util.Set;

@Component
public class CreateProductValidator extends AbstractValidator<CreateProduct.Command> {

    private static final Set<String> ALLOWED_CATEGORIES =
        Set.of("Electronics", "Software", "Hardware");

    private final ProductRepository repository;

    public CreateProductValidator(ProductRepository repository) {
        this.repository = repository;

        ruleFor(CreateProduct.Command::name)
            .cascade(CascadeMode.STOP)   // Detener en el primer error
            .notEmpty().withMessage("El nombre es obligatorio")
            .length(3, 100).withMessage("El nombre debe tener entre 3 y 100 caracteres")
            .matches("^[a-zA-Z0-9\\s\\-]+$")
                .withMessage("El nombre solo puede contener letras, números, espacios y guiones")
            .must(this::beUniqueProductName)
                .withMessage("Ya existe un producto con ese nombre");

        ruleFor(CreateProduct.Command::description)
            .notEmpty().withMessage("La descripción es obligatoria")
            .minimumLength(10).withMessage("La descripción debe tener al menos 10 caracteres")
            .maximumLength(500).withMessage("La descripción no puede exceder 500 caracteres");

        ruleFor(CreateProduct.Command::price)
            .notNull()
            .greaterThan(BigDecimal.ZERO).withMessage("El precio debe ser mayor a 0")
            .lessThanOrEqualTo(new BigDecimal("999999.99"))
                .withMessage("El precio no puede exceder 999999.99")
            .scalePrecision(2, 8).withMessage("El precio no puede tener más de 2 decimales");

        ruleFor(CreateProduct.Command::stock)
            .greaterThanOrEqualTo(0).withMessage("El stock no puede ser negativo")
            .lessThanOrEqualTo(999999).withMessage("El stock no puede exceder 999999");

        // Regla condicional
        ruleFor(CreateProduct.Command::price)
            .lessThan(new BigDecimal("100"))
            .when(cmd -> cmd.stock() < 10)
            .withMessage("Productos con stock bajo deben tener precio menor a 100");
    }

    private boolean beUniqueProductName(String name) {
        return !repository.existsByName(name);
    }
}
```

## Testing de validadores

```java
import com.shopmicro.catalog.features.products.create.CreateProduct;
import com.shopmicro.catalog.features.products.create.CreateProductValidator;
import org.junit.jupiter.api.BeforeEach;
import org.junit.jupiter.api.Test;

import java.math.BigDecimal;

import static org.assertj.core.api.Assertions.assertThat;
import static org.mockito.Mockito.mock;
import static org.mockito.Mockito.when;

class CreateProductValidatorTest {

    private CreateProductValidator validator;
    private ProductRepository repository;

    @BeforeEach
    void setUp() {
        repository = mock(ProductRepository.class);
        when(repository.existsByName(org.mockito.ArgumentMatchers.anyString())).thenReturn(false);
        validator = new CreateProductValidator(repository);
    }

    @Test
    void shouldFailWhenNameIsEmpty() {
        var command = new CreateProduct.Command("", "Description", new BigDecimal("99.99"), 10);
        var result = validator.validate(command);
        assertThat(result).anyMatch(f -> f.propertyName().equals("name"));
    }

    @Test
    void shouldFailWhenPriceIsZero() {
        var command = new CreateProduct.Command("Product", "A description here", BigDecimal.ZERO, 10);
        var result = validator.validate(command);
        assertThat(result)
            .anyMatch(f -> f.propertyName().equals("price")
                && f.errorMessage().equals("El precio debe ser mayor a 0"));
    }

    @Test
    void shouldPassWhenDataIsValid() {
        var command = new CreateProduct.Command(
            "Valid Product", "A valid description", new BigDecimal("99.99"), 10);
        var result = validator.validate(command);
        assertThat(result).isEmpty();
    }
}
```

> Equivalencias de testing:
> - `Xunit [Fact]` → JUnit 5 `@Test`
> - `Assert.*` / `ShouldHaveValidationErrorFor` → AssertJ `assertThat(...)`
> - `MockProductRepository` → Mockito `mock(...)`

## Cascade Mode

Controla cuándo se detiene la validación:

```java
@Component
public class CreateProductValidator extends AbstractValidator<CreateProduct.Command> {

    public CreateProductValidator() {
        // STOP detiene en el primer error de la regla
        ruleFor(CreateProduct.Command::name)
            .cascade(CascadeMode.STOP)
            .notEmpty()
            .length(3, 100)
            .matches("^[a-zA-Z0-9]+$");

        // CONTINUE valida todas las reglas (default)
        ruleFor(CreateProduct.Command::price)
            .cascade(CascadeMode.CONTINUE)
            .greaterThan(BigDecimal.ZERO)
            .lessThanOrEqualTo(new BigDecimal("999999.99"));
    }
}
```

## Localización de mensajes

En Spring lo idiomático es usar un `MessageSource` con archivos `messages_es.properties`,
`messages_en.properties`, etc.

### application.yml

```yaml
spring:
  messages:
    basename: messages
    encoding: UTF-8
```

### src/main/resources/messages_es.properties

```properties
validation.notEmpty=El campo {0} es obligatorio
validation.length=El campo {0} debe tener entre {1} y {2} caracteres
validation.greaterThan=El campo {0} debe ser mayor a {1}
```

### Uso desde el validador

```java
@Component
public class CreateProductValidator extends AbstractValidator<CreateProduct.Command> {

    public CreateProductValidator(MessageSource messages) {
        Locale locale = LocaleContextHolder.getLocale();

        ruleFor(CreateProduct.Command::name)
            .notEmpty()
            .withMessage(messages.getMessage(
                "validation.notEmpty", new Object[] {"name"}, locale));
    }
}
```

> `ValidatorOptions.Global.ErrorMessageSource` de .NET equivale a configurar el
> `MessageSource` de Spring + `LocaleContextHolder` para resolver el idioma de la
> petición (cabecera `Accept-Language`).

## Puntos clave

- **FluentValidation:** Desacopla validaciones del modelo
- **Acceso a BD:** Inyecta el repositorio en el validador para verificar unicidad
- **PipelinR Middleware:** Inyecta validación automáticamente en el pipeline
- **Validaciones complejas:** Soporta cross-field y condicionales (`when`)
- **Reutilización:** Custom validators para reglas comunes
- **Testing:** Fácil de testear con JUnit + AssertJ + Mockito
- **Mensajes localizables:** `MessageSource` + `messages_*.properties`
- **Cascade Mode:** Controla cuándo detener la validación
