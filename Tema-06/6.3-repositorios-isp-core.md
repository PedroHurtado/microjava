# 6.3 — Repositorios con ISP: el núcleo reutilizable

> **Módulo 6 · Lección 3 de 9** · ★ **Núcleo del módulo**
> **Duración estimada:** ~45 min
> **Tipo:** Teoría + código

---

## La idea en una frase

> En lugar de un `Repository<T>` gigante con `add`, `get`, `update`, `remove`, `getAll`,
> `exists`… que **todos** los repositorios heredan aunque no lo usen, definimos **una interfaz por
> operación** y cada repositorio **compone solo las que necesita**.

Eso es el **Principio de Segregación de Interfaces (ISP)**, la "I" de SOLID:

> *"Ningún cliente debería verse obligado a depender de métodos que no usa."*

Y vamos a ver que, lejos de complicar nada, **simplifica** la capa de datos y la hace honesta:
un repositorio que solo sabe añadir y leer **declara exactamente eso**, ni un método más.

---

## El error que estamos evitando

El repositorio genérico clásico:

```java
// ❌ El "God repository": todos lo heredan, casi nadie lo usa entero
public interface Repository<T, ID> {
    void add(T entity);
    T get(ID id);
    void update(T entity);
    void remove(ID id);
    List<T> getAll();
    boolean exists(ID id);
    // ... y sigue creciendo
}
```

Problemas:

- Un repositorio de "solo lectura" se ve **obligado a implementar** `add`, `update`,
  `remove` (típicamente lanzando `UnsupportedOperationException` — un olor a diseño).
- El cliente que solo necesita leer **depende** de toda la superficie de escritura.
- La interfaz **nunca deja de crecer**, y cada método nuevo afecta a todos los implementadores.

---

## La alternativa: cuatro acciones atómicas

Definimos cuatro interfaces, **una operación cada una**, en el núcleo del servicio
(`core/repository`):

```java
package com.shopmicro.core.repository;

/** Acción de añadir. Añadir no requiere leer, por eso NO extiende IGet. */
public interface IAdd<T> {
    void add(T entity);
}
```

```java
package com.shopmicro.core.repository;

/**
 * Acción de leer por id. {@code get} devuelve {@code T} o lanza
 * {@link com.shopmicro.core.domain.EntityNotFoundException}; nunca devuelve null ni un Optional.
 */
public interface IGet<T, ID> {
    T get(ID id);
}
```

```java
package com.shopmicro.core.repository;

/** Acción de borrar. Extiende IGet: para borrar, primero hay que garantizar que existe. */
public interface IRemove<T, ID> extends IGet<T, ID> {
    void remove(ID id);
}
```

```java
package com.shopmicro.core.repository;

/** Acción de actualizar. Extiende IGet: para actualizar, primero hay que leer. */
public interface IUpdate<T, ID> extends IGet<T, ID> {
    void update(T entity);
}
```

### Las dependencias entre interfaces no son arbitrarias

Fíjate en **quién extiende a quién** — es una decisión de diseño que codifica reglas de negocio:

```
        IAdd<T>          (añadir no necesita leer → independiente)

        IGet<T,ID>       (leer es la operación base)
          ▲   ▲
          │   │
  IRemove │   │ IUpdate  (borrar y actualizar EXIGEN leer primero:
          │   │           hay que garantizar que la entidad existe)
```

- **`IAdd` no extiende `IGet`**: para añadir una entidad nueva no hace falta leer nada.
- **`IRemove` e `IUpdate` extienden `IGet`**: no puedes borrar ni actualizar algo cuya existencia
  no has comprobado. La herencia de interfaces **expresa esa precondición** en el sistema de tipos.

## El contrato que conecta dominio e infraestructura

Hasta aquí, las interfaces solo hablan de **dominio** (`T`, `ID`). Para implementarlas sobre Spring
Data JPA necesitamos dos cosas: el `JpaRepository` concreto y un **mapper** dominio ↔ persistencia.
Las declaramos en un contrato común:

```java
package com.shopmicro.core.repository;

/**
 * Contrato común de los mappers dominio ↔ JPA. Permite que las interfaces base
 * mapeen genéricamente sin conocer el tipo concreto.
 *
 * @param <T> tipo de dominio
 * @param <J> tipo de persistencia (entidad JPA)
 */
public interface Mapper<T, J> {
    J toJpa(T domain);
    T toDomain(J jpa);
}
```

```java
package com.shopmicro.core.repository;

import com.shopmicro.core.domain.EntityBase;
import org.springframework.data.jpa.repository.JpaRepository;

/**
 * Base "sin herencia de clases" de los repositorios JPA. Aporta los tres únicos puntos
 * variables que cada repositorio concreto debe cablear: el JpaRepository, el mapper y
 * el tipo de dominio (para construir la excepción de "no encontrado").
 *
 * @param <T>  tipo de dominio (extiende EntityBase para extraer el id)
 * @param <ID> tipo del id
 * @param <J>  tipo de persistencia
 */
public interface IRepositoryJpa<T extends EntityBase<ID>, ID, J> {
    JpaRepository<J, ID> jpa();   // save / findById / deleteById
    Mapper<T, J> mapper();        // dominio ↔ JPA
    Class<T> domainType();        // typeo de dominio para la excepción 404
}
```

> **El punto variable de datos.** Toda la dependencia con la persistencia queda en un único
> `JpaRepository<J, ID>`: da las consultas (`findById`, queries derivadas) y el commit (`save`,
> `deleteById` dentro de la transacción). Queda **encapsulado** en esta interfaz.
> `domainType()` hace falta porque en Java los genéricos se **borran** en tiempo de ejecución
> (*type erasure*): no se puede hacer `T.class` dentro de un método; el repositorio concreto lo
> aporta devolviendo, por ejemplo, `Pizza.class`.

## La pieza clave de Java: *default methods*

Desde **Java 8**, una interfaz puede aportar **implementación por defecto** de sus métodos
(`default`). Es lo que nos permite **escribir la lógica de cada operación una sola vez** y reutilizarla
por composición, **sin herencia de clases**.

Una interfaz `*Jpa` por cada acción, que **compone** `IRepositoryJpa` + la acción correspondiente e
implementa el método como `default`:

```java
package com.shopmicro.core.repository;

import com.shopmicro.core.domain.EntityBase;

/** Implementa IAdd.add como default, componiendo IRepositoryJpa + IAdd. */
public interface IAddJpa<T extends EntityBase<ID>, ID, J>
        extends IRepositoryJpa<T, ID, J>, IAdd<T> {

    @Override
    default void add(T entity) {
        jpa().save(mapper().toJpa(entity));
    }
}
```

```java
package com.shopmicro.core.repository;

import com.shopmicro.core.domain.EntityBase;
import com.shopmicro.core.domain.EntityNotFoundException;

/** Implementa IGet.get como default: mapea a dominio o lanza si no existe. */
public interface IGetJpa<T extends EntityBase<ID>, ID, J>
        extends IRepositoryJpa<T, ID, J>, IGet<T, ID> {

    @Override
    default T get(ID id) {
        J jpa = jpa().findById(id)
            .orElseThrow(() -> new EntityNotFoundException(domainType(), id));
        return mapper().toDomain(jpa);
    }
}
```

```java
package com.shopmicro.core.repository;

import com.shopmicro.core.domain.EntityBase;

/**
 * Implementa IRemove.remove como default. Extiende IGetJpa para garantizar
 * existencia (y lanzar si no) antes del borrado.
 */
public interface IRemoveJpa<T extends EntityBase<ID>, ID, J>
        extends IGetJpa<T, ID, J>, IRemove<T, ID> {

    @Override
    default void remove(ID id) {
        get(id);                 // lanza EntityNotFoundException si no existe
        jpa().deleteById(id);
    }
}
```

```java
package com.shopmicro.core.repository;

import com.shopmicro.core.domain.EntityBase;

/**
 * Implementa IUpdate.update como default. Extiende IGetJpa para reutilizar su
 * get (garantiza existencia antes de guardar).
 */
public interface IUpdateJpa<T extends EntityBase<ID>, ID, J>
        extends IGetJpa<T, ID, J>, IUpdate<T, ID> {

    @Override
    default void update(T entity) {
        get(entity.getId());     // lanza EntityNotFoundException si no existe
        jpa().save(mapper().toJpa(entity));
    }
}
```

> **Detalle de Java.** El `default` de una interfaz puede implementar un método declarado en *otra*
> interfaz heredada (aquí, el `add` de `IAdd`, el `get` de `IGet`…). La lógica vive en la interfaz, no
> en una clase base: por eso un repositorio puede **componer** las acciones que necesite sin extender
> ninguna clase.

## EntityBase y la excepción

Dos piezas de apoyo en el núcleo del dominio (`core/domain`):

```java
package com.shopmicro.core.domain;

/** Raíz de las entidades de dominio. Expone el id para las operaciones genéricas. */
public abstract class EntityBase<ID> {

    private ID id;

    public ID getId() {
        return id;
    }

    protected void setId(ID id) {
        this.id = id;
    }
}
```

```java
package com.shopmicro.core.domain;

/** Se lanza cuando una entidad no existe. La capa de errores la traduce a 404. */
public class EntityNotFoundException extends RuntimeException {

    public EntityNotFoundException(Class<?> domainType, Object id) {
        super(domainType.getSimpleName() + " con id '" + id + "' no encontrado.");
    }
}
```

> Esto encaja con el Módulo 3: `EntityNotFoundException` se mapea a un `ProblemDetail` con
> `status: 404` en el manejador global de errores (`@RestControllerAdvice`). La capa de datos
> **lanza**, la capa web **traduce**.

## Lo que hemos ganado

```
   Repositorio que solo lee:        implements IGetJpa
   Repositorio de solo escritura:   implements IAddJpa
   Repositorio CRUD completo:       implements IAddJpa, IGetJpa, IUpdateJpa, IRemoveJpa
```

Cada repositorio **declara en su firma exactamente lo que sabe hacer**. No hay métodos heredados
que lanzan `UnsupportedOperationException`. No hay un cliente acoplado a operaciones que no usa. Y la
**implementación se escribió una sola vez**, en las interfaces `*Jpa`.

---

## Puntos clave

- **ISP = una interfaz por operación.** `IAdd`, `IGet`, `IRemove`, `IUpdate`.
- La **herencia entre interfaces expresa precondiciones**: `IRemove`/`IUpdate` extienden `IGet`
  porque exigen que la entidad exista; `IAdd` no.
- Los **default methods de Java** (≥ Java 8) dan implementación reutilizable **sin herencia de
  clases**: un repositorio compone las acciones que necesita.
- `IRepositoryJpa` aísla los **puntos variables**: el `JpaRepository`, el mapper y el `domainType()`
  (necesario por el *type erasure* de Java).
- `get` **devuelve `T` o lanza** — nunca `null` ni `Optional`. La capa web traduce la excepción a 404.
