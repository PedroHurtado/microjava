> **Hilo conductor:** ShopMicro - Separación de responsabilidades

## ¿Por qué DTOs?

**DTO** (*Data Transfer Object*) es un patrón que encapsula datos para transferencia entre capas o servicios.

### El problema sin DTOs

```java
// Endpoint recibe y devuelve la entidad de dominio directamente
@PostMapping
public ResponseEntity<Product> create(@RequestBody Product product) {
    // ⚠️ Problema 1: Expones la entidad de dominio
    // ⚠️ Problema 2: El cliente puede modificar campos que no debería (ej: createdAt)
    // ⚠️ Problema 3: Cambios en Product rompen clientes
    repository.add(product);
    URI location = URI.create("/api/v1/products/" + product.getId());
    return ResponseEntity.created(location).body(product);
}
```

### Ventajas de los DTOs

1. **Contrato explícito:** El cliente sabe exactamente qué campos puede enviar/recibir
2. **Flexibilidad:** Cambios en la entidad no afectan clientes
3. **Seguridad:** No expongas campos sensibles (ej: `internalCost`, `margin`)
4. **Validación específica:** DTOs pueden tener validaciones diferentes que la entidad
5. **Performance:** Selecciona solo los campos necesarios para cada operación

## Tipos de DTOs

En Java usamos `record` para DTOs inmutables (equivalente al `record` de C#).

### 1. CreateProductRequest (DTO de entrada)

```java
package com.shopmicro.catalog.features.products.dtos;

import java.math.BigDecimal;

public record CreateProductRequest(
    String name,
    String description,
    BigDecimal price,
    int stock
) { }
```

**Usar en:** El Endpoint recibe este DTO, lo transforma en Command de dominio.

### 2. UpdateProductRequest

```java
public record UpdateProductRequest(
    String name,
    String description,
    BigDecimal price,
    int stock
) { }
```

### 3. ProductResponse (DTO de salida)

```java
public record ProductResponse(
    UUID id,
    String name,
    String description,
    BigDecimal price,
    int stock,
    Instant createdAt,
    Instant updatedAt
) { }
```

**Usar en:** El Handler devuelve este DTO para exponerlo al cliente.

### 4. ProductListItemResponse (DTO de listado optimizado)

```java
public record ProductListItemResponse(
    UUID id,
    String name,
    BigDecimal price,
    int stock
) { }
```

**Ventaja:** En un listado, no necesitamos toda la información, reducimos payload.

## Mapeo entre capas

El mapeo transforma datos entre DTOs, Commands y entidades de dominio.

### Opción 1: Mapeo manual

```java
// En el Endpoint
var command = new CreateProduct.Command(
    request.name(),
    request.description(),
    request.price(),
    request.stock()
);

var response = pipeline.send(command);

// En el Handler
Product product = Product.create(
    UUID.randomUUID(),
    command.name(),
    command.description(),
    command.price(),
    command.stock()
);
repository.add(product);

return new ProductResponse(
    product.getId(),
    product.getName(),
    product.getDescription(),
    product.getPrice(),
    product.getStock(),
    product.getCreatedAt(),
    product.getUpdatedAt()
);
```

**Ventaja:** Explícito, sin dependencias externas.  
**Desventaja:** Repetitivo y propenso a errores.

### Opción 2: Usar MapStruct (recomendado para proyectos grandes)

`MapStruct` es el equivalente Java a `AutoMapper`: genera el código de mapeo en
tiempo de compilación (sin reflexión), por lo que es rápido y type-safe.

**Configurar (`pom.xml`):**

```xml
<dependency>
    <groupId>org.mapstruct</groupId>
    <artifactId>mapstruct</artifactId>
    <version>1.6.0</version>
</dependency>
<!-- processor en el maven-compiler-plugin -->
<path>
    <groupId>org.mapstruct</groupId>
    <artifactId>mapstruct-processor</artifactId>
    <version>1.6.0</version>
</path>
```

**Definir el Mapper:**

```java
package com.shopmicro.catalog.infrastructure.mapping;

import com.shopmicro.catalog.features.products.create.CreateProduct;
import com.shopmicro.catalog.features.products.domain.Product;
import com.shopmicro.catalog.features.products.dtos.*;
import org.mapstruct.Mapper;

@Mapper(componentModel = "spring")
public interface ProductMapper {

    // Request -> Command
    CreateProduct.Command toCommand(CreateProductRequest request);

    // Product entity -> Response
    ProductResponse toResponse(Product product);

    // Product entity -> ListItem Response
    ProductListItemResponse toListItem(Product product);
}
```

> `componentModel = "spring"` hace que MapStruct genere un bean `@Component`,
> inyectable directamente en endpoints y handlers — equivalente a registrar el perfil
> de AutoMapper en `Program.cs` con `builder.Services.AddAutoMapper(...)`.

**Usar en Endpoint:**

```java
@PostMapping
public ResponseEntity<CreateProduct.Response> create(@RequestBody CreateProductRequest request) {
    var command = mapper.toCommand(request);
    var response = pipeline.send(command);
    URI location = URI.create("/api/v1/products/" + response.id());
    return ResponseEntity.created(location).body(response);
}
```

### Opción 3: Mappers específicos (métodos estáticos)

```java
package com.shopmicro.catalog.features.products.mappers;

import com.shopmicro.catalog.features.products.create.CreateProduct;
import com.shopmicro.catalog.features.products.domain.Product;
import com.shopmicro.catalog.features.products.dtos.*;

public final class ProductMappers {

    private ProductMappers() { }

    public static CreateProduct.Command toCommand(CreateProductRequest request) {
        return new CreateProduct.Command(
            request.name(), request.description(), request.price(), request.stock());
    }

    public static ProductResponse toResponse(Product product) {
        return new ProductResponse(
            product.getId(),
            product.getName(),
            product.getDescription(),
            product.getPrice(),
            product.getStock(),
            product.getCreatedAt(),
            product.getUpdatedAt()
        );
    }

    public static ProductListItemResponse toListItem(Product product) {
        return new ProductListItemResponse(
            product.getId(), product.getName(), product.getPrice(), product.getStock());
    }
}
```

**Usar:**

```java
var command = ProductMappers.toCommand(request);
var response = ProductMappers.toResponse(product);
```

**Ventaja:** Lightweight, sin reflexión.  
**Desventaja:** Requiere mantenimiento manual.

> En C# los *mappers específicos* suelen escribirse como **métodos de extensión**
> (`request.ToCommand()`). Java no tiene métodos de extensión, así que usamos
> **métodos estáticos** en una clase utilitaria.

## Ejemplo completo: GetProductById

### GetProductById.java (nested classes)

```java
package com.shopmicro.catalog.features.products.getbyid;

import an.awesome.pipelinr.Command;
import an.awesome.pipelinr.Pipeline;
import com.shopmicro.catalog.features.products.domain.ProductRepository;
import com.shopmicro.catalog.features.products.domain.exceptions.ProductNotFoundException;
import org.springframework.http.ResponseEntity;
import org.springframework.stereotype.Component;
import org.springframework.web.bind.annotation.*;

import java.math.BigDecimal;
import java.time.Instant;
import java.util.UUID;

public final class GetProductById {

    private GetProductById() { }

    // Query
    public record Query(UUID id) implements Command<Response> { }

    // Endpoint
    @RestController
    @RequestMapping("/api/v1/products")
    public static class Endpoint {

        private final Pipeline pipeline;

        public Endpoint(Pipeline pipeline) {
            this.pipeline = pipeline;
        }

        @GetMapping("/{id}")
        public ResponseEntity<Response> getById(@PathVariable UUID id) {
            Response response = pipeline.send(new Query(id));
            return ResponseEntity.ok(response);
        }
    }

    // Handler
    @Component
    public static class Handler implements Command.Handler<Query, Response> {

        private final ProductRepository repository;

        public Handler(ProductRepository repository) {
            this.repository = repository;
        }

        @Override
        public Response handle(Query query) {
            var product = repository.getById(query.id())
                .orElseThrow(() -> new ProductNotFoundException(query.id()));

            return new Response(
                product.getId(),
                product.getName(),
                product.getDescription(),
                product.getPrice(),
                product.getStock(),
                product.getCreatedAt(),
                product.getUpdatedAt()
            );
        }
    }

    // Response
    public record Response(
        UUID id,
        String name,
        String description,
        BigDecimal price,
        int stock,
        Instant createdAt,
        Instant updatedAt
    ) { }
}
```

> El `Optional<Product>` de Spring Data permite `orElseThrow(...)`, sustituyendo al
> patrón `if (product is null) throw ...` de C#.

## Excepciones de dominio

```java
package com.shopmicro.catalog.features.products.domain.exceptions;

import java.util.UUID;

public class ProductNotFoundException extends RuntimeException {
    public ProductNotFoundException(UUID id) {
        super("El producto con ID " + id + " no fue encontrado");
    }
}
```

```java
package com.shopmicro.catalog.features.products.domain.exceptions;

public class InvalidProductException extends RuntimeException {
    public InvalidProductException(String message) {
        super(message);
    }
}
```

> En Java extendemos `RuntimeException` (excepción no comprobada) para no obligar a
> declarar `throws` en cada firma, igual que en C# las excepciones no se declaran.

## Manejo global de excepciones

En .NET se usa un *middleware*. En Spring Boot el mecanismo idiomático es un
`@RestControllerAdvice` con métodos `@ExceptionHandler`, que devuelve `ProblemDetail`
(RFC 7807).

```java
package com.shopmicro.catalog.infrastructure.web;

import com.shopmicro.catalog.features.products.domain.exceptions.InvalidProductException;
import com.shopmicro.catalog.features.products.domain.exceptions.ProductNotFoundException;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import org.springframework.http.HttpStatus;
import org.springframework.http.ProblemDetail;
import org.springframework.web.bind.annotation.ExceptionHandler;
import org.springframework.web.bind.annotation.RestControllerAdvice;

@RestControllerAdvice
public class GlobalExceptionHandler {

    private static final Logger log = LoggerFactory.getLogger(GlobalExceptionHandler.class);

    @ExceptionHandler(ProductNotFoundException.class)
    public ProblemDetail handleNotFound(ProductNotFoundException ex) {
        var problem = ProblemDetail.forStatusAndDetail(HttpStatus.NOT_FOUND, ex.getMessage());
        problem.setTitle("Not Found");
        return problem;
    }

    @ExceptionHandler(InvalidProductException.class)
    public ProblemDetail handleInvalid(InvalidProductException ex) {
        var problem = ProblemDetail.forStatusAndDetail(HttpStatus.BAD_REQUEST, ex.getMessage());
        problem.setTitle("Bad Request");
        return problem;
    }

    @ExceptionHandler(Exception.class)
    public ProblemDetail handleUnexpected(Exception ex) {
        log.error("Unhandled exception occurred", ex);
        var problem = ProblemDetail.forStatusAndDetail(
            HttpStatus.INTERNAL_SERVER_ERROR, "An unexpected error occurred");
        problem.setTitle("Internal Server Error");
        return problem;
    }
}
```

> Spring serializa `ProblemDetail` con `Content-Type: application/problem+json`
> automáticamente. No hace falta registrar nada en una clase de arranque: basta con
> que la clase tenga `@RestControllerAdvice`.

## Comparativa: Mapeo manual vs MapStruct

| Aspecto | Manual | MapStruct |
|---------|--------|-----------|
| Setup | Ninguno | Anotación de procesador |
| Performance | Rápido (compilado) | Rápido (genera código en compilación) |
| Mantenibilidad | Explícito | Menos código |
| Debugging | Directo | Código generado legible |
| Escalabilidad | Tedioso en proyectos grandes | Excelente |
| Complejidad | Simple | Intermedia |

> A diferencia de AutoMapper (que usa reflexión en runtime), **MapStruct genera el
> código de mapeo en tiempo de compilación**, por lo que no penaliza el rendimiento.

**Recomendación:** Para **ShopMicro** (proyecto educativo pequeño), usa **mappers específicos**. Para proyectos en producción con 20+ entidades, usa **MapStruct**.

---

## Puntos clave

- **DTOs:** Desacopla la API del dominio, permite evolucionar independientemente
- **Request vs Response:** DTOs diferentes para entrada y salida (usa `record`)
- **Mapeo:** Elige entre manual (simple) o MapStruct (escalable)
- **Capas:** Organiza por responsabilidad, no por tipo técnico
- **Excepciones de dominio:** Comunica errores de negocio (`RuntimeException`)
- **ProblemDetail:** Estándar RFC 7807 nativo en Spring Boot 3 para respuestas de error
