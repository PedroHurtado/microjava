> **Módulo 6 · Lección 2 de 9**
> **Duración estimada:** ~25 min
> **Tipo:** Teoría + código

---

## Qué es Spring Data JPA y qué papel juega

**JPA** (Jakarta Persistence API) es la especificación de ORM de Java; **Hibernate** es su
implementación más usada y **Spring Data JPA** la capa que la envuelve para que escribas repositorios
casi sin código. En nuestra arquitectura, JPA vive **exclusivamente en la capa de infraestructura**
del servicio. El dominio no lo conoce; lo veremos con detalle en 6.3 y 6.4.

Las dos piezas que hay que entender:

- **`EntityManager`** — la unidad de trabajo y la sesión con la base de datos. Representa una
  transacción lógica: acumulas cambios y se confirman al cerrar la transacción (`@Transactional`).
  Spring Data lo maneja por ti detrás de cada `JpaRepository`.
- **`JpaRepository<J, ID>`** — una interfaz que representa el acceso a una "tabla" de entidades de
  tipo `J`. Aporta `save`, `findById`, `deleteById`, consultas derivadas, etc., sin que escribas la
  implementación.

```
┌──────────────────────────────────────────────┐
│        JpaRepository<PizzaJpa, UUID>           │
│                                                │
│   save(...)        ─────▶ INSERT / UPDATE      │
│   findById(...)    ─────▶ SELECT por id        │
│   deleteById(...)  ─────▶ DELETE               │
│                                                │
│   (la transacción la abre/cierra @Transactional)│
└──────────────────────────────────────────────┘
```

## Por qué H2 en memoria en este módulo

JPA/Hibernate soporta muchos **dialectos**: PostgreSQL, MySQL, SQL Server, Oracle… y **H2**, una base
de datos relacional ligera que puede correr **enteramente en memoria**. La elegimos aquí porque:

- **No requiere infraestructura:** ni instalar Postgres, ni contenedores, ni cadenas de conexión.
- El **código de los repositorios es idéntico** sea cual sea el motor. Solo cambian unas líneas en
  `application.properties`.
- Nos deja concentrarnos en lo que importa en este módulo: el **diseño ISP de la capa de datos**.

> ⚠️ **H2 en memoria se borra al apagar la aplicación.** Cada arranque crea el esquema vacío
> (`create-drop`). Sirve para aprender y para tests, **no para producción**. En producción se cambia
> el motor (ver más abajo).

## Configuración (H2 en memoria)

`application.properties` de Catalog.API:

```properties
# --- Persistencia: H2 en memoria ---------------------------------------
spring.datasource.url=jdbc:h2:mem:catalog;DB_CLOSE_DELAY=-1
spring.datasource.driver-class-name=org.h2.Driver
spring.datasource.username=sa
spring.datasource.password=

# Hibernate crea el esquema al arrancar y lo borra al parar
spring.jpa.hibernate.ddl-auto=create-drop
spring.jpa.properties.hibernate.dialect=org.hibernate.dialect.H2Dialect
```

El día que quieras PostgreSQL **solo cambias esas líneas**:

```properties
# spring.datasource.url=jdbc:postgresql://localhost:5432/catalog
# spring.datasource.driver-class-name=org.postgresql.Driver
# spring.jpa.properties.hibernate.dialect=org.hibernate.dialect.PostgreSQLDialect
```

El resto del servicio —repositorios, mappers, dominio, endpoints— **no se entera**.

## El repositorio Spring Data de Catalog

En Spring Data declaras **una interfaz por agregado** que extiende `JpaRepository`. No escribes la
implementación: Spring la genera en tiempo de arranque:

```java
package com.shopmicro.catalog.pizza.persistence;

import org.springframework.data.jpa.repository.JpaRepository;
import java.util.UUID;

public interface PizzaJpaRepository extends JpaRepository<PizzaJpa, UUID> {
    // consultas derivadas y @Query se añaden aquí (lo veremos en 6.4)
}
```

```java
package com.shopmicro.catalog.ingredient.persistence;

import org.springframework.data.jpa.repository.JpaRepository;
import java.util.UUID;

public interface IngredientJpaRepository extends JpaRepository<IngredientJpa, UUID> {
}
```

La **relación N:M** Pizza–Ingredient se declara con **anotaciones** directamente sobre la entidad
`PizzaJpa` (la veremos al detalle en 6.4):

```java
@ManyToMany
@JoinTable(
    name = "pizza_ingredient",
    joinColumns = @JoinColumn(name = "pizza_id"),
    inverseJoinColumns = @JoinColumn(name = "ingredient_id"))
private Set<IngredientJpa> ingredients = new HashSet<>();
```

> **Nota sobre migraciones.** Con un motor de producción usaríamos **Flyway** o **Liquibase** para
> versionar el esquema con scripts. Con `ddl-auto=create-drop` **no hay migraciones** —Hibernate crea
> la "base de datos" vacía en cada arranque—, por eso en este módulo no entramos en ese flujo. El
> concepto queda anotado: *cada servicio versiona su propio esquema de forma independiente* (coherente
> con 6.1).

## Sembrar datos para la clase

Como H2 arranca vacío, conviene un *seed* para tener pizzas con las que jugar en el aula. Un
`CommandLineRunner` se ejecuta una vez al terminar el arranque:

```java
package com.shopmicro.catalog;

import com.shopmicro.catalog.pizza.persistence.PizzaJpa;
import com.shopmicro.catalog.pizza.persistence.PizzaJpaRepository;
import org.springframework.boot.CommandLineRunner;
import org.springframework.stereotype.Component;

import java.util.HashSet;
import java.util.UUID;

@Component
public class CatalogSeed implements CommandLineRunner {

    private final PizzaJpaRepository pizzas;

    public CatalogSeed(PizzaJpaRepository pizzas) {
        this.pizzas = pizzas;
    }

    @Override
    public void run(String... args) {
        if (pizzas.count() > 0) return;   // idempotente

        PizzaJpa margarita = new PizzaJpa(
            UUID.randomUUID(),
            "Margarita",
            "La clásica de toda la vida",
            "https://shopmicro/img/margarita.png",
            new HashSet<>());

        pizzas.save(margarita);
    }
}
```

> Como alternativa, Spring Boot ejecuta automáticamente un `data.sql` del classpath al arrancar. El
> `CommandLineRunner` se prefiere aquí porque trabaja con las mismas clases de persistencia que el
> resto del servicio.

---

## Puntos clave

- **JPA vive solo en infraestructura.** El dominio nunca lo referencia.
- `EntityManager` = unidad de trabajo; `JpaRepository<J, ID>` = acceso a la tabla.
- **H2 en memoria** elimina la infraestructura y deja el foco en el diseño; el código de repositorios
  es el mismo que con PostgreSQL.
- **Cambiar de motor son unas líneas** en `application.properties`.
- H2 en memoria con `create-drop` **no tiene migraciones** y se borra al parar: vale para aprender y
  para tests, no para producción.
