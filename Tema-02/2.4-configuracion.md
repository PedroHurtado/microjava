# 2.4 Configuración (`application.yml`, perfiles, variables de entorno, `@ConfigurationProperties`)

> **Módulo 2 — Fundamentos de Spring Boot para Microservicios**
> Duración estimada: ~40 min

---

## Por qué importa especialmente en microservicios

Un microservicio se ejecuta en muchos entornos (tu máquina, CI, *staging*, producción) y en contenedores donde la configuración llega **desde fuera** (variables de entorno, *secrets* de Kubernetes). Entender bien el sistema de configuración de Spring Boot es imprescindible para que el mismo binario funcione en todas partes sin recompilar.

---

## El sistema de configuración por capas

Spring Boot construye la configuración **apilando varias fuentes** (*property sources*), donde **cada fuente sobrescribe a la anterior**. Este es el orden simplificado (de menor a mayor prioridad):

```
1. application.yml / application.properties   (base, en el repo)
2. application-{perfil}.yml                    (p.ej. application-dev.yml)
3. Variables de entorno / propiedades del sistema   ◄── clave en contenedores
4. Argumentos de línea de comandos             (máxima prioridad)
        │
        ▼
  Environment final (las de abajo ganan)
```

Esto es potentísimo: defines valores por defecto en `application.yml` y los **sobreescribes en producción con variables de entorno**, sin tocar el código ni la imagen del contenedor.

---

## `application.yml`

Un archivo YAML jerárquico (también puedes usar `application.properties`):

```yaml
logging:
  level:
    root: INFO

spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/catalog
    username: postgres
    password: dev

catalog:
  page-size: 20
  enable-cache: true
```

### Perfiles (profiles)

Spring lee la propiedad `spring.profiles.active` (o la variable `SPRING_PROFILES_ACTIVE`) para saber en qué entorno está (`dev`, `staging`, `prod`...). Según su valor, carga además el `application-{perfil}.yml` correspondiente, que sobreescribe al base.

Puedes condicionar beans o lógica al perfil activo con `@Profile` o consultando el `Environment`:

```java
@Bean
@Profile("dev")
CommandLineRunner seedData(...) {      // este bean solo existe en el perfil "dev"
    return args -> { /* cargar datos de ejemplo */ };
}
```

## Leer configuración: las formas

### Acceso directo con `@Value` (rápido, pero poco recomendable a gran escala)

```java
@Value("${catalog.page-size}")
private int pageSize;

@Value("${spring.datasource.url}")
private String dbUrl;
```

### El patrón `@ConfigurationProperties` (el recomendado)

En lugar de leer claves sueltas con *strings* por todo el código, **enlazas una sección a una clase fuertemente tipada**:

```java
// 1. Una clase que modela la sección de config
@ConfigurationProperties(prefix = "catalog")
public class CatalogSettings {
    private int pageSize;
    private boolean enableCache;
    // getters y setters...
}

// 2. La habilitamos (en una @Configuration o en la clase principal)
@Configuration
@EnableConfigurationProperties(CatalogSettings.class)
public class CatalogConfig { }

// 3. La inyectamos como cualquier otro bean
@RestController
public class ProductController {
    private final CatalogSettings settings;

    public ProductController(CatalogSettings settings) {
        this.settings = settings;       // ← la instancia tipada, ya enlazada
    }

    @GetMapping("/products")
    public List<Product> getAll() {
        return service.getPage(settings.getPageSize());
    }
}
```

Ventajas: tipado fuerte (nada de *strings* mágicos repetidos), validación, agrupación lógica y *autocompletado* en el IDE (con `spring-boot-configuration-processor`).

### Recarga en caliente de la configuración

| Mecanismo | Comportamiento |
|---|---|
| `@ConfigurationProperties` (normal) | Valores fijos, enlazados al arrancar |
| `@RefreshScope` (Spring Cloud) | Recrea el bean cuando se refresca la configuración en caliente |
| Config externa (Config Server / K8s) | Recarga sin reiniciar el servicio |

Para la mayoría de casos, `@ConfigurationProperties` enlazado al arranque es suficiente.

## Variables de entorno: el puente hacia los contenedores

En contenedores y Kubernetes, la configuración casi siempre llega por **variables de entorno**. Spring Boot usa **relaxed binding**: una propiedad como `catalog.page-size` se puede definir con la variable de entorno `CATALOG_PAGESIZE` (mayúsculas, los puntos pasan a `_`):

```bash
# Esto sobreescribe catalog.page-size del application.yml
export CATALOG_PAGESIZE=50

# Esto sobreescribe la cadena de conexión
export SPRING_DATASOURCE_URL="jdbc:postgresql://db:5432/catalog"
```

En un `docker-compose.yml` (Módulo 7) se verá así:

```yaml
services:
  catalog-api:
    image: shopmicro/catalog
    environment:
      - SPRING_PROFILES_ACTIVE=prod
      - SPRING_DATASOURCE_URL=jdbc:postgresql://postgres:5432/catalog
      - SPRING_DATASOURCE_USERNAME=app
      - SPRING_DATASOURCE_PASSWORD=...
      - CATALOG_PAGESIZE=50
```

> **Idea central:** el **mismo binario y la misma imagen** se comportan distinto en cada entorno solo cambiando variables de entorno. Nunca hardcodees configuración que cambie entre entornos.

## Validación de configuración al arranque

Una buena práctica en microservicios: que el servicio **falle rápido al arrancar** si le falta configuración crítica, en vez de petar a mitad de una petición. Combina `@ConfigurationProperties` con `@Validated` y las anotaciones de Bean Validation:

```java
@ConfigurationProperties(prefix = "catalog")
@Validated
public class CatalogSettings {

    @Min(value = 1, message = "page-size debe ser mayor que 0")
    private int pageSize;

    @NotNull
    private Boolean enableCache;

    // getters y setters...
}
```

Si la configuración no cumple las restricciones, la aplicación **no arranca** y muestra el error claramente, en lugar de fallar en la primera petición que dependa de ese valor.

---

## Puntos clave

- La configuración se construye **por capas**; cada fuente sobrescribe a la anterior.
- Orden de prioridad: `application.yml` → `application-{perfil}.yml` → **variables de entorno** → CLI.
- `spring.profiles.active` (o `SPRING_PROFILES_ACTIVE`) selecciona el `application-{perfil}.yml` a cargar.
- Usa el patrón **`@ConfigurationProperties`** para configuración tipada en vez de leer *strings* sueltos con `@Value`.
- En contenedores, las propiedades se mapean a variables de entorno con **relaxed binding** (`catalog.page-size` → `CATALOG_PAGESIZE`).
- Valida la configuración con **`@Validated`** + Bean Validation para fallar rápido al arrancar.
