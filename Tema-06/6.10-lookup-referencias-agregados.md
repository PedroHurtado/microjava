# 6.10 — Lookup: resolver referencias entre agregados (OCP)

> **Módulo 6 · Lección 10 de 9** *(adenda)*
> **Duración estimada:** ~30 min
> **Tipo:** Teoría + código
> **Requisitos:** 6.3 (núcleo ISP) y 6.4 (agregado Pizza)

---

## El problema que resuelve

Cuando creamos una `Pizza`, el comando **no trae objetos `Ingredient` completos**: trae una lista
de **ids** de ingredientes. Alguien tiene que **resolver esos ids a entidades de dominio reales** —
y verificar que existen — antes de construir el agregado.

```
   Command de "crear pizza"                Agregado de dominio
   ────────────────────────                ───────────────────
   name = "Cuatro quesos"                   Pizza
   ingredientIds = [ g1, g2, g3, g4 ]  ──▶  ingredients = { Mozzarella, ... }
                       │                          ▲
                       │   ¿quién traduce          │
                       └── ids → Ingredient? ──────┘
```

Podríamos inyectar `IGetIngredient` directamente en el handler de pizza y llamarlo en bucle. Pero
fíjate en lo que pasaría a medida que crecen los agregados que **referencian** a otros:

- El handler de Pizza necesita resolver `Ingredient`.
- Mañana, un agregado `Menu` necesita resolver `Pizza` **y** `Bebida`.
- Pasado, `Pedido` necesita resolver `Producto`, `Cliente`, `Cupón`…

Cada caso acabaría inyectando y cableando a mano sus resolvedores. Hay un patrón mejor: un
**mecanismo único de resolución de referencias por tipo**, abierto a extensión y cerrado a
modificación (**OCP**, la "O" de SOLID).

---

## La idea: `ILookup<T>` + `LookupResolver`

Dos piezas, en el núcleo del dominio (`core/lookup`):

- **`ILookup<T>`** — sabe resolver **un** agregado destino de FK por id. Cada agregado que pueda ser
  *referenciado* por otro registra su propia implementación.
- **`LookupResolver`** — punto único. Recibe **todos** los `ILookup` registrados, los **indexa por
  el tipo que resuelven** y ofrece `find(Class<T>, id)` / `findAll(Class<T>, ids)`.

```
            ┌─────────────────────────────────────────────┐
            │              LookupResolver                  │
            │   índice:  Class<?> ─────────▶ ILookup       │
            │            Ingredient ──────▶ IngredientLookup│
            │            Bebida     ──────▶ BebidaLookup    │
            │            ...        ──────▶ ...             │
            └───────────────────┬─────────────────────────┘
                                │  resolver.findAll(Ingredient.class, ids)
                                ▼
              devuelve  Set<Ingredient>  (o lanza si algún id no existe)

   Añadir un nuevo destino (p. ej. Topping):
       basta registrar un ILookup<Topping> nuevo (un @Component más).
       LookupResolver NO se modifica  →  Open/Closed.
```

> **Detalle de diseño.** La interfaz `ILookup<T>` **no declara qué tipo resuelve**: el resolver lo
> **infiere del propio genérico** por reflexión al construir el índice. Así la interfaz queda
> reducida a lo esencial — una sola operación, `find` — y el implementador no tiene que repetir
> el tipo por ningún sitio.

---

## La interfaz `ILookup<T>`

```java
package com.shopmicro.core.lookup;

/**
 * Interfaz marcadora (vacía). No aporta métodos: existe solo para que Spring pueda
 * inyectar TODOS los lookups de golpe como List&lt;LookupMarker&gt;, sin tener que
 * registrar cada uno por separado.
 */
public interface LookupMarker { }
```

```java
package com.shopmicro.core.lookup;

import java.util.UUID;

/**
 * Resuelve una referencia (FK) a un agregado de dominio por id. Cada agregado que sea
 * destino de FK registra su propia implementación de ILookup&lt;T&gt;.
 *
 * find devuelve T o lanza EntityNotFoundException si no existe — misma semántica que
 * IGet (lección 6.3): nunca null, nunca Optional.
 */
public interface ILookup<T> extends LookupMarker {
    T find(UUID id);
}
```

Una sola operación útil. El tipo que resuelve se infiere del genérico; no se declara aquí. La
marcadora `LookupMarker` no se implementa a mano: viene "gratis" por extender `ILookup<T>`.

> **La técnica de la interfaz marcadora.** Una interfaz vacía sirve de **etiqueta** común a un
> conjunto de tipos heterogéneos para que el contenedor los recoja juntos. Es exactamente lo que
> hace Spring cuando inyectas `List<LookupMarker>`: te entrega **todos** los beans que la implementan.
> El mismo truco lo reutilizaremos para **auto-registrar endpoints**: una marcadora `Endpoint`, cada
> slice la implementa, y un único barrido (`List<Endpoint>`) las registra todas. Apréndela bien aquí.

## El `LookupResolver`

Una nota antes del código: en Java los genéricos se **borran** en tiempo de ejecución (*type
erasure*). Un método `find(UUID id)` no podría recuperar `T` para buscar en el índice. Por eso la API
recibe el `Class<T>` explícito: `find(Ingredient.class, id)`. Es el patrón idiomático en Java cuando
hace falta el tipo en runtime.

```java
package com.shopmicro.core.lookup;

import org.springframework.stereotype.Component;

import java.lang.reflect.ParameterizedType;
import java.lang.reflect.Type;
import java.util.Collection;
import java.util.HashSet;
import java.util.List;
import java.util.Map;
import java.util.Set;
import java.util.UUID;
import java.util.function.Function;
import java.util.stream.Collectors;

/**
 * Punto único para resolver FKs por tipo de agregado. Spring le inyecta por colección
 * todos los ILookup&lt;T&gt; registrados y los indexa por el tipo T que resuelve cada uno
 * (inferido del genérico, sin que la interfaz tenga que declararlo).
 *
 * Añadir un nuevo destino de FK NO obliga a tocar esta clase: basta registrar un nuevo
 * ILookup&lt;T&gt; como bean. Eso es Open/Closed (OCP) en la práctica.
 */
@Component
public class LookupResolver {

    private final Map<Class<?>, LookupMarker> lookups;

    // Spring inyecta todos los beans que implementan LookupMarker. Cada uno cierra
    // ILookup<T> sobre un T distinto; el tipo concreto se recupera al indexar.
    public LookupResolver(List<LookupMarker> lookups) {
        this.lookups = lookups.stream()
            .collect(Collectors.toMap(LookupResolver::targetTypeOf, Function.identity()));
    }

    /** Extrae el T de un objeto que implementa ILookup&lt;T&gt;. */
    private static Class<?> targetTypeOf(LookupMarker lookup) {
        for (Type iface : lookup.getClass().getGenericInterfaces()) {
            if (iface instanceof ParameterizedType pt && pt.getRawType() == ILookup.class) {
                return (Class<?>) pt.getActualTypeArguments()[0];   // el T de ILookup<T>
            }
        }
        throw new IllegalStateException(
            lookup.getClass().getName() + " no implementa ILookup<T> directamente.");
    }

    /** Resuelve una referencia. Lanza si el id no existe (propaga de find). */
    @SuppressWarnings("unchecked")
    public <T> T find(Class<T> type, UUID id) {
        LookupMarker lookup = lookups.get(type);
        if (lookup == null) {
            throw new NoLookupRegisteredException(type);
        }
        return ((ILookup<T>) lookup).find(id);   // cast encapsulado AQUÍ, no en el llamante
    }

    /**
     * Resuelve una colección de referencias. Si algún id no existe, find lanza
     * EntityNotFoundException → la operación entera falla (no devuelve resultados parciales).
     */
    public <T> Set<T> findAll(Class<T> type, Collection<UUID> ids) {
        Set<T> result = new HashSet<>();
        for (UUID id : ids) {
            result.add(find(type, id));
        }
        return result;
    }
}
```

```java
package com.shopmicro.core.lookup;

/** Se lanza al pedir un tipo para el que no hay ILookup registrado (fallo de cableado). */
public class NoLookupRegisteredException extends RuntimeException {

    public NoLookupRegisteredException(Class<?> type) {
        super("No hay Lookup registrado para el tipo " + type.getSimpleName() + ".");
    }
}
```

> **El único cast feo está encapsulado** dentro de `find`, que es seguro porque el índice se
> construyó justo con ese tipo. El código que **usa** el resolver (`findAll(Ingredient.class, ids)`)
> trabaja con genéricos fuertemente tipados — nunca ve un cast ni un `Object`.

## Una implementación: `IngredientLookup`

El agregado `Ingredient` es destino de FK desde `Pizza`, así que registra su lookup. Reutiliza el
puerto de lectura que ya tiene del núcleo (6.3):

```java
package com.shopmicro.catalog.ingredient.persistence;

import com.shopmicro.core.lookup.ILookup;
import com.shopmicro.catalog.ingredient.domain.Ingredient;
import org.springframework.stereotype.Component;

import java.util.UUID;

/**
 * Lookup de Ingredient. Se apoya en IGetIngredient (puerto de lectura del núcleo),
 * que ya lanza EntityNotFoundException si el id no existe.
 */
@Component
public class IngredientLookup implements ILookup<Ingredient> {

    private final IGetIngredient get;

    public IngredientLookup(IGetIngredient get) {
        this.get = get;
    }

    @Override
    public Ingredient find(UUID id) {
        return get.get(id);
    }
}
```

El lookup simplemente **reenvía** el `get` del puerto: sin adaptadores, la lógica de "leer o lanzar"
ya vive en el núcleo y aquí se reaprovecha tal cual.

## Uso en el handler de crear pizza

Aquí se cierra el círculo. El handler resuelve los ids → ingredientes y construye el agregado con
la **fábrica** (que sí emite el evento, lección 6.4):

```java
package com.shopmicro.catalog.pizza.features.create;

import com.shopmicro.core.lookup.LookupResolver;
import com.shopmicro.catalog.ingredient.domain.Ingredient;
import com.shopmicro.catalog.pizza.domain.Pizza;
import com.shopmicro.catalog.pizza.persistence.IAddPizza;
import org.springframework.stereotype.Service;

import java.util.List;
import java.util.Set;
import java.util.UUID;

@Service
public class Handler {

    private final IAddPizza repository;
    private final LookupResolver resolver;

    public Handler(IAddPizza repository, LookupResolver resolver) {
        this.repository = repository;
        this.resolver = resolver;
    }

    public Response handle(Command command) {
        // ids → entidades de dominio (lanza 404 si algún ingrediente no existe)
        Set<Ingredient> ingredients = resolver.findAll(Ingredient.class, command.ingredientIds());

        // construir con la FÁBRICA: emite PizzaCreated
        Pizza pizza = Pizza.create(command.name(), command.description(), command.url(), ingredients);

        repository.add(pizza);
        return new Response(pizza.getId());
    }
}
```

```java
package com.shopmicro.catalog.pizza.features.create;

import java.util.List;
import java.util.UUID;

public record Command(String name, String description, String url, List<UUID> ingredientIds) { }
public record Response(UUID id) { }
```

El handler **no conoce** `IngredientLookup` ni ningún lookup concreto: solo habla con
`LookupResolver`. Si mañana la pizza referencia también `Salsa`, el handler haría
`resolver.findAll(Salsa.class, ...)` y bastaría con **registrar** un `ILookup<Salsa>` — sin tocar el
resolver ni este handler.

## Registro en Spring (la inyección de colección)

Cada lookup es un `@Component`. Como todos implementan `LookupMarker` (vía `ILookup<T>`), Spring
inyecta la **lista completa** en el constructor de `LookupResolver` automáticamente. Nada de
registrar cada uno a mano:

```java
// Cada agregado destino de FK registra su lookup como un @Component normal
@Component
public class IngredientLookup implements ILookup<Ingredient> { /* ... */ }
// ... futuros: @Component SalsaLookup implements ILookup<Salsa>, etc.

// El resolver recibe TODOS los lookups por inyección de colección (List<LookupMarker>)
@Component
public class LookupResolver {
    public LookupResolver(List<LookupMarker> lookups) { /* indexa por tipo */ }
}
```

Spring recoge **solo** los beans que implementan `LookupMarker` — nunca tipos ajenos. Añadir un
destino de FK nuevo es un `@Component` más; el resolver no se toca.

---

## Por qué esto es Open/Closed

```
   AÑADIR un nuevo destino de FK (p. ej. Salsa)
   ───────────────────────────────────────────
   1. Crear  SalsaLookup implements ILookup<Salsa>
   2. Anotarlo con @Component

   Lo que NO se toca:
   ✗ LookupResolver        (cerrado a modificación)
   ✗ los handlers que usan el resolver
   ✗ los demás lookups

   El sistema queda ABIERTO a extensión (nuevos lookups) y
   CERRADO a modificación (el resolver no cambia nunca).
```

---

## Puntos clave

- **`Lookup` resuelve referencias (FKs) entre agregados**: traduce ids → entidades de dominio,
  verificando existencia.
- **`LookupResolver` es Open/Closed**: añadir un destino de FK = registrar un `ILookup<T>` nuevo
  (`@Component`), **sin modificar** el resolver ni los handlers que lo usan.
- La interfaz queda **limpia** (solo `find`): el tipo se **infiere del genérico** por reflexión, no
  se declara dentro de la interfaz.
- Por el *type erasure* de Java, la API del resolver recibe el `Class<T>` explícito
  (`find(Ingredient.class, id)`) para recuperar el tipo en tiempo de ejecución.
- Reutiliza la semántica de 6.3: **`find` lanza** si no existe; `findAll` falla entera si falta un id
  (sin resultados parciales). El **cast inseguro queda encapsulado** en el resolver.
