# Módulo 4 · Lección 4.2 — Comunicación HTTP con `RestClient`: la forma base

> **Duración estimada:** 50 min
> **Servicio del proyecto:** `ordering-service` consumiendo `catalog-service`
> **Requisitos:** Lección 4.1; inyección de dependencias de Spring (Módulo 2)

---

## 1. El caso que nos acompañará en 4.2 → 4.3 → 4.4

Antes de crear un pedido, **`ordering-service` necesita confirmar que el producto existe** (y,
de paso, traer su precio actual). Esa información vive en `catalog-service`. Es una consulta
puntual cuya respuesta necesitamos *ahora* para decidir → comunicación **síncrona** por HTTP.

Vamos a implementar esa llamada tres veces, cada una mejor que la anterior:

- **4.2 (esta):** a mano, con `RestClient`. Para entender la base.
- **4.3:** con una *HTTP Interface* declarativa, eliminando el *boilerplate*.
- **4.4:** con Resilience4j, haciéndola resiliente.

---

## 2. Por qué NO se abre una conexión nueva por llamada

El error clásico del principiante: crear un cliente HTTP y su pool de conexiones en cada
invocación.

```java
// ❌ NO hagas esto.
public boolean exists(int productId) {
    // Un cliente nuevo (y su pool de conexiones) en cada llamada.
    HttpClient client = HttpClient.newHttpClient();
    HttpRequest request = HttpRequest.newBuilder()
            .uri(URI.create("https://catalog/products/" + productId))
            .build();
    HttpResponse<Void> response = client.send(request, BodyHandlers.discarding());
    return response.statusCode() == 200;
}
```

Dos problemas graves: **agotamiento de sockets / hilos** (cada cliente abre su propio pool de
conexiones; bajo carga te quedas sin recursos) y trabajo de configuración (timeouts, TLS,
pool) repetido y sin control central. El cliente HTTP está pensado para **crearse una vez y
reutilizarse**.

En Spring, `RestClient` resuelve ambos: lo declaras **una sola vez como bean**, con su pool de
conexiones y sus timeouts, y lo reutilizas en toda la aplicación. **Siempre** se inyecta el
cliente ya configurado; nunca se construye dentro de un método de negocio.

---

## 3. Cliente tipado: la forma recomendada

Encapsulamos las llamadas a un servicio concreto en **una clase tipada** (`CatalogClient`) que
recibe un `RestClient` ya configurado. Encaja con la inyección de dependencias y mantiene la
URL base y la configuración en un solo sitio.

### 3.1 El contrato (en la capa de aplicación de Ordering)

Siguiendo ISP, la interfaz declara **solo lo que Ordering necesita** de Catalog, no un cliente
genérico:

```java
package com.shopmicro.ordering.application.catalog;

public interface CatalogClient {
    ProductInfo getProduct(int productId);
}

public record ProductInfo(int id, String name, BigDecimal price, boolean available) { }
```

### 3.2 La implementación (cliente tipado en infraestructura)

```java
package com.shopmicro.ordering.infrastructure.catalog;

import com.shopmicro.ordering.application.catalog.CatalogClient;
import com.shopmicro.ordering.application.catalog.ProductInfo;
import org.springframework.http.HttpStatus;
import org.springframework.stereotype.Component;
import org.springframework.web.client.RestClient;

@Component
public class HttpCatalogClient implements CatalogClient {

    private final RestClient restClient;

    public HttpCatalogClient(RestClient catalogRestClient) {
        this.restClient = catalogRestClient;
    }

    @Override
    public ProductInfo getProduct(int productId) {
        return restClient.get()
                .uri("products/{id}", productId)
                .retrieve()
                .onStatus(status -> status == HttpStatus.NOT_FOUND,
                        (request, response) -> { throw new ProductNotFoundException(productId); })
                .body(ProductInfo.class);
    }
}
```

> Fíjate: la URI es **relativa** (`products/{id}`). La base (`https://catalog/`) se configura
> una sola vez al crear el `RestClient`. El cliente tipado no sabe *dónde* está Catalog; eso es
> configuración, no código.

## 4. Registro como bean

```java
package com.shopmicro.ordering.infrastructure.catalog;

import java.time.Duration;
import org.springframework.beans.factory.annotation.Value;
import org.springframework.boot.web.client.ClientHttpRequestFactorySettings;
import org.springframework.boot.web.client.ClientHttpRequestFactories;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.web.client.RestClient;

@Configuration
public class CatalogClientConfig {

    @Bean
    RestClient catalogRestClient(RestClient.Builder builder,
                                 @Value("${services.catalog.base-url}") String baseUrl) {
        var settings = ClientHttpRequestFactorySettings.DEFAULTS
                .withConnectTimeout(Duration.ofSeconds(2))
                .withReadTimeout(Duration.ofSeconds(10));

        return builder
                .baseUrl(baseUrl)
                .requestFactory(ClientHttpRequestFactories.get(settings))
                .build();
    }
}
```

Spring Boot ofrece un `RestClient.Builder` ya preconfigurado (con observabilidad y el pool de
conexiones del proyecto). Partimos de él, le fijamos la **URL base** y los **timeouts**, y lo
exponemos como bean. A partir de aquí, cualquier componente de Ordering puede pedir
`CatalogClient` por constructor.

```yaml
# application.yml de ordering-service
services:
  catalog:
    base-url: https://catalog/
```

```java
public OrderId handle(CreateOrderCommand command) {
    ProductInfo product = catalogClient.getProduct(command.productId());

    if (product == null || !product.available()) {
        throw new ProductNotAvailableException(command.productId());
    }

    // ... crear el pedido con el precio traído de Catalog ...
}
```

## 6. Lo que todavía falta (y motiva 4.3 y 4.4)

Esta versión funciona, pero arrastra problemas que iremos resolviendo:

- **Boilerplate**: construir URIs a mano, comprobar status codes, deserializar. Multiplícalo
  por cada endpoint de cada servicio. → **4.3 (HTTP Interface)** lo elimina.
- **Frágil ante fallos**: si Catalog da un *timeout* puntual o un error transitorio, la llamada
  muere sin reintentar. No hay *circuit breaker*. → **4.4 (Resilience4j)** lo arregla.

---

## Puntos clave

- **Nunca abras un cliente/pool HTTP por llamada**: agota recursos y dispersa la configuración.
  Declara **un `RestClient` como bean** y reutilízalo.
- El **cliente tipado** encapsula las llamadas a un servicio en una clase, integrada con DI; la
  **URL base y los timeouts son configuración** (`application.yml`), no código.
- El contrato (`CatalogClient`) declara **solo lo que el consumidor necesita** (ISP), no un
  cliente HTTP genérico.
- Un `@Bean RestClient` con `baseUrl` + timeouts centraliza la conexión a un servicio en un solo
  sitio.
- Esta forma base **funciona pero tiene boilerplate y es frágil**: 4.3 quita el boilerplate, 4.4
  añade resiliencia.
