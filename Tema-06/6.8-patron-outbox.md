# 6.8 — Patrón Outbox para no perder eventos

> **Módulo 6 · Lección 8 de 9**
> **Duración estimada:** ~30 min
> **Tipo:** Teoría + esqueleto de código

---

## El fallo sutil que Outbox resuelve

La Saga (6.7) depende de que, cuando un servicio guarda algo en su BD, el **evento
correspondiente se publique**. El problema es que son **dos sistemas distintos**: la base de datos
y el bus de mensajes. Y coordinar dos sistemas sin transacción común tiene una trampa.

Mira este código aparentemente inocente:

```java
// ❌ Doble escritura: dos sistemas, ninguna garantía de atomicidad
orders.add(order);                            // 1. escribe en la BD
bus.publish(new OrderCreated(order.getId())); // 2. publica el evento
```

¿Qué pasa si el servicio **se cae entre la línea 1 y la línea 2**?

```
   ✓ línea 1: el pedido QUEDÓ guardado en la BD
   ✗ línea 2: NUNCA se ejecutó → el evento se PERDIÓ

   Resultado: hay un pedido que nadie sabe que existe.
   Catalog jamás reserva stock. Payments jamás cobra. La Saga muere en el paso 1.
```

Y el orden inverso tampoco salva:

```java
bus.publish(new OrderCreated(order.getId())); // publica primero
orders.add(order);                            // ...y luego se cae aquí
// Resultado: se publicó un evento de un pedido que NO existe. Aún peor.
```

Esto es el problema de la **doble escritura** (*dual write*): no puedes escribir
atómicamente en la BD y en el bus a la vez, porque son dos infraestructuras separadas.

---

## La idea de Outbox

> **Convierte las dos escrituras en una sola.** Guarda el evento en una **tabla `outbox` de tu
> propia base de datos**, dentro de la **misma transacción** que guarda los datos de negocio. Luego,
> un proceso aparte lee esa tabla y publica los eventos al bus.

```
   ┌─────────────── UNA SOLA TRANSACCIÓN LOCAL (ACID) ───────────────┐
   │                                                                  │
   │   INSERT INTO orders   (el pedido)                               │
   │   INSERT INTO outbox   (el evento OrderCreated, serializado)     │
   │                                                                  │
   └──────────────────────────── COMMIT ─────────────────────────────┘
                                    │
                                    │  (más tarde, proceso independiente)
                                    ▼
                       ┌──────────────────────┐
                       │  Outbox Publisher     │  lee filas no publicadas
                       │  (worker @Scheduled)  │  de la tabla outbox
                       └──────────┬────────────┘
                                  │ publica al bus
                                  ▼
                          ┌──────────────┐
                          │   Event Bus  │  → dispara la Saga
                          └──────────────┘
```

Como el evento y el dato de negocio se guardan en **la misma transacción de la misma BD**, **o se
guardan ambos o ninguno**. Ya no hay forma de tener un pedido sin su evento. La publicación al bus
pasa a ser un problema aparte y **reintentable**: si el publisher falla, la fila sigue en `outbox`
esperando el próximo intento.

## Esqueleto: la tabla outbox

```java
package com.shopmicro.core.outbox;

import jakarta.persistence.Entity;
import jakarta.persistence.Id;
import jakarta.persistence.Lob;
import jakarta.persistence.Table;

import java.time.Instant;
import java.util.UUID;

/** Una fila de la tabla outbox: un evento pendiente de publicar. */
@Entity
@Table(name = "outbox")
public class OutboxMessage {

    @Id
    private UUID id = UUID.randomUUID();

    private String type;             // nombre del tipo de evento

    @Lob
    private String payload;          // evento serializado (JSON)

    private Instant occurredOn = Instant.now();
    private Instant publishedOn;     // null = aún no publicado

    protected OutboxMessage() { }    // requerido por JPA

    public OutboxMessage(String type, String payload) {
        this.type = type;
        this.payload = payload;
    }

    public UUID getId()             { return id; }
    public String getType()         { return type; }
    public String getPayload()      { return payload; }
    public Instant getOccurredOn()  { return occurredOn; }
    public Instant getPublishedOn() { return publishedOn; }

    public void markPublished()     { this.publishedOn = Instant.now(); }
}
```

Su repositorio Spring Data, con la consulta de pendientes:

```java
package com.shopmicro.core.outbox;

import org.springframework.data.jpa.repository.JpaRepository;

import java.util.List;
import java.util.UUID;

public interface OutboxRepository extends JpaRepository<OutboxMessage, UUID> {
    List<OutboxMessage> findByPublishedOnIsNull();
}
```

## Esqueleto: escribir dato + evento en la misma transacción

El `@Transactional` garantiza que el `save` del pedido y el `save` del evento se confirman **juntos
o ninguno**:

```java
package com.shopmicro.ordering.features;

import com.fasterxml.jackson.databind.ObjectMapper;
import com.shopmicro.core.outbox.OutboxMessage;
import com.shopmicro.core.outbox.OutboxRepository;
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;

import java.math.BigDecimal;
import java.util.UUID;

@Service
public class CreateOrderHandler {

    private final OutboxRepository outbox;
    private final ObjectMapper json;
    // private final OrderJpaRepository orders;

    public CreateOrderHandler(OutboxRepository outbox, ObjectMapper json) {
        this.outbox = outbox;
        this.json = json;
    }

    @Transactional
    public void handle(/* Command command */) throws Exception {
        // Order order = Order.create(...);
        // orders.save(toJpa(order));

        OrderCreated event = new OrderCreated(UUID.randomUUID(), UUID.randomUUID(), 1, BigDecimal.TEN);

        outbox.save(new OutboxMessage(
            OrderCreated.class.getSimpleName(),
            json.writeValueAsString(event)));

        // Al cerrar el método, @Transactional confirma pedido y evento juntos (o ninguno).
    }
}
```

## Esqueleto: el publisher (worker programado)

```java
package com.shopmicro.core.outbox;

import com.shopmicro.core.messaging.InMemoryEventBus;
import org.springframework.scheduling.annotation.Scheduled;
import org.springframework.stereotype.Component;
import org.springframework.transaction.annotation.Transactional;

import java.util.List;

/**
 * Lee periódicamente la tabla outbox y publica al bus los eventos no publicados.
 * Si la publicación falla, la fila sigue pendiente y se reintenta. Esqueleto didáctico.
 */
@Component
public class OutboxPublisher {

    private final OutboxRepository outbox;
    private final InMemoryEventBus bus;

    public OutboxPublisher(OutboxRepository outbox, InMemoryEventBus bus) {
        this.outbox = outbox;
        this.bus = bus;
    }

    @Scheduled(fixedDelay = 1000)   // cada segundo
    @Transactional
    public void publishPending() {
        List<OutboxMessage> pending = outbox.findByPublishedOnIsNull();

        for (OutboxMessage msg : pending) {
            // bus.publish(/* deserializar msg.getPayload() según msg.getType() */);
            msg.markPublished();    // marcar como publicado (se persiste al cerrar la transacción)
        }
    }
}
```

Para activar el scheduler, basta anotar la clase de arranque con `@EnableScheduling`:

```java
@SpringBootApplication
@EnableScheduling
public class OrderingApplication { /* ... */ }
```

> Esqueleto deliberado, coherente con 6.7: el bus es el `InMemoryEventBus`, y la deserialización
> por tipo se completa al montar la mensajería real. Lo que importa es la **garantía**: dato y
> evento en la misma transacción; la publicación, reintentable.

## Outbox + idempotencia = entrega fiable

Outbox garantiza **"al menos una vez"** (*at-least-once*): el evento se publicará, posiblemente
**más de una vez** (si el worker se cae justo después de publicar pero antes de marcar
`publishedOn`). Por eso encaja con lo que vimos en la Saga:

```
   Outbox      → garantiza que el evento NO se pierde   (at-least-once)
       +
   Idempotencia → garantiza que procesarlo 2 veces no rompe nada
       =
   Entrega fiable de extremo a extremo
```

Las dos piezas se complementan: una asegura que el mensaje **llega**, la otra que **llegar dos
veces es inofensivo**.

---

## Puntos clave

- La **doble escritura** (BD + bus por separado) puede **perder o inventar eventos** si el servicio
  se cae entre ambas.
- **Outbox** guarda el evento en una **tabla de la propia BD**, en la **misma transacción** que el
  dato de negocio (`@Transactional`): o se guardan ambos o ninguno.
- Un **worker programado** (`@Scheduled`) lee la tabla y publica al bus; si falla, **reintenta** (la
  fila sigue pendiente).
- Outbox da entrega **at-least-once**; combinado con **idempotencia** (de la Saga) se obtiene
  **entrega fiable** de extremo a extremo.
