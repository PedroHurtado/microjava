## Dónde estamos

En 6.3 montamos el **núcleo reutilizable** (`IAdd`/`IGet`/`IRemove`/`IUpdate` + las `*Jpa` con la
implementación por defecto). Ahora lo **aplicamos a un agregado concreto**: `Pizza`, con sus
`Ingredient` en relación N:M.

Vamos a ver cuatro piezas, en este orden:

```
   1. Dominio puro:        Pizza  (sin JPA, con fábrica que emite evento + ctor de hidratación)
   2. Persistencia:        PizzaJpa  (clase @Entity separada del dominio)
   3. Mapeo:               PizzaMapper  (manual, con Hydration → sin eventos al reconstruir)
   4. Vistas + repositorio: IAddPizza... + PizzaRepository
```

---

## Por qué una clase de persistencia SEPARADA del dominio

Esta es una **decisión de diseño explícita**: podríamos anotar el dominio `Pizza` directamente con
JPA. **No lo hacemos**, y conviene que el alumnado entienda por qué:

- El agregado `Pizza` recibe sus ingredientes como un `Set<Ingredient>` inmutable y protege sus
  invariantes con constructores controlados. Configurar Hibernate para que reconstruya **eso**
  (campos `final`, sin setters públicos, constructor sin argumentos que JPA exige) es **pelearse con
  el ORM** sin necesidad.
- Es más sencillo —y más limpio— tener una **clase intermedia `PizzaJpa`** pensada para JPA (con sus
  setters, su `HashSet`, sus anotaciones) y un **mapper manual** que traduce en ambos sentidos.
- Beneficio extra: **el dominio queda 100% libre de JPA.** No hay un solo `import jakarta.persistence`
  en la capa de dominio.

> Regla práctica: *la clase de dominio modela el negocio; la clase JPA modela la fila de la tabla.*
> Son dos responsabilidades distintas y por eso son dos clases distintas.

## 1. El dominio: `Pizza` (recordatorio autocontenido)

El agregado de dominio. Lo importante para este módulo: tiene **dos vías de construcción**.

```java
package com.shopmicro.catalog.pizza.domain;

import com.shopmicro.core.domain.EntityBase;
import com.shopmicro.catalog.ingredient.domain.Ingredient;

import java.util.ArrayList;
import java.util.Collection;
import java.util.Collections;
import java.util.HashSet;
import java.util.List;
import java.util.Set;
import java.util.UUID;

public class Pizza extends EntityBase<UUID> {

    private final Set<Ingredient> ingredients;
    private String name;
    private String description;
    private String url;

    // Eventos de dominio acumulados (lo trataremos a fondo con la mensajería).
    private final List<Object> events = new ArrayList<>();

    public String getName()        { return name; }
    public String getDescription() { return description; }
    public String getUrl()         { return url; }
    public Set<Ingredient> getIngredients() { return Collections.unmodifiableSet(ingredients); }

    public List<Object> events()   { return Collections.unmodifiableList(events); }
    public void clearEvents()      { events.clear(); }

    /**
     * FÁBRICA: crea una pizza NUEVA. Emite el evento de dominio pizza.created.
     * Es la única vía pública de creación.
     */
    public static Pizza create(String name, String description, String url,
                               Collection<Ingredient> ingredients) {
        Pizza pizza = new Pizza(UUID.randomUUID(), name, description, url, ingredients);
        pizza.events.add(new PizzaCreated(pizza.getId(), pizza.getName()));   // ← emite evento
        return pizza;
    }

    /**
     * CONSTRUCTOR DE HIDRATACIÓN (protected): reconstruye desde persistencia.
     * NO emite eventos. Lo usará el mapper para volver de la BD al dominio.
     */
    protected Pizza(UUID id, String name, String description, String url,
                    Collection<Ingredient> ingredients) {
        setId(id);
        this.name = name;
        this.description = description;
        this.url = url;
        this.ingredients = new HashSet<>(ingredients);
    }
}
```

```java
package com.shopmicro.catalog.pizza.domain;

import java.util.UUID;

public record PizzaCreated(UUID pizzaId, String name) { }
```

| Vía | Visibilidad | ¿Emite evento? | Uso |
|---|---|---|---|
| `Pizza.create(...)` | pública (fábrica) | **Sí** (`PizzaCreated`) | Crear una pizza nueva en negocio |
| `protected Pizza(...)` | protegida | **No** | Reconstruir desde la BD (mapper) |

> **El problema que esto resuelve.** Si reconstruyéramos desde BD con la fábrica, **cada lectura
> dispararía un `PizzaCreated`** — un evento falso por cada `SELECT`. El constructor protegido de
> hidratación lo evita de raíz. (Es la alternativa, vista en clase, a llamar a `clearEvents()`
> después de cada carga.)

## 2. La persistencia: `PizzaJpa`

La clase que JPA sí conoce. Modela la **fila** y la **relación N:M** con ingredientes. Tiene
setters y un constructor sin lógica de negocio: es un simple contenedor de datos.

```java
package com.shopmicro.catalog.pizza.persistence;

import com.shopmicro.catalog.ingredient.persistence.IngredientJpa;
import jakarta.persistence.Entity;
import jakarta.persistence.Id;
import jakarta.persistence.JoinColumn;
import jakarta.persistence.JoinTable;
import jakarta.persistence.ManyToMany;
import jakarta.persistence.Table;

import java.util.HashSet;
import java.util.Set;
import java.util.UUID;

/**
 * Entidad JPA de Pizza. La relación con ingredientes se modela como N:M a través de una
 * tabla intermedia (pizza_ingredient). Es un contenedor de datos: sin invariantes ni eventos.
 */
@Entity
@Table(name = "pizzas")
public class PizzaJpa {

    @Id
    private UUID id;

    private String name;
    private String description;
    private String url;

    @ManyToMany
    @JoinTable(
        name = "pizza_ingredient",
        joinColumns = @JoinColumn(name = "pizza_id"),
        inverseJoinColumns = @JoinColumn(name = "ingredient_id"))
    private Set<IngredientJpa> ingredients = new HashSet<>();

    // Constructor sin parámetros requerido por JPA
    protected PizzaJpa() { }

    public PizzaJpa(UUID id, String name, String description, String url,
                    Set<IngredientJpa> ingredients) {
        this.id = id;
        this.name = name;
        this.description = description;
        this.url = url;
        this.ingredients = ingredients;
    }

    public UUID getId()                 { return id; }
    public String getName()             { return name; }
    public String getDescription()      { return description; }
    public String getUrl()              { return url; }
    public Set<IngredientJpa> getIngredients() { return ingredients; }

    public void setId(UUID id)                       { this.id = id; }
    public void setName(String name)                 { this.name = name; }
    public void setDescription(String description)   { this.description = description; }
    public void setUrl(String url)                   { this.url = url; }
    public void setIngredients(Set<IngredientJpa> i) { this.ingredients = i; }
}
```

La relación N:M se declara con **anotaciones** sobre el propio campo (`@ManyToMany` + `@JoinTable`):
Hibernate crea y gestiona la tabla intermedia `pizza_ingredient` automáticamente.

## 3. El mapeo: `PizzaMapper` con `Hydration`

El mapper manual dominio ↔ JPA. La pieza fina es **cómo reconstruye el dominio sin emitir eventos**:
usando una subclase privada `Hydration` que invoca el **constructor protegido**.

```java
package com.shopmicro.catalog.pizza.persistence;

import com.shopmicro.core.repository.Mapper;
import com.shopmicro.catalog.ingredient.domain.Ingredient;
import com.shopmicro.catalog.ingredient.persistence.IngredientJpa;
import com.shopmicro.catalog.ingredient.persistence.IngredientMapper;
import com.shopmicro.catalog.pizza.domain.Pizza;
import org.springframework.stereotype.Component;

import java.util.Collection;
import java.util.Set;
import java.util.UUID;
import java.util.stream.Collectors;

/**
 * Mapper manual Pizza ↔ PizzaJpa. Se compone con IngredientMapper para mapear la
 * colección de ingredientes.
 *
 * toDomain reconstruye la pizza con la subclase Hydration, que invoca el constructor
 * protected de Pizza. Ese constructor NO emite eventos, de modo que reconstruir desde BD
 * no dispara pizza.created. Es la alternativa, vista en clase, a llamar a clearEvents()
 * tras la fábrica.
 */
@Component
public class PizzaMapper implements Mapper<Pizza, PizzaJpa> {

    private final IngredientMapper ingredientMapper;

    public PizzaMapper(IngredientMapper ingredientMapper) {
        this.ingredientMapper = ingredientMapper;
    }

    @Override
    public PizzaJpa toJpa(Pizza pizza) {
        Set<IngredientJpa> ingredients = pizza.getIngredients().stream()
            .map(ingredientMapper::toJpa)
            .collect(Collectors.toSet());

        return new PizzaJpa(pizza.getId(), pizza.getName(), pizza.getDescription(),
                            pizza.getUrl(), ingredients);
    }

    @Override
    public Pizza toDomain(PizzaJpa jpa) {
        Set<Ingredient> ingredients = jpa.getIngredients().stream()
            .map(ingredientMapper::toDomain)
            .collect(Collectors.toSet());

        return new Hydration(jpa.getId(), jpa.getName(), jpa.getDescription(),
                             jpa.getUrl(), ingredients);
    }

    /** Subclase de hidratación: invoca el constructor protected → sin eventos. */
    private static final class Hydration extends Pizza {
        Hydration(UUID id, String name, String description, String url,
                  Collection<Ingredient> ingredients) {
            super(id, name, description, url, ingredients);
        }
    }
}
```

El `PizzaMapper` con su `Hydration` interna. El truco —subclase que abre el constructor protegido—
es lo que permite reconstruir el agregado **sin disparar eventos**.

## 4. Las vistas por agregado y el repositorio

### Vistas: tipos fijados, sin JPA

Definimos "vistas" que **fijan los genéricos** del núcleo para `Pizza`. Solo hablan de dominio; no
saben nada de JPA. Son los **puertos** que la aplicación inyecta (ISP en acción: cada slice pide solo
la vista que necesita).

```java
package com.shopmicro.catalog.pizza.persistence;

import com.shopmicro.core.repository.IAdd;
import com.shopmicro.core.repository.IGet;
import com.shopmicro.core.repository.IRemove;
import com.shopmicro.core.repository.IUpdate;
import com.shopmicro.catalog.pizza.domain.Pizza;

import java.util.UUID;

/** Vista de "añadir pizza". Solo dominio, sin JPA. */
public interface IAddPizza extends IAdd<Pizza> { }

/** Vista de "leer pizza por id". get lanza si no existe. */
public interface IGetPizza extends IGet<Pizza, UUID> { }

/** Vista de "actualizar pizza". Hereda get de IUpdate → IGet. */
public interface IUpdatePizza extends IUpdate<Pizza, UUID> { }

/** Vista de "borrar pizza". Hereda get de IRemove → IGet. */
public interface IRemovePizza extends IRemove<Pizza, UUID> { }
```

> Estas cuatro interfaces son la **superficie pública** de la persistencia hacia la aplicación. Un
> handler de "crear pizza" inyecta `IAddPizza`; uno de "ver pizza", `IGetPizza`. Nadie depende de
> más de lo que usa.

### El repositorio Spring Data y la consulta con `JOIN FETCH`

Primero, la interfaz `JpaRepository` de Pizza, con una consulta que carga el agregado completo:

```java
package com.shopmicro.catalog.pizza.persistence;

import org.springframework.data.jpa.repository.JpaRepository;
import org.springframework.data.jpa.repository.Query;
import org.springframework.data.repository.query.Param;

import java.util.Optional;
import java.util.UUID;

public interface PizzaJpaRepository extends JpaRepository<PizzaJpa, UUID> {

    @Query("select p from PizzaJpa p left join fetch p.ingredients where p.id = :id")
    Optional<PizzaJpa> findByIdWithIngredients(@Param("id") UUID id);
}
```

### El repositorio concreto

`PizzaRepository` **compone** las vistas + las `*Jpa` del núcleo, cablea los tres puntos variables
(`jpa()`, `mapper()`, `domainType()`) y **sobrescribe `get`** para cargar el agregado completo con
`JOIN FETCH`.

```java
package com.shopmicro.catalog.pizza.persistence;

import com.shopmicro.core.domain.EntityNotFoundException;
import com.shopmicro.core.repository.IAddJpa;
import com.shopmicro.core.repository.IRemoveJpa;
import com.shopmicro.core.repository.IUpdateJpa;
import com.shopmicro.core.repository.Mapper;
import com.shopmicro.catalog.pizza.domain.Pizza;
import org.springframework.data.jpa.repository.JpaRepository;
import org.springframework.stereotype.Component;

import java.util.UUID;

/**
 * Repositorio de Pizza. add/update/remove provienen de los default methods de las
 * interfaces *Jpa; solo se sobrescribe get para cargar el agregado completo con
 * JOIN FETCH (evita que los ingredientes lleguen vacíos). Sobrescribir un default es
 * idéntico a sobrescribir un método heredado: el diseño lo permite cuando un caso lo necesita.
 */
@Component
public class PizzaRepository implements
        IAddPizza, IGetPizza, IUpdatePizza, IRemovePizza,
        IAddJpa<Pizza, UUID, PizzaJpa>,
        IUpdateJpa<Pizza, UUID, PizzaJpa>,
        IRemoveJpa<Pizza, UUID, PizzaJpa> {

    private final PizzaJpaRepository jpa;
    private final PizzaMapper mapper;

    public PizzaRepository(PizzaJpaRepository jpa, PizzaMapper mapper) {
        this.jpa = jpa;
        this.mapper = mapper;
    }

    // --- los tres puntos variables que exige IRepositoryJpa ---
    @Override public JpaRepository<PizzaJpa, UUID> jpa() { return jpa; }
    @Override public Mapper<Pizza, PizzaJpa> mapper()    { return mapper; }
    @Override public Class<Pizza> domainType()           { return Pizza.class; }

    // --- único método sobrescrito: carga el agregado completo ---
    @Override
    public Pizza get(UUID id) {
        PizzaJpa found = jpa.findByIdWithIngredients(id)   // ← JOIN FETCH de los ingredientes
            .orElseThrow(() -> new EntityNotFoundException(Pizza.class, id));

        return mapper.toDomain(found);
    }
}
```

`add`, `update` y `remove` **no se escriben aquí**: vienen de los default methods de
`IAddJpa`, `IUpdateJpa`, `IRemoveJpa`. Solo `get` necesita lógica propia (el `JOIN FETCH`), así que
es lo único que el repositorio sobrescribe.

## Registro de beans (Spring)

En Spring **no hay un fichero central de registro**: las anotaciones lo hacen. `@Component` sobre
`PizzaRepository`, `PizzaMapper`, `IngredientMapper` los registra como beans, y Spring Data genera el
bean de `PizzaJpaRepository` por extender `JpaRepository`.

Como `PizzaRepository` **implementa las cuatro vistas**, inyectar cualquiera de ellas resuelve al
**mismo bean** automáticamente — no hay que registrar una exposición por vista:

```java
@Component
public class PizzaRepository implements
        IAddPizza, IGetPizza, IUpdatePizza, IRemovePizza, /* ... */ { /* ... */ }
```

Y en un handler de un slice (estilo Vertical Slice del Módulo 3):

```java
package com.shopmicro.catalog.pizza.features.create;

import com.shopmicro.catalog.pizza.domain.Pizza;
import com.shopmicro.catalog.pizza.persistence.IAddPizza;
import org.springframework.stereotype.Service;

@Service
public class Handler {

    private final IAddPizza repository;   // ← solo la vista que necesita

    public Handler(IAddPizza repository) {
        this.repository = repository;
    }

    public Response handle(Command command) {
        Pizza pizza = Pizza.create(command.name(), command.description(), command.url(), /* ... */);
        repository.add(pizza);
        return new Response(pizza.getId());
    }
}
```

## Puntos clave

- **Clase de persistencia separada del dominio**, decisión deliberada: evita pelearse con Hibernate
  para reconstruir el agregado y mantiene el **dominio libre de JPA**.
- El dominio tiene **dos vías de construcción**: fábrica pública (`create`, emite evento) y
  **constructor protegido de hidratación** (no emite). El mapper usa la subclase `Hydration` para
  reconstruir **sin disparar eventos**.
- Las **vistas** (`IAddPizza`…) fijan los genéricos y son la superficie pública hacia la
  aplicación. Cada handler inyecta **solo la vista que usa** (ISP).
- `PizzaRepository` **compone** vistas + `*Jpa`; hereda `add/update/remove` de los default methods y
  **solo sobrescribe `get`** para el `JOIN FETCH` (agregado completo).
- Un único bean que implementa varias interfaces se inyecta por **cualquiera** de ellas: Spring lo
  resuelve sin registro extra.
