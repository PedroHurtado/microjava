# 2.3 Inyección de dependencias y el contenedor de beans

> **Módulo 2 — Fundamentos de Spring Boot para Microservicios**
> Duración estimada: ~40 min

---

## El problema que resuelve

Imagina un servicio que necesita acceder a datos. Sin inyección de dependencias, crearías las cosas a mano:

```java
// ❌ Acoplamiento fuerte: la clase CREA sus dependencias
public class CatalogService {
    private final ProductRepository repo =
        new ProductRepository(new DbConnection("..."));   // ¡rígido!

    public List<Product> getAll() {
        return repo.getAll();
    }
}
```

Problemas: imposible de testear (no puedes sustituir el repositorio por uno falso), repites la creación por todas partes, y si cambia el constructor de `ProductRepository` tienes que tocar mil sitios.

La **inyección de dependencias (DI)** invierte esto: en vez de que cada componente cree lo que necesita, **lo recibe ya construido desde fuera**. Es uno de los principios SOLID (la "D": *Dependency Inversion*).

> Spring está construido **alrededor** de la inyección de dependencias. El framework entero es, en su núcleo, un contenedor de DI (el `ApplicationContext`). No es un añadido: es la pieza central.

---

## El contenedor de beans

El **contenedor de Spring** (el `ApplicationContext`) es un registro de objetos gestionados, llamados **beans**. Le dices "esta clase es un componente que gestionas tú" y, cuando alguien necesite una de sus dependencias, el contenedor la construye y la entrega.

Registrar un bean se hace marcando la clase con una anotación estereotipo (`@Component`, `@Service`, `@Repository`, `@RestController`) — el **escaneo de componentes** las descubre automáticamente:

```java
// 1. Definimos una abstracción y su implementación
public interface CatalogService {
    List<Product> getAll();
}

@Service                                  // ← registrado como bean en el contenedor
public class CatalogServiceImpl implements CatalogService {
    public List<Product> getAll() { /* ... */ }
}

// 2. La PEDIMOS donde la necesitemos: por inyección de constructor
@RestController
@RequestMapping("/products")
public class ProductController {

    private final CatalogService service;

    public ProductController(CatalogService service) {   // ← Spring inyecta la instancia
        this.service = service;
    }

    @GetMapping
    public List<Product> getAll() {
        return service.getAll();
    }
}
```

El controlador ya no sabe **cómo** se construye `CatalogServiceImpl` ni qué dependencias tiene. Solo declara en su constructor "necesito un `CatalogService`" y el contenedor se lo da, resolviendo recursivamente toda la cadena de dependencias.

> **Preferencia clara:** usa **inyección por constructor** (no por campo con `@Autowired`). Hace las dependencias explícitas y obligatorias, permite campos `final` y facilita los tests. Si la clase tiene un único constructor, Spring lo usa sin necesidad de `@Autowired`.

## Los scopes (ciclos de vida)

Al registrar un bean, eliges cuánto vive cada instancia. Aquí hay una diferencia importante respecto a otros frameworks: **en Spring, el scope por defecto es `singleton`** (una sola instancia para todo el contenedor), y es el adecuado para la inmensa mayoría de beans, porque los servicios suelen ser **sin estado**.

| Scope | Anotación | Se crea... | Úsalo para |
|---|---|---|---|
| **singleton** | (por defecto) | **una sola instancia** para todo el contenedor | la mayoría: servicios, repositorios, clientes sin estado |
| **prototype** | `@Scope("prototype")` | una instancia **nueva cada vez** que se pide | beans con estado mutable no compartible |
| **request** | `@RequestScope` | una instancia **por petición HTTP** | estado ligado a la petición |
| **session** | `@SessionScope` | una por sesión de usuario | estado de sesión (poco habitual en microservicios) |

```java
@Service                       // singleton por defecto
public class CatalogService { }

@Component
@Scope("prototype")            // una instancia nueva en cada inyección
public class EmailFormatter { }

@Component
@RequestScope                  // una por petición HTTP
public class RequestContext { }
```

### Visualización de cuándo se crea cada uno

```
Petición HTTP #1          Petición HTTP #2
─────────────────         ─────────────────
Singleton: la MISMA  ◄──── Singleton: la MISMA
 instancia de siempre       (compartida entre
                            todas las peticiones)
Request:   UNA para        Request:   OTRA para
 toda la petición #1        toda la petición #2
Prototype: nueva cada      Prototype: nueva cada
 vez que se inyecta         vez que se inyecta
```

> Como los beans singleton se comparten entre todas las peticiones (y por tanto entre hilos), **deben ser *thread-safe*** — normalmente, manteniéndolos **sin estado mutable**. Esta es la razón #1 de bugs sutiles de concurrencia en Spring.

## El error clásico: inyectar un scope corto en uno largo

Esto te va a pasar, así que grábatelo. **Cuidado al inyectar un bean de vida corta (request/prototype) dentro de uno singleton.**

```java
// ❌ PELIGRO: un singleton que captura un bean request-scoped
@Service                          // singleton: vive toda la app
public class PriceCache {
    private final RequestContext ctx;   // ¡request scope! debería vivir una petición

    public PriceCache(RequestContext ctx) {
        this.ctx = ctx;   // queda "capturado": el singleton retiene
                          // la PRIMERA instancia para siempre
    }
}
```

El `RequestContext` (request) debería morir al acabar la petición, pero el singleton lo retiene vivo eternamente y reutiliza esa instancia obsoleta en todas las peticiones siguientes. Resultado: datos incorrectos y comportamiento errático difícil de depurar.

> **Solución en Spring:** usar un **proxy de scope** (`@Scope(value = "request", proxyMode = ScopedProxyMode.TARGET_CLASS)`) para que el singleton reciba un proxy que, en cada uso, resuelve la instancia correcta de la petición actual. Alternativamente, inyectar un `ObjectProvider<RequestContext>` y pedir la instancia bajo demanda. Spring, por suerte, **detecta y avisa** de muchas de estas situaciones al arrancar.

## Definir beans que no son tuyos: `@Configuration` + `@Bean`

Las anotaciones estereotipo sirven para tus propias clases. Para registrar objetos de librerías de terceros (que no puedes anotar), usas un método `@Bean` dentro de una clase `@Configuration`:

```java
@Configuration
public class CatalogConfig {

    @Bean
    public CatalogService catalogService(ProductRepository repo) {
        return new CatalogServiceImpl(repo);   // tú controlas la construcción
    }
}
```

Además, la mayoría de capacidades llegan **auto-configuradas** por los *starters*: añadir `spring-boot-starter-data-jpa`, `spring-boot-starter-security` o un cliente HTTP registra automáticamente todos sus beans, sin que tú escribas el cableado.

```java
// Ejemplos de capacidades que se auto-registran al añadir el starter correspondiente:
// - spring-boot-starter-data-jpa  → EntityManager, repositorios (Módulo 6)
// - spring-cloud-openfeign         → clientes HTTP declarativos (Módulo 4)
// - spring-boot-starter-security   → cadena de filtros de seguridad (Módulo 5)
// - spring-boot-starter-validation → Bean Validation (Módulo 3)
```

## Puntos clave

- La DI **invierte** la creación de dependencias: los componentes las **reciben**, no las crean.
- Spring **es** un contenedor de DI (`ApplicationContext`); registras beans con `@Component`/`@Service`/`@Repository` y el escaneo los descubre.
- Usa **inyección por constructor** (campos `final`, explícita, testeable).
- El scope por defecto es **`singleton`** (uno para todo el contenedor); existen también `prototype`, `request` y `session`.
- Los singletons se comparten entre hilos: **mantenlos sin estado mutable**.
- Evita capturar un bean de vida corta dentro de un singleton; usa **scoped proxies** u `ObjectProvider`.
- Para objetos de terceros usa `@Bean` en una `@Configuration`; los *starters* auto-registran casi todo.
