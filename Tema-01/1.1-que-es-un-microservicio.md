# 1.1 ¿Qué es un microservicio? Definición y características

> **Módulo 1 — Introducción a los Microservicios**
> Duración estimada: ~35 min

---

## Definición

Un **microservicio** es un servicio pequeño, autónomo y desplegable de forma independiente que modela una capacidad de negocio concreta. En lugar de construir una única aplicación grande que lo hace todo (un *monolito*), descomponemos el sistema en un conjunto de servicios que colaboran entre sí mediante mensajes a través de la red.

No existe una definición oficial universal, pero la más citada es la de Sam Newman:

> *"Los microservicios son servicios pequeños y autónomos que trabajan juntos, modelados en torno a un dominio de negocio."*

La palabra clave no es "pequeño" (eso es relativo y a menudo se malinterpreta), sino **autónomo**: cada servicio debe poder desarrollarse, desplegarse, escalarse y fallar de forma independiente del resto.

---

## Características esenciales

### 1. Desplegable de forma independiente

Es **la** característica que define a un microservicio. Si para desplegar el servicio A tienes que desplegar también el B y el C a la vez, no tienes microservicios: tienes un monolito distribuido (lo peor de ambos mundos).

```
✅ Microservicios          ❌ Monolito distribuido
┌─────┐ ┌─────┐ ┌─────┐    ┌───────────────────┐
│  A  │ │  B  │ │  C  │    │   A  +  B  +  C    │
└─────┘ └─────┘ └─────┘    └───────────────────┘
deploy  deploy  deploy      un único deploy
 indep.  indep.  indep.     acoplado
```

### 2. Modelado en torno a una capacidad de negocio

Un microservicio se organiza alrededor de **qué hace para el negocio**, no alrededor de una capa técnica. En ShopMicro tendremos `catalog`, `basket`, `ordering`... no tendremos un servicio "de base de datos" y otro "de lógica" y otro "de presentación".

### 3. Posee sus propios datos

Cada servicio gestiona su propia base de datos. **Ningún otro servicio accede directamente a ella.** Esto es el patrón *database per service* (lo veremos a fondo en el Módulo 6). Si Ordering necesita un dato de Catalog, se lo **pide** a Catalog; no entra en su base de datos.

### 4. Se comunica por la red

Los servicios hablan entre sí mediante mensajes: llamadas HTTP/REST, gRPC, o mensajería asíncrona (todo esto es el Módulo 4). Esto introduce latencia y posibilidad de fallo que en un monolito no existían — es el coste a pagar.

### 5. Autonomía de equipo y tecnología

Cada servicio puede, en teoría, usar el lenguaje, framework o base de datos que mejor le convenga, y ser propiedad de un equipo distinto. En la práctica se suele estandarizar para reducir carga cognitiva, pero la *posibilidad* existe.

---

## ¿Cómo de "micro" es un microservicio?

Esta es la pregunta equivocada. El tamaño no se mide en líneas de código. Una heurística mucho más útil:

> Un microservicio debería ser **lo bastante pequeño como para caber en la cabeza de una persona**, y debería poder **reescribirse en un par de semanas** si hiciera falta.

La regla de oro real es el **bounded context** (contexto delimitado) del Domain-Driven Design: un servicio debería corresponderse con un límite natural del dominio de negocio. Lo veremos en el punto 1.4.

---

## Un primer vistazo a código

Un microservicio en Spring Boot, en su forma más mínima, no es nada intimidante. Esto ya es un microservicio funcional:

```java
// CatalogApplication.java — un microservicio "Catalog" mínimo
@SpringBootApplication
@RestController
public class CatalogApplication {

    // Datos de ejemplo en memoria (en el curso usaremos PostgreSQL)
    record Producto(int id, String nombre, double precio) {}

    @GetMapping("/products")
    List<Producto> products() {
        return List.of(
            new Producto(1, "Teclado mecánico", 79.99),
            new Producto(2, "Ratón inalámbrico", 29.99)
        );
    }

    public static void main(String[] args) {
        SpringApplication.run(CatalogApplication.class, args);
    }
}
```

Con `mvn spring-boot:run`, esto ya expone un endpoint HTTP que responde con productos. A lo largo del curso lo iremos convirtiendo en un servicio de verdad: con base de datos, validación, seguridad, observabilidad y desplegado en contenedores.

---

## Puntos clave

- Un microservicio es **autónomo** y **desplegable de forma independiente** — esa es la característica central.
- Se modela en torno a una **capacidad de negocio**, no a una capa técnica.
- **Posee sus propios datos**; nadie más toca su base de datos.
- Se comunica por la **red**, lo que añade latencia y posibilidad de fallo.
- El tamaño no es lo importante: lo es el **límite del dominio** (*bounded context*).
