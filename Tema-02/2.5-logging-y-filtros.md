# 2.5 Logging estructurado y la cadena de filtros

> **Módulo 2 — Fundamentos de Spring Boot para Microservicios**
> Duración estimada: ~45 min

---

Este punto cubre dos temas que se tocan: la **cadena de filtros e interceptores** (cómo se procesa cada petición) y el **logging estructurado** (cómo dejamos rastro de lo que pasa). Ambos son la base de la observabilidad que profundizaremos en el Módulo 7.

---

## Parte 1: La cadena de filtros (filter chain)

### Qué es

Cada petición HTTP que entra atraviesa una **cadena de componentes** —los *filtros* de Servlet (y, más arriba, los *interceptores* de Spring MVC)— uno tras otro, antes de llegar a tu controlador. Y la respuesta los vuelve a atravesar en sentido inverso. Es como una cebolla:

```
Petición →  [Logging] → [Seguridad] → [tu controller]
                                              │
Respuesta ← [Logging] ← [Seguridad] ←─────────┘
```

Cada filtro puede: hacer algo con la petición, decidir pasarla al siguiente (`chain.doFilter`), o **cortocircuitar** (responder sin llamar al siguiente).

### El orden IMPORTA (mucho)

Los filtros se ejecutan en un **orden** definido por su prioridad (`@Order`, o `FilterRegistrationBean.setOrder`). Esto es fuente de bugs si te equivocas. La **seguridad de Spring** es, por dentro, una cadena de filtros cuidadosamente ordenada:

```
1. Filtro de logging / correlación
2. Filtros de Spring Security:
   - Autenticación  → ¿quién eres?    (se ejecuta primero)
   - Autorización   → ¿puedes hacer esto?   (después)
3. El DispatcherServlet → enruta a tu controller
```

> Dentro de Spring Security, la **autenticación** (saber quién eres) siempre se procesa **antes** que la **autorización** (saber qué puedes hacer); el framework ordena su cadena por ti. Pero si escribes tus propios filtros, **tú** eres responsable de su `@Order`. **Regla mnemotécnica:** autenticar siempre va antes que autorizar.

### Escribir tu propio filtro

Para lógica transversal (que afecta a todas las peticiones). Ejemplo: medir cuánto tarda cada petición.

```java
@Component
@Order(1)
public class TimingFilter implements Filter {

    @Override
    public void doFilter(ServletRequest req, ServletResponse res, FilterChain chain)
            throws IOException, ServletException {
        long start = System.nanoTime();
        chain.doFilter(req, res);                 // ← pasa al siguiente filtro/controller
        long ms = (System.nanoTime() - start) / 1_000_000;
        var path = ((HttpServletRequest) req).getRequestURI();
        System.out.println(path + " tardó " + ms + "ms");
    }
}
```

En el Módulo 3 usaremos un concepto hermano para el **manejo global de errores**: un `@ControllerAdvice` con `@ExceptionHandler` que captura excepciones y devuelve un `ProblemDetail` (el estándar RFC 7807, integrado en Spring 6).

## Parte 2: Logging estructurado

### Logging tradicional vs. estructurado

El logging "de toda la vida" produce **texto plano**:

```
Producto 42 no encontrado para el usuario juan
```

El **logging estructurado** produce **datos con campos nombrados**: el mensaje y sus parámetros se guardan por separado, de forma que luego puedes **buscar y filtrar** por esos campos.

En microservicios esto es **crítico**: con decenas de servicios escupiendo logs a un sistema central, necesitas poder consultar *"dame todos los logs donde productId=42, de cualquier servicio"*. Con texto plano eso es imposible; con logs estructurados es una consulta trivial.

### Cómo se usa: SLF4J + Logback

Spring Boot trae **SLF4J** como API de logging y **Logback** como implementación por defecto. El logger se obtiene por clase:

```java
public class ProductController {
    private static final Logger log = LoggerFactory.getLogger(ProductController.class);
    // ...
}
```

#### Mensajes con marcadores `{}` (¡no concatenación!)

```java
log.warn("Producto {} no encontrado para usuario {}", 42, "juan");
```

> **Detalle importante:** usa **marcadores de posición** `{}`, **no concatenación ni interpolación** de strings. Es decir, `log.warn("... {}", id)` y **no** `log.warn("... " + id)`. La primera forma es más eficiente (solo construye el texto si el nivel está activo) y evita errores.

#### Campos nombrados de verdad: la API fluida de SLF4J 2

Los marcadores `{}` son posicionales: ahorran coste, pero no producen *campos nombrados* por sí solos. Para logging estructurado real (campos consultables), usa la API fluida con `addKeyValue`:

```java
log.atWarn()
   .setMessage("Producto no encontrado")
   .addKeyValue("productId", 42)
   .addKeyValue("user", "juan")
   .log();
```

Esto adjunta los pares `productId=42` y `user=juan` como **campos estructurados**, listos para indexar y filtrar en un sistema central.

### Niveles de log

De menos a más grave. Configurables por entorno (en `application.yml`):

| Nivel | Cuándo usarlo |
|---|---|
| `TRACE` | Detalle finísimo de depuración (nunca en producción) |
| `DEBUG` | Información de depuración para desarrollo |
| `INFO` | Flujo normal: "petición recibida", "pedido creado" |
| `WARN` | Algo inesperado pero recuperable |
| `ERROR` | Un fallo en la operación actual (incluye los casos críticos) |

```yaml
logging:
  level:
    root: INFO
    org.springframework.web: WARN
```

### Contexto compartido: MDC

El **MDC** (*Mapped Diagnostic Context*) adjunta datos comunes a todos los logs que ocurran dentro de un bloque — útil para correlacionar:

```java
try (var ignored = MDC.putCloseable("orderId", orderId)) {
    log.info("Validando líneas");    // llevará orderId adjunto
    log.info("Calculando total");     // este también
}
```

Esto enlaza con la idea de **correlación entre servicios** del Módulo 7: cuando una petición salta entre servicios, queremos un identificador común (*trace id*) que aparezca en todos los logs de todos los servicios involucrados. **Micrometer Tracing** (con OpenTelemetry) añade automáticamente `traceId` y `spanId` al MDC.

### Hacia producción: salida a JSON

En producción se envían los logs en formato **JSON** a sistemas centralizados (Elasticsearch, Loki, Seq...). Spring Boot 3.4+ trae **logging estructurado integrado**: basta con configurar el formato, sin librerías extra.

```yaml
# Con esto, los logs de consola salen como JSON estructurado (Elastic Common Schema),
# listos para ingerir en un sistema central de búsqueda.
logging:
  structured:
    format:
      console: ecs
```

(También es muy común usar `logstash-logback-encoder` para el mismo fin en versiones anteriores.)

## Cómo encajan ambos temas

El logging y los filtros se combinan en un patrón omnipresente: un filtro que registra cada petición entrante y saliente, sentando las bases de la **observabilidad**. Spring Boot ya trae uno (`CommonsRequestLoggingFilter`), y en el Módulo 7 lo ampliaremos con **trazas distribuidas (OpenTelemetry / Micrometer Tracing)** para seguir una petición a través de todo ShopMicro.

---

## Puntos clave

- Cada petición atraviesa la **cadena de filtros**; el **orden (`@Order`) importa**.
- Autenticación siempre **antes** que autorización (Spring Security ordena su cadena por ti).
- Un filtro puede continuar (`chain.doFilter`) o **cortocircuitar** la petición.
- El **logging estructurado** guarda campos nombrados, no texto plano — esencial para buscar entre muchos servicios.
- Usa marcadores `{}` (nunca concatenación) y la API fluida `addKeyValue` para campos estructurados.
- Obtén el logger con `LoggerFactory.getLogger(...)`; elige el **nivel** adecuado; usa **MDC** para correlacionar.
- En producción: salida **JSON** (logging estructurado de Spring Boot 3.4+ o `logstash-logback-encoder`) a un sistema central (Módulo 7).
