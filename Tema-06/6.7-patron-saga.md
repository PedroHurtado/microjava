## Qué es una Saga

Una **Saga** es una operación de negocio que cruza varios servicios, implementada como una
**secuencia de transacciones locales**. Cada paso:

1. Hace su trabajo en **su propia base de datos** (transacción ACID local, eso sí lo tenemos).
2. Publica un **evento** que dispara el siguiente paso.

Y, lo esencial: cada paso tiene una **acción compensatoria** que **deshace** lo que hizo, por si un
paso posterior falla. La Saga no tiene "rollback" global (no hay transacción global); tiene
**compensaciones** que revierten paso a paso.

```
   PASO 1            PASO 2             PASO 3
   crear Order  ──▶  reservar stock ──▶ cobrar pago
   (Pendiente)       (Catalog)          (Payments)

   Si el PASO 3 falla, se ejecutan las COMPENSACIONES en orden inverso:
   cancelar Order ◀── liberar stock ◀── (pago no llegó a cargarse)
```

---

## Dos estilos: coreografía vs. orquestación

| | **Coreografía** | **Orquestación** |
|---|---|---|
| Quién decide | Cada servicio reacciona a eventos | Un **orquestador** central dirige los pasos |
| Acoplamiento | Bajo (servicios no se conocen) | El orquestador conoce a todos |
| Visibilidad del flujo | Difusa (está "repartida") | Centralizada y explícita |
| Cuándo | Flujos cortos, pocos pasos | Flujos largos o muy condicionales |

En este módulo vemos **coreografía**, que es la más natural cuando ya tienes mensajería por
eventos: nadie manda, cada servicio **escucha** los eventos que le importan y **publica** los suyos.

```
   CHOREOGRAPHY (coreografía) — nadie manda, todos reaccionan

   Ordering        publica  OrderCreated
                              │
                              ▼
   Catalog   escucha OrderCreated → reserva stock → publica StockReserved
                                                      │
                                                      ▼
   Payments  escucha StockReserved → cobra → publica PaymentCompleted
                                              │
                                              ▼
   Ordering  escucha PaymentCompleted → marca Order = Confirmado
```

Y el camino de fallo:

```
   Payments  escucha StockReserved → el pago FALLA → publica PaymentFailed
                                                      │
                          ┌───────────────────────────┤
                          ▼                            ▼
   Catalog escucha PaymentFailed         Ordering escucha PaymentFailed
   → libera stock (compensación)         → marca Order = Cancelado (compensación)
```

## Sobre la mensajería en este módulo

La mensajería real (RabbitMQ + Spring AMQP, broker, colas) la montaremos cuando lleguemos a ese
punto. Para **ver la Saga funcionando ahora sin infraestructura**, usamos el **bus de eventos en
memoria** que trae Spring: el `ApplicationEventPublisher` (publicar) y `@EventListener` (suscribirse).
Es síncrono y vive dentro del propio proceso, suficiente para ilustrar el patrón en el aula.

> El objetivo aquí es que el alumnado **entienda el flujo de eventos y compensaciones**, no
> aprender un broker. El día que enchufemos RabbitMQ, solo cambia **quién transporta el evento**;
> la lógica de la Saga es la misma.

## Esqueleto: el bus en memoria

Spring ya aporta el bus: no hay que construir nada. Un servicio publica con
`ApplicationEventPublisher` y los oyentes reaccionan con `@EventListener`. Si quisiéramos una fachada
explícita —para que mañana cambiar a RabbitMQ sea un único punto— bastaría:

```java
package com.shopmicro.core.messaging;

import org.springframework.context.ApplicationEventPublisher;
import org.springframework.stereotype.Component;

/**
 * Bus de eventos en memoria para el aula. Envuelve el ApplicationEventPublisher de Spring.
 * Publicar = entregar a los @EventListener del proceso. El día de RabbitMQ, solo cambia esta clase.
 */
@Component
public class InMemoryEventBus {

    private final ApplicationEventPublisher publisher;

    public InMemoryEventBus(ApplicationEventPublisher publisher) {
        this.publisher = publisher;
    }

    public void publish(Object event) {
        publisher.publishEvent(event);
    }
}
```

## Esqueleto: los eventos de la Saga

package com.shopmicro.ordering.saga;

## Esqueleto: un paso con su compensación (Catalog reserva stock)

```java
package com.shopmicro.catalog.saga;

import com.shopmicro.core.messaging.InMemoryEventBus;
import com.shopmicro.ordering.saga.OrderCreated;
import com.shopmicro.ordering.saga.PaymentFailed;
import com.shopmicro.ordering.saga.StockReserved;
import org.springframework.context.event.EventListener;
import org.springframework.stereotype.Component;

/**
 * Paso de la Saga en Catalog: reacciona a OrderCreated reservando stock, y a
 * PaymentFailed liberándolo (compensación). Esqueleto didáctico.
 */
@Component
public class CatalogSagaHandler {

    private final InMemoryEventBus bus;
    // private final IReserveStock stock;   // puerto del propio servicio

    public CatalogSagaHandler(InMemoryEventBus bus) {
        this.bus = bus;
    }

    @EventListener
    public void on(OrderCreated e) {
        // 1. transacción LOCAL: reservar stock en la BD de Catalog
        // stock.reserve(e.productId(), e.quantity());
        // 2. publicar el evento que dispara el siguiente paso
        bus.publish(new StockReserved(e.orderId(), e.productId(), e.quantity()));
    }

    @EventListener
    public void on(PaymentFailed e) {
        // COMPENSACIÓN: deshacer la reserva de este pedido
        // stock.release(e.orderId());
    }
}
```

## Esqueleto: el cierre de la Saga (Ordering)

```java
package com.shopmicro.ordering.saga;

import org.springframework.context.event.EventListener;
import org.springframework.stereotype.Component;

import java.util.UUID;

@Component
public class OrderingSagaHandler {

    // private final IUpdateOrder orders;

    @EventListener
    public void on(PaymentCompleted e) {
        markConfirmed(e.orderId());   // estado final OK
    }

    @EventListener
    public void on(PaymentFailed e) {
        markCancelled(e.orderId());   // compensación: cancelar
    }

    private void markConfirmed(UUID orderId) { /* Order = Confirmado */ }
    private void markCancelled(UUID orderId) { /* Order = Cancelado  */ }
}
```

> Es **esqueleto deliberado**: los puertos (`IReserveStock`, `IUpdateOrder`) se completan al montar
> la mensajería. Lo que el alumnado debe llevarse es el **patrón**: pasos locales + eventos +
> compensaciones en orden inverso.

---

## Idempotencia: la regla que no se puede olvidar

En un sistema de mensajes, un evento **puede llegar más de una vez**. Si "reservar stock" se
ejecuta dos veces por un `OrderCreated` duplicado, restas el doble. Por eso **cada paso de una Saga
debe ser idempotente**: aplicarlo N veces tiene el mismo efecto que aplicarlo una. Se suele
conseguir registrando qué `orderId` ya se procesó antes de actuar.

---

## Puntos clave

- Una **Saga** parte una operación distribuida en **transacciones locales** encadenadas por
  **eventos**, cada una con su **compensación**.
- No hay rollback global: si un paso falla, se ejecutan las **compensaciones en orden inverso**.
- **Coreografía** = cada servicio reacciona a eventos (bajo acoplamiento); **orquestación** = un
  coordinador central dirige.
- En el aula usamos el **bus en memoria de Spring** (`ApplicationEventPublisher` + `@EventListener`);
  con un broker real solo cambia el transporte.
- **Cada paso debe ser idempotente**: los eventos pueden duplicarse.
