# Módulo 4 · Lección 4.3 — HTTP Interface: clientes HTTP declarativos por interface

> **Duración estimada:** 45 min
> **Servicio del proyecto:** `ordering-service` consumiendo `catalog-service` (mismo caso que 4.2)
> **Requisitos:** Lección 4.2 (cliente tipado con `RestClient`)

---

## 1. La idea: describir, no implementar

En 4.2 escribimos a mano la construcción de la URI, la comprobación del status code y la
deserialización. La **HTTP Interface** de Spring invierte el enfoque: tú **declaras** una
interface con anotaciones que describen las peticiones HTTP, y Spring **genera** la
implementación por ti (un proxy en tiempo de ejecución).

Es exactamente el mismo principio que ya aplicas con las interfaces de repositorio de Spring
Data: defines el *qué*, no el *cómo*. Aquí el "cómo" (montar la petición, parsear la respuesta)
lo pone Spring.

> **Alternativa habitual.** Spring Cloud OpenFeign ofrece el mismo estilo declarativo con
> `@FeignClient`. Usamos la **HTTP Interface nativa de Spring** (`@HttpExchange`) porque no
> necesita dependencias extra y se apoya directamente en el `RestClient` de 4.2 y en la
> resiliencia de 4.4. Si tu proyecto ya usa Spring Cloud, Feign es una opción perfectamente
> válida con la misma filosofía.

---

## 2. El antes y el después

**Antes (4.2):** una clase `HttpCatalogClient` con un método que construye la petición, mira el
`NOT_FOUND`, deserializa. ~15 líneas por endpoint.

**Después (HTTP Interface):** una interface. Cero líneas de implementación.

```java
package com.shopmicro.ordering.infrastructure.catalog;

import com.shopmicro.ordering.application.catalog.ProductInfo;
import org.springframework.web.service.annotation.GetExchange;
import org.springframework.web.bind.annotation.PathVariable;

public interface CatalogApi {

    @GetExchange("/products/{productId}")
    ProductInfo getProduct(@PathVariable int productId);
}
```

La anotación `@GetExchange("/products/{productId}")` mapea el parámetro `productId` al segmento
de la URI por nombre. Spring genera el código que hace la llamada y deserializa la respuesta
JSON a `ProductInfo`.

> **Manejo del 404.** Por defecto el proxy propaga una excepción ante un status de error. Para
> que un 404 se traduzca a `null` (o a una excepción de dominio), lo resolvemos en el adaptador
> (punto 4), capturando `HttpClientErrorException.NotFound`.

---

## 3. ¿Y nuestro contrato `CatalogClient`?

Aquí hay una decisión de diseño importante. Tenemos **dos interfaces** que parecen solaparse:

- `CatalogClient` — el contrato **de la capa de aplicación**, expresado en el lenguaje de
  Ordering (definido en 4.2).
- `CatalogApi` — el contrato **técnico de la HTTP Interface**, que describe los endpoints HTTP
  de Catalog.

**No son lo mismo y conviene no fusionarlos.** `CatalogClient` pertenece al dominio de la
aplicación: Ordering pide "dame la info de este producto" sin saber que por debajo hay HTTP.
`CatalogApi` es un detalle de infraestructura: la forma de la API REST de Catalog. Mantener
ambos preserva la regla de dependencias (la aplicación no depende del cliente HTTP).

El puente entre ambos es un **adaptador**:

```java
package com.shopmicro.ordering.infrastructure.catalog;

import com.shopmicro.ordering.application.catalog.CatalogClient;
import com.shopmicro.ordering.application.catalog.ProductInfo;
import org.springframework.stereotype.Component;
import org.springframework.web.client.HttpClientErrorException;

// Implementa el contrato de aplicación apoyándose en el cliente declarativo.
@Component
public class CatalogClientAdapter implements CatalogClient {

    private final CatalogApi api;

    public CatalogClientAdapter(CatalogApi api) {
        this.api = api;
    }

    @Override
    public ProductInfo getProduct(int productId) {
        try {
            return api.getProduct(productId);
        } catch (HttpClientErrorException.NotFound ex) {
            return null;
        }
    }
}
```

> **¿Es imprescindible el adaptador?** No siempre. Si no te importa que la capa de aplicación
> conozca la `CatalogApi`, puedes inyectarla directamente en los handlers. Pero en un curso que
> enseña capas limpias, el adaptador muestra **dónde cortar la dependencia**: el detalle HTTP
> queda confinado en infraestructura. Decide según cuánta pureza quieras en cada servicio.

## 4. Registro como bean

La HTTP Interface se construye con un `HttpServiceProxyFactory` apoyado en el `RestClient` de
4.2. Esto es **clave para 4.4**: la resiliencia se engancha en ese `RestClient`.

```java
package com.shopmicro.ordering.infrastructure.catalog;

import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.web.client.RestClient;
import org.springframework.web.client.support.RestClientAdapter;
import org.springframework.web.service.invoker.HttpServiceProxyFactory;

@Configuration
public class CatalogApiConfig {

    @Bean
    CatalogApi catalogApi(RestClient catalogRestClient) {     // el RestClient configurado en 4.2
        var adapter = RestClientAdapter.create(catalogRestClient);
        return HttpServiceProxyFactory.builderFor(adapter)
                .build()
                .createClient(CatalogApi.class);
    }
}
```

El handler de `CreateOrder` **no cambia ni una línea** respecto a 4.2: sigue pidiendo
`CatalogClient`. Hemos cambiado la implementación por debajo sin tocar a quien la usa. Eso es
trabajar contra abstracciones.

## 5. Qué ganamos y qué cuesta

| | **Cliente tipado manual (4.2)** | **HTTP Interface (4.3)** |
|---|---|---|
| Líneas por endpoint | ~15 | 1 (la firma) |
| Construcción de URI | a mano | por anotaciones |
| Deserialización | explícita | automática |
| Dependencia extra | ninguna | ninguna (parte de Spring Web) |
| Control fino del request | total | alto, vía anotaciones/opciones |

La HTTP Interface brilla cuando consumes **muchos** endpoints de un servicio. Para una sola
llamada trivial, el cliente tipado manual puede bastar. En ShopMicro, donde Ordering hablará
con varios servicios, el estilo declarativo paga su coste rápido.

---

## Puntos clave

- La HTTP Interface te deja **declarar** la API como una interface con anotaciones
  (`@HttpExchange` / `@GetExchange`); Spring **genera** la implementación. Mismo principio que
  los repositorios de Spring Data: defines el *qué*.
- Separa **dos contratos**: `CatalogClient` (lenguaje de la aplicación) y `CatalogApi`
  (endpoints HTTP). Un **adaptador** los une y **confina el detalle HTTP en infraestructura**.
- El proxy se construye sobre el **`RestClient`** de 4.2: la base sobre la que 4.4 montará la
  resiliencia.
- El **consumidor (`CatalogClient`) no se entera del cambio**: cambiamos la implementación sin
  tocar los handlers.
- Gana cuanto **más endpoints** consumes; para una llamada aislada, el cliente tipado manual es
  defendible. Spring Cloud OpenFeign es la alternativa popular con la misma filosofía.
