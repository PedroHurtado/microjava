# Módulo 4 · Lección 4.5 — gRPC: contratos con Protobuf y comunicación de alto rendimiento

> **Duración estimada:** 55 min
> **Servicio del proyecto:** comunicación interna entre servicios de ShopMicro
> **Requisitos:** Lecciones 4.1–4.2 (síncrono, HTTP)

---

## 1. Dónde encaja gRPC

REST sobre HTTP/JSON es la lengua franca de las APIs públicas: universal, legible, *cacheable*,
perfecta de cara al cliente final. Pero **entre servicios internos**, donde el emisor y el
receptor son tuyos y las llamadas son muy frecuentes, JSON de texto y HTTP/1.1 dejan
rendimiento sobre la mesa.

**gRPC** es una alternativa para esa comunicación **interna, de alto rendimiento y contrato
estricto**: usa HTTP/2 y serializa con **Protobuf** (binario y compacto en lugar de texto).

---

## 2. La idea: el contrato primero (`.proto`)

En REST el contrato es implícito (documentado, idealmente, con OpenAPI). En gRPC el contrato es
**un fichero `.proto`** del que se **genera código** para cliente y servidor. Una sola fuente de
verdad, fuertemente tipada, compartida por ambos extremos. El `.proto` es **independiente del
lenguaje**: el mismo fichero genera Java, Kotlin, Go o cualquier otro.

```protobuf
syntax = "proto3";

option java_package = "com.shopmicro.catalog.grpc";
option java_multiple_files = true;

package catalog;

// El servicio y sus operaciones.
service CatalogGrpc {
  rpc GetProduct (ProductRequest) returns (ProductReply);
}

message ProductRequest {
  int32 product_id = 1;
}

message ProductReply {
  int32 id = 1;
  string name = 2;
  double price = 3;
  bool is_available = 4;
}
```

Los números (`= 1`, `= 2`) son las **etiquetas de campo**: identifican cada campo en el formato
binario. Son la clave de la **compatibilidad**: puedes añadir campos nuevos con etiquetas nuevas
sin romper a los consumidores antiguos, que simplemente los ignoran.

> **Generación de código en Java.** El plugin `protobuf-maven-plugin` (o su equivalente en
> Gradle) compila el `.proto` y genera las clases de mensajes (`ProductRequest`,
> `ProductReply`) y la clase base del servicio (`CatalogGrpcGrpc.CatalogGrpcImplBase`) más el
> *stub* de cliente. El `.proto` vive en `src/main/proto`.

## 3. El servidor (en catalog-service)

El generador de gRPC crea una clase base abstracta a partir del `.proto`; tú heredas e
implementas la lógica. Con el *starter* de gRPC para Spring Boot, basta anotar la implementación
con `@GrpcService`.

```java
package com.shopmicro.catalog.grpc;

import io.grpc.Status;
import io.grpc.stub.StreamObserver;
import net.devh.boot.grpc.server.service.GrpcService;

@GrpcService
public class CatalogGrpcService extends CatalogGrpcGrpc.CatalogGrpcImplBase {

    private final ProductRepository repository;

    public CatalogGrpcService(ProductRepository repository) {
        this.repository = repository;
    }

    @Override
    public void getProduct(ProductRequest request, StreamObserver<ProductReply> responseObserver) {
        var product = repository.findById(request.getProductId()).orElse(null);

        if (product == null) {
            responseObserver.onError(Status.NOT_FOUND
                    .withDescription("Producto no encontrado")
                    .asRuntimeException());
            return;
        }

        var reply = ProductReply.newBuilder()
                .setId(product.getId())
                .setName(product.getName())
                .setPrice(product.getPrice().doubleValue())
                .setIsAvailable(product.getStock() > 0)
                .build();

        responseObserver.onNext(reply);
        responseObserver.onCompleted();
    }
}
```

Configuración del servidor:

```yaml
# application.yml de catalog-service
grpc:
  server:
    port: 9090
```

## 4. El cliente (en ordering-service)

El mismo `.proto` genera un *stub* fuertemente tipado. El *starter* lo inyecta con `@GrpcClient`:

```java
package com.shopmicro.ordering.infrastructure.catalog;

import net.devh.boot.grpc.client.inject.GrpcClient;
import org.springframework.stereotype.Component;

@Component
public class CatalogGrpcAdapter {

    @GrpcClient("catalog")
    private CatalogGrpcGrpc.CatalogGrpcBlockingStub catalogStub;

    public ProductReply getProduct(int productId) {
        var request = ProductRequest.newBuilder()
                .setProductId(productId)
                .build();
        return catalogStub.getProduct(request);   // llamada bloqueante, petición/respuesta
    }
}
```

```yaml
# application.yml de ordering-service
grpc:
  client:
    catalog:
      address: static://catalog-service:9090
      negotiation-type: plaintext
```

> **Resiliencia.** El *stub* gRPC se puede envolver con Resilience4j igual que en 4.4
> (`@Retry`, `@CircuitBreaker` sobre el método del adaptador, o interceptores de cliente gRPC).
> La misma filosofía de defensas se aplica.

## 5. REST vs. gRPC: cuándo cada uno

Esta es la decisión que los alumnos deben saber tomar. No es "gRPC mejor que REST"; es **cada
uno en su sitio**.

| Criterio | **REST / JSON** | **gRPC / Protobuf** |
|---|---|---|
| **Formato** | texto (JSON), legible | binario, compacto |
| **Transporte** | HTTP/1.1 (o 2) | HTTP/2 (multiplexado) |
| **Contrato** | implícito / OpenAPI | explícito (`.proto`), código generado |
| **Rendimiento** | bueno | superior (menos bytes, menos parsing) |
| **Streaming** | limitado | bidireccional nativo |
| **Navegador / público** | universal | requiere gRPC-Web / proxy |
| **Legibilidad / debug** | alta (curl, navegador) | baja (binario) |
| **Encaje ideal** | **APIs públicas, cara al cliente** | **servicio↔servicio interno, alto volumen** |

### Regla práctica para ShopMicro

- **El Gateway hacia el exterior y las APIs que consume el frontend** → REST. Universal y
  depurable.
- **Llamadas internas frecuentes y sensibles a latencia** entre servicios → candidatas a gRPC.
  Por ejemplo, si Ordering consultara Catalog miles de veces por minuto, gRPC reduce bytes y CPU
  frente a JSON.
- **Notificación de hechos** ("pedido creado") → ni REST ni gRPC: **asíncrono** por el broker
  (4.6–4.8). gRPC sigue siendo petición/respuesta síncrona; no sustituye a la mensajería.

> **Matiz honesto.** gRPC aporta de verdad cuando el volumen o la latencia lo justifican. Para
> una malla pequeña como ShopMicro en un curso, REST + HTTP Interface + Resilience4j cubre casi
> todo con menos fricción. Presentamos gRPC para que los alumnos **sepan elegirlo cuando toque**,
> no para usarlo en todas partes. Meter gRPC donde no hace falta es complejidad sin retorno.

---

## Puntos clave

- **gRPC** = HTTP/2 + **Protobuf** (binario). Pensado para comunicación **interna
  servicio↔servicio** de **alto rendimiento**.
- El **contrato va primero**: un `.proto` del que se **genera** cliente y servidor tipados. Una
  sola fuente de verdad, **independiente del lenguaje**.
- Las **etiquetas de campo** (`= 1`, `= 2`) dan **compatibilidad**: añadir campos no rompe a los
  consumidores antiguos.
- En Spring Boot, el servidor se marca con **`@GrpcService`** y el cliente se inyecta con
  **`@GrpcClient`**; el *stub* admite la misma **resiliencia de Resilience4j** que la HTTP
  Interface.
- **REST para lo público y depurable; gRPC para lo interno de alto volumen; broker para
  notificar hechos.** gRPC sigue siendo síncrono: **no** sustituye a la mensajería asíncrona.
- En ShopMicro, gRPC se presenta para **saber elegirlo**, no como reemplazo universal de REST.
