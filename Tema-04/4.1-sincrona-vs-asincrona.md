# Módulo 4 · Lección 4.1 — Comunicación síncrona vs. asíncrona: el gran dilema

> **Duración estimada:** 40 min
> **Servicio del proyecto:** transversal (visión de toda la malla de ShopMicro)
> **Requisitos:** Módulos 1–3; tener clara la anatomía de ShopMicro

---

## 1. El problema que aparece al partir el monolito

En un monolito, "que el módulo de pedidos pregunte al de catálogo si un producto existe" es
una llamada a método: instantánea, transaccional, imposible de fallar por la red. Al partir
ShopMicro en servicios, esa misma pregunta cruza un cable. Y todo cambia.

La decisión central de este módulo: **cuándo dos servicios deben hablarse de forma síncrona
(te pregunto y espero tu respuesta) y cuándo de forma asíncrona (te lanzo un mensaje y sigo
con lo mío).** No hay una respuesta única; hay un dilema con trade-offs que debes saber leer.

---

## 2. Las dos formas, en una frase

| | **Síncrona** | **Asíncrona** |
|---|---|---|
| **Metáfora** | llamada de teléfono | dejar un mensaje en el buzón |
| **El emisor...** | espera bloqueado la respuesta | sigue trabajando, no espera |
| **Acoplamiento temporal** | alto: ambos vivos a la vez | bajo: el receptor puede estar caído |
| **Mecanismo típico** | HTTP/REST, gRPC | broker de mensajes (cola, topic) |
| **Estilo** | petición/respuesta | publicar/suscribir, eventos |

---

## 3. El coste oculto de lo síncrono: acoplamiento temporal

Cuando `ordering-service` llama por HTTP a `catalog-service` para validar un producto **y
espera la respuesta para continuar**, está atado a Catalog en el tiempo: si Catalog está
caído o lento, Ordering está caído o lento. Acabas de convertir dos fallos independientes en
uno solo.

```
Petición de crear pedido
        │
   ordering-service ──HTTP GET /products/42──► catalog-service
        │                                          │
        │         ◄────── 200 / producto ──────────┘
        │
   (si Catalog tarda 3s, el cliente de Ordering espera 3s)
   (si Catalog está caído, Ordering NO puede crear el pedido)
```

Esto se llama **acoplamiento temporal** y es el gran peligro de abusar de la comunicación
síncrona: una **cadena** de llamadas síncronas A→B→C→D tiene la disponibilidad *multiplicada*
(si cada uno está al 99%, cuatro en cadena dan ~96%) y la latencia *sumada*.

---

## 4. El coste oculto de lo asíncrono: complejidad y consistencia eventual

Lo asíncrono rompe el acoplamiento temporal —Ordering publica "pedido creado" y sigue, le da
igual si Catalog está vivo ahora mismo—, pero a cambio:

- **Consistencia eventual**: el efecto en el otro servicio ocurre *después*, no al instante.
  El sistema está temporalmente incoherente y hay que diseñar para ello.
- **Más infraestructura**: necesitas un broker (RabbitMQ), gestionar colas, *dead letters*,
  reintentos, idempotencia.
- **Razonamiento más difícil**: el flujo deja de leerse de arriba abajo. "¿Qué pasó con este
  pedido?" se responde mirando varios servicios y sus logs.

---

## 5. La regla práctica para ShopMicro

No es "síncrono malo, asíncrono bueno". Es **leer la naturaleza de la interacción**:

- **¿Necesito la respuesta AHORA para continuar y es una consulta puntual?** → síncrono.
  Ejemplo: el Gateway enruta una petición de lectura del catálogo al usuario final.
- **¿Estoy notificando que algo ocurrió y no me importa quién reaccione ni cuándo?** →
  asíncrono. Ejemplo: "se creó un pedido" → que Catalog ajuste stock, que se envíe un correo,
  que se actualicen métricas. Ninguno bloquea la creación del pedido.
- **¿Una operación de negocio abarca varios servicios con pasos que pueden fallar y
  compensarse?** → asíncrono con **Saga** (Módulo 6).

> **Conexión con lo que ya viste.** El flujo de eventos de dominio → Outbox → MessageRelay →
> RabbitMQ que trabajamos en 4.6–4.8 **es** la comunicación asíncrona de ShopMicro. Las
> lecciones 4.2–4.5 que vienen ahora cubren el otro lado: la comunicación **síncrona** bien
> hecha (HTTP resiliente, gRPC), para cuando sí la necesitas.

## Puntos clave

- Partir el monolito convierte llamadas a método en llamadas de red: **falibles, lentas, con
  el otro extremo potencialmente caído**.
- **Síncrono** = petición/respuesta, esperas bloqueado. Simple de razonar, pero introduce
  **acoplamiento temporal**: los fallos y latencias se encadenan.
- **Asíncrono** = mensajes/eventos, no esperas. Rompe el acoplamiento temporal a cambio de
  **consistencia eventual** y más infraestructura.
- La cadena de llamadas síncronas **multiplica la indisponibilidad y suma la latencia**: el
  enemigo a vigilar.
- La elección se hace **por la naturaleza de la interacción**: ¿necesito la respuesta ya
  (síncrono) o estoy notificando un hecho (asíncrono)?
- El flujo de eventos de dominio de 4.6–4.8 **es** la cara asíncrona de ShopMicro.
