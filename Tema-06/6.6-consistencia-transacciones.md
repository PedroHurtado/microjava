# 6.6 — El problema de la consistencia: transacciones distribuidas

> **Módulo 6 · Lección 6 de 9**
> **Duración estimada:** ~30 min
> **Tipo:** Teoría

---

## El problema, planteado de nuevo

Lo anticipamos en 6.1: si cada servicio tiene su propia base de datos, **una operación de negocio
que cruza varios servicios ya no cabe en una transacción ACID única**.

Caso de ShopMicro — "realizar un pedido":

```
   Cliente confirma el pedido
            │
            ▼
   ┌──────────────┐   ¿reservar stock?   ┌──────────────┐
   │ Ordering.API │ ───────────────────▶ │ Catalog.API  │
   │  (crea Order)│                       │ (baja stock) │
   └──────┬───────┘                       └──────────────┘
          │ ¿cobrar?
          ▼
   ┌──────────────┐
   │ Payments.API │
   │ (carga pago) │
   └──────────────┘
```

Tres servicios, **tres bases de datos**. No existe un `COMMIT` que las abarque. ¿Qué pasa si el
pedido se crea, el stock se reserva, pero **el pago falla**? Tendríamos un pedido y un stock
reservado para algo que nadie pagó. **Inconsistencia.**

---

## Por qué no usamos transacciones distribuidas "clásicas" (2PC)

Existe una técnica para coordinar varias bases de datos en una sola transacción: **Two-Phase
Commit (2PC)**. Un coordinador pregunta a todos "¿podéis confirmar?" (fase 1) y, si todos dicen sí,
les ordena confirmar (fase 2).

```
        Coordinador
          │  fase 1: ¿prepare?
   ┌──────┼──────┐
   ▼      ▼      ▼
  BD-A   BD-B   BD-C        (todas responden "listo")
   │      │      │
          │  fase 2: ¡commit!
   ▼      ▼      ▼
  BD-A   BD-B   BD-C
```

**En microservicios se evita casi siempre**, porque:

| Problema de 2PC | Consecuencia |
|---|---|
| **Bloqueante** | Cada BD queda bloqueada esperando al coordinador → mata el rendimiento y la escalabilidad |
| **Coordinador = punto único de fallo** | Si cae a mitad de la fase 2, las BD quedan en limbo |
| **Acopla los servicios** | Todos deben hablar el mismo protocolo transaccional y estar disponibles a la vez |
| Mal soporte | Muchos almacenes (Redis, colas, NoSQL) ni siquiera lo soportan |

2PC busca **consistencia fuerte** a toda costa, y en un sistema distribuido eso cuesta
disponibilidad y escalabilidad. En microservicios casi siempre preferimos lo contrario.

---

## El cambio de mentalidad: consistencia eventual

En lugar de "todo confirma a la vez o nada lo hace", aceptamos que:

> El sistema pasará por **estados intermedios temporalmente inconsistentes**, pero **convergerá** a
> un estado consistente en un tiempo acotado.

Esto es **consistencia eventual**. El pedido del ejemplo seguiría este camino:

```
   t0:  Order = Pendiente            (creado, aún sin confirmar)
   t1:  Stock reservado
   t2a: Pago OK    → Order = Confirmado          ✓ converge
   t2b: Pago falla → liberar stock → Order = Cancelado   ✓ converge igualmente
```

En `t1` el sistema está "a medias", pero **nunca se queda ahí**: o avanza a Confirmado o
retrocede a Cancelado liberando lo reservado. La clave es ese **camino de vuelta**: las
compensaciones.

---

## Dos herramientas para implementar esto

El resto del módulo da las dos piezas concretas:

- **6.7 — Patrón Saga.** Orquesta el *flujo* de la operación multi-servicio como una secuencia de
  pasos locales, cada uno con su **acción compensatoria** para deshacer si algo falla más adelante.
- **6.8 — Patrón Outbox.** Garantiza que los **eventos** que disparan cada paso de la Saga **no se
  pierdan** aunque el servicio se caiga justo después de guardar en su BD.

```
   Problema (6.6):  no hay transacción ACID entre servicios
        │
        ├──▶ Saga (6.7):   ¿cómo encadeno los pasos y deshago si falla?
        └──▶ Outbox (6.8): ¿cómo garantizo que el evento que dispara el paso no se pierda?
```

---

## Puntos clave

- Con **database per service** no existe transacción ACID que cruce servicios.
- **2PC** (transacción distribuida clásica) **se evita** en microservicios: es bloqueante, frágil y
  acopla los servicios.
- Se sustituye por **consistencia eventual**: el sistema pasa por estados intermedios pero
  **converge** a un estado consistente.
- El camino se implementa con **Saga** (flujo + compensaciones) y se protege con **Outbox**
  (entrega fiable de eventos).
