# 6.1 — Database per service: el principio y sus consecuencias

> **Módulo 6 · Lección 1 de 9**
> **Duración estimada:** ~25 min
> **Tipo:** Teoría

---

## El principio

En una arquitectura de microservicios, **cada servicio es el único dueño de sus datos**. Tiene su
propia base de datos (o esquema, o tabla), y **ningún otro servicio accede a ella directamente**.
Si el servicio de Pedidos necesita un dato del Catálogo, lo pide **a través de la API** del
Catálogo — nunca abriendo una conexión a su base de datos.

```
        ✗ PROHIBIDO                          ✓ CORRECTO

  ┌─────────────┐                      ┌─────────────┐
  │ Ordering    │                      │ Ordering    │
  │   .API      │                      │   .API      │
  └──────┬──────┘                      └──────┬──────┘
         │ SELECT * FROM                      │ GET /products/42
         │ catalog.products                   │ (HTTP / mensaje)
         ▼                                    ▼
  ┌─────────────┐                      ┌─────────────┐
  │  Catalog DB │                      │ Catalog.API │
  └─────────────┘                      └──────┬──────┘
                                              ▼
                                       ┌─────────────┐
                                       │  Catalog DB │
                                       └─────────────┘
```

La frontera de datos coincide con la frontera del servicio, que a su vez coincide con el
**bounded context** (lo vimos en el Módulo 1). Esto no es un detalle de implementación: es lo que
hace que un microservicio sea de verdad **autónomo**.

---

## Por qué es tan importante

| Si compartes la base de datos… | Consecuencia |
|---|---|
| Dos servicios leen/escriben la misma tabla | Quedan **acoplados** por el esquema: cambiar una columna rompe a ambos |
| El esquema es un "contrato implícito" | Nadie se atreve a tocarlo → la BD se fosiliza |
| Un servicio bloquea filas | El otro servicio sufre la contención |
| Despliegas un servicio | Tienes que coordinar la migración con el otro |

Con **database per service**, cada equipo puede **elegir su tecnología de almacenamiento**,
**migrar su esquema** y **desplegar** sin pedir permiso a nadie. Esa autonomía es el 80% del valor
de los microservicios.

En ShopMicro esto se traduce en:

| Servicio | Almacén "real" (curso) | En este módulo |
|---|---|---|
| **Catalog.API** | PostgreSQL (Spring Data JPA) | H2 **en memoria** (Spring Data JPA) |
| **Basket.API** | Redis | Mapa en memoria |
| **Ordering.API** | PostgreSQL (Spring Data JPA) | H2 **en memoria** (Spring Data JPA) |

---

## La consecuencia incómoda

Si cada servicio tiene su propia base de datos, entonces **una operación de negocio que cruza
varios servicios ya no cabe en una sola transacción ACID**.

Ejemplo: "crear un pedido" implica al servicio de Pedidos (crea el pedido), al de Catálogo
(reserva stock) y quizá al de Pagos. Con un monolito y una sola BD haríamos:

```sql
BEGIN TRANSACTION;
  INSERT INTO orders ...;
  UPDATE products SET stock = stock - 1 ...;
COMMIT;   -- todo o nada
```

Con tres bases de datos distintas **eso es imposible**: no hay un `COMMIT` que abarque tres
servicios. Esta es la tensión central de la gestión de datos distribuida, y es lo que motiva el
resto del módulo:

- **6.6** plantea el problema de la consistencia.
- **6.7 (Saga)** resuelve el *flujo* de una operación multi-servicio.
- **6.8 (Outbox)** garantiza que los eventos que coordinan esa Saga **no se pierdan**.

---

## Puntos clave

- **Cada microservicio es dueño exclusivo de sus datos.** El acceso siempre pasa por su API.
- Compartir base de datos entre servicios reintroduce el **acoplamiento** que veníamos a evitar.
- La autonomía de datos permite **elegir tecnología, migrar y desplegar** de forma independiente.
- El precio: **se pierde la transacción ACID global**. Lo recuperaremos como consistencia
  eventual con **Saga** y lo blindaremos con **Outbox**.
