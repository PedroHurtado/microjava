> **Duración:** ~90 minutos  
> **Contexto:** Implementar catalog-api con arquitectura Vertical Slice  
> **Hilo conductor:** ShopMicro - Primer microservicio funcional  
> **Patrón:** Nested Classes, Vertical Slice, FluentValidation, PipelinR/Dispatcher

---

## Arquitectura Vertical Slice (VSA)

La **Vertical Slice Architecture** organiza el código por **feature**, no por capas técnicas.

### Estructura tradicional (por capas)

```
controller/
  ProductController.java
service/
  ProductService.java
repository/
  ProductRepository.java
model/
  Product.java
  CreateProductRequest.java
  CreateProductResponse.java
```

**Problema:** Para agregar una feature, tocar 5 archivos en 5 paquetes diferentes.

### Estructura Vertical Slice (recomendada)

```
features/
  products/
    create/
      CreateProductCommand.java
      CreateProductHandler.java
      CreateProductEndpoint.java
      CreateProductValidator.java
    list/
      ListProductsQuery.java
      ListProductsHandler.java
      ListProductsEndpoint.java
    getbyid/
      GetProductByIdQuery.java
      GetProductByIdHandler.java
      GetProductByIdEndpoint.java
    domain/
      Product.java
      ProductRepository.java
```

**Ventaja:** Una feature = un paquete = cambios localizados = menos conflictos.

## Patrón: Nested Classes con Vertical Slice

Organizaremos cada feature como un archivo único con clases anidadas, igual que en el
ejemplo original de Spring Boot que da nombre a este patrón.

### Estructura de un archivo de feature (ejemplo: CreateProduct)

```java
package com.shopmicro.catalog.features.products.create;

import an.awesome.pipelinr.Command;
import an.awesome.pipelinr.Pipeline;
import com.shopmicro.catalog.core.fluentvalidation.AbstractValidator;
import com.shopmicro.catalog.features.products.domain.Product;
import com.shopmicro.catalog.features.products.domain.ProductRepository;
import org.springframework.http.ResponseEntity;
import org.springframework.stereotype.Component;
import org.springframework.web.bind.annotation.*;

import java.math.BigDecimal;
import java.net.URI;
import java.util.UUID;

public final class CreateProduct {

    private CreateProduct() { }

    // Endpoint (Controller)
    @RestController
    @RequestMapping("/api/v1/products")
    public static class Endpoint {

        private final Pipeline pipeline;

        public Endpoint(Pipeline pipeline) {
            this.pipeline = pipeline;
        }

        @PostMapping
        public ResponseEntity<Response> create(@RequestBody Command command) {
            Response response = pipeline.send(command);
            URI location = URI.create("/api/v1/products/" + response.id());
            return ResponseEntity.created(location).body(response);
        }
    }

    // Handler (Service)
    @Component
    public static class Handler implements Command.Handler<Command, Response> {

        private final ProductRepository repository;

        public Handler(ProductRepository repository) {
            this.repository = repository;
        }

        @Override
        public Response handle(Command command) {
            Product product = Product.create(
                UUID.randomUUID(),
                command.name(),
                command.description(),
                command.price(),
                command.stock()
            );

            repository.add(product);
            return new Response(product.getId(), product.getPrice());
        }
    }

    // Validador
    @Component
    public static class Validator extends AbstractValidator<Command> {

        public Validator() {
            ruleFor(Command::name)
                .notEmpty().withMessage("El nombre es obligatorio")
                .maximumLength(100).withMessage("El nombre no puede exceder 100 caracteres");

            ruleFor(Command::description)
                .notEmpty().withMessage("La descripción es obligatoria");

            ruleFor(Command::price)
                .greaterThan(BigDecimal.ZERO).withMessage("El precio debe ser mayor a 0");

            ruleFor(Command::stock)
                .greaterThanOrEqualTo(0).withMessage("El stock no puede ser negativo");
        }
    }

    // Command (Request)
    public record Command(
        String name,
        String description,
        BigDecimal price,
        int stock
    ) implements an.awesome.pipelinr.Command<Response> { }

    // Response
    public record Response(UUID id, BigDecimal price) { }
}
```

> **Nota:** en Java las clases anidadas estáticas dentro de una clase contenedora
> (`CreateProduct`) cumplen el mismo rol que las *nested classes* de C#: agrupan
> Endpoint, Handler, Validator, Command y Response en un único archivo cohesivo.

## Dominio (Domain)

### Product.java

```java
package com.shopmicro.catalog.features.products.domain;

import jakarta.persistence.*;
import java.math.BigDecimal;
import java.time.Instant;
import java.util.UUID;

@Entity
@Table(name = "products")
public class Product {

    @Id
    private UUID id;

    @Column(nullable = false, length = 100)
    private String name;

    @Column(nullable = false)
    private String description;

    @Column(nullable = false, precision = 18, scale = 2)
    private BigDecimal price;

    @Column(nullable = false)
    private int stock;

    @Column(nullable = false)
    private Instant createdAt;

    @Column
    private Instant updatedAt;

    protected Product() { } // JPA / Hibernate

    public static Product create(UUID id, String name, String description,
                                 BigDecimal price, int stock) {
        Product product = new Product();
        product.id = id;
        product.name = name;
        product.description = description;
        product.price = price;
        product.stock = stock;
        product.createdAt = Instant.now();
        return product;
    }

    public void update(String name, String description, BigDecimal price, int stock) {
        this.name = name;
        this.description = description;
        this.price = price;
        this.stock = stock;
        this.updatedAt = Instant.now();
    }

    // Getters (sin setters públicos: el estado se modifica con métodos de dominio)
    public UUID getId() { return id; }
    public String getName() { return name; }
    public String getDescription() { return description; }
    public BigDecimal getPrice() { return price; }
    public int getStock() { return stock; }
    public Instant getCreatedAt() { return createdAt; }
    public Instant getUpdatedAt() { return updatedAt; }
}
```

> Equivalencias respecto a la versión .NET:
> - `Guid` → `java.util.UUID`
> - `decimal` → `java.math.BigDecimal`
> - `DateTime` (UTC) → `java.time.Instant`
> - propiedades con `private set` → campos privados + getters + métodos de dominio

## Repositorio (Interface Segregation Principle)

Según **SOLID/ISP**, cada repositorio debe tener responsabilidades específicas y bien definidas.

### ProductRepository.java (abstracción de dominio)

```java
package com.shopmicro.catalog.features.products.domain;

import java.util.List;
import java.util.Optional;
import java.util.UUID;

public interface ProductRepository {
    void add(Product product);
    Optional<Product> getById(UUID id);
    List<Product> getAll(int skip, int take);
    void update(Product product);
    void delete(UUID id);
    boolean exists(UUID id);
}
```

**Nota:** Cada operación tiene su propósito. Alternativa con sub-interfaces (ISP extremo):

```java
public interface ProductReadRepository {
    Optional<Product> getById(UUID id);
    List<Product> getAll(int skip, int take);
}

public interface ProductWriteRepository {
    void add(Product product);
    void update(Product product);
    void delete(UUID id);
}

public interface ProductRepository extends ProductReadRepository, ProductWriteRepository { }
```

## Implementación del Repositorio

Definimos primero un `JpaRepository` de Spring Data (Spring genera la implementación)
y luego un adaptador que expone nuestra abstracción de dominio (`ProductRepository`).

### SpringDataProductRepository.java (Spring Data JPA)

```java
package com.shopmicro.catalog.features.products.infrastructure;

import com.shopmicro.catalog.features.products.domain.Product;
import org.springframework.data.jpa.repository.JpaRepository;

import java.util.UUID;

public interface SpringDataProductRepository extends JpaRepository<Product, UUID> {
    boolean existsByName(String name);
}
```

### JpaProductRepository.java (adaptador ISP)

```java
package com.shopmicro.catalog.features.products.infrastructure;

import com.shopmicro.catalog.features.products.domain.Product;
import com.shopmicro.catalog.features.products.domain.ProductRepository;
import org.springframework.data.domain.PageRequest;
import org.springframework.stereotype.Repository;

import java.util.List;
import java.util.Optional;
import java.util.UUID;

@Repository
public class JpaProductRepository implements ProductRepository {

    private final SpringDataProductRepository jpa;

    public JpaProductRepository(SpringDataProductRepository jpa) {
        this.jpa = jpa;
    }

    @Override
    public void add(Product product) {
        jpa.save(product);
    }

    @Override
    public Optional<Product> getById(UUID id) {
        return jpa.findById(id);
    }

    @Override
    public List<Product> getAll(int skip, int take) {
        int page = take > 0 ? skip / take : 0;
        return jpa.findAll(PageRequest.of(page, take)).getContent();
    }

    @Override
    public void update(Product product) {
        jpa.save(product);
    }

    @Override
    public void delete(UUID id) {
        jpa.findById(id).ifPresent(jpa::delete);
    }

    @Override
    public boolean exists(UUID id) {
        return jpa.existsById(id);
    }
}
```

> En .NET el `DbContext` de EF Core gestiona el seguimiento de entidades y
> `SaveChangesAsync()`. En Spring Data JPA, `save()` realiza el *insert/update* y el
> `flush` ocurre al cierre de la transacción (gestionada con `@Transactional`).

## Configuración de PipelinR

En Spring no hay un `Program.cs` imperativo: registramos el `Pipeline` de PipelinR
como bean en una clase `@Configuration`. PipelinR descubre automáticamente todos los
`Command.Handler` y `Command.Middleware` que sean beans del contenedor.

### PipelinrConfiguration.java

```java
package com.shopmicro.catalog.config;

import an.awesome.pipelinr.Command;
import an.awesome.pipelinr.Pipeline;
import an.awesome.pipelinr.Pipelinr;
import org.springframework.beans.factory.ObjectProvider;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class PipelinrConfiguration {

    @Bean
    public Pipeline pipeline(ObjectProvider<Command.Handler> handlers,
                             ObjectProvider<Command.Middleware> middlewares) {
        return new Pipelinr()
            .with(handlers::stream)
            .with(middlewares::stream);
    }
}
```

> Equivalencia con MediatR:
> - `builder.Services.AddMediatR(...)` → registro automático de handlers vía Spring +
>   bean `Pipelinr`.
> - `IRequest<TResponse>` → `an.awesome.pipelinr.Command<R>`.
> - `IRequestHandler<TRequest,TResponse>` → `Command.Handler<C, R>`.
> - `IPipelineBehavior<,>` → `Command.Middleware`.
> - `_mediator.Send(command)` → `pipeline.send(command)`.

### Clase principal de arranque

```java

import org.springframework.boot.autoconfigure.SpringBootApplication;
```

## Pipeline de validación (Middleware)

Para que FluentValidation se ejecute automáticamente antes de cada Handler, creamos un
`Command.Middleware` (equivalente al `IPipelineBehavior` de MediatR).

### ValidationMiddleware.java

```java
package com.shopmicro.catalog.core.behaviors;

import an.awesome.pipelinr.Command;
import com.shopmicro.catalog.core.fluentvalidation.AbstractValidator;
import com.shopmicro.catalog.core.fluentvalidation.ValidationException;
import com.shopmicro.catalog.core.fluentvalidation.ValidationFailure;
import org.springframework.beans.factory.ObjectProvider;
import org.springframework.core.ResolvableType;
import org.springframework.stereotype.Component;

import java.util.List;

@Component
public class ValidationMiddleware implements Command.Middleware {

    private final ObjectProvider<AbstractValidator<?>> validators;

    public ValidationMiddleware(ObjectProvider<AbstractValidator<?>> validators) {
        this.validators = validators;
    }

    @Override
    @SuppressWarnings("unchecked")
    public <R, C extends Command<R>> R invoke(C command, Next<R> next) {
        List<ValidationFailure> failures = validators.stream()
            .filter(v -> supports(v, command))
            .map(v -> (AbstractValidator<C>) v)
            .flatMap(v -> v.validate(command).stream())
            .toList();

        if (!failures.isEmpty()) {
            throw new ValidationException(failures);
        }

        return next.invoke();
    }

    private boolean supports(AbstractValidator<?> validator, Object command) {
        ResolvableType type = ResolvableType.forClass(AbstractValidator.class, validator.getClass());
        Class<?> validated = type.getGeneric(0).resolve();
        return validated != null && validated.isInstance(command);
    }
}
```

> PipelinR ejecuta los `Command.Middleware` en cadena antes de llegar al handler,
> tal como el *pipeline* de MediatR. Si la validación falla, lanzamos
> `ValidationException`, que será transformada en `ProblemDetail` (422) por el manejador
> global de errores (Módulo 3.5).

## Alternativa: Implementar tu propio Dispatcher

Si prefieres no usar **PipelinR**, puedes implementar un dispatcher simple apoyándote
en el `ApplicationContext` de Spring.

### CommandDispatcher.java

```java
package com.shopmicro.catalog.core;

public interface Command<R> { }

public interface CommandHandler<C extends Command<R>, R> {
    R handle(C command);
}

public interface CommandDispatcher {
    <C extends Command<R>, R> R dispatch(C command);
}
```

### SpringCommandDispatcher.java

```java
package com.shopmicro.catalog.core;

import org.springframework.context.ApplicationContext;
import org.springframework.core.ResolvableType;
import org.springframework.stereotype.Component;

@Component
public class SpringCommandDispatcher implements CommandDispatcher {

    private final ApplicationContext context;

    public SpringCommandDispatcher(ApplicationContext context) {
        this.context = context;
    }

    @Override
    @SuppressWarnings("unchecked")
    public <C extends Command<R>, R> R dispatch(C command) {
        ResolvableType handlerType = ResolvableType.forClassWithGenerics(
            CommandHandler.class, command.getClass(), Object.class);

        String[] names = context.getBeanNamesForType(handlerType);
        if (names.length == 0) {
            throw new IllegalStateException(
                "No handler registered for " + command.getClass().getSimpleName());
        }

        CommandHandler<C, R> handler =
            (CommandHandler<C, R>) context.getBean(names[0]);
        return handler.handle(command);
    }
}
```

**Usar en Endpoint:**

```java
@PostMapping
public ResponseEntity<Response> create(@RequestBody Command command) {
    Response response = dispatcher.dispatch(command);
    URI location = URI.create("/api/v1/products/" + response.id());
    return ResponseEntity.created(location).body(response);
}
```

> Spring ya resuelve handlers por tipo genérico gracias a `ResolvableType`, así que el
> dispatcher manual es muy ligero. Aun así, **PipelinR** aporta el *pipeline* de
> middlewares (validación, logging, transacciones) listo para usar.

## Puntos clave

- **Vertical Slice:** Una feature = un paquete = máxima cohesión
- **Nested Classes:** Endpoint, Handler, Validator, Command, Response en la misma clase contenedora
- **PipelinR:** Desacoplamiento entre Endpoint y Handler, pipeline de middlewares
- **FluentValidation:** Validación declarativa y reutilizable
- **ISP:** Repositorios especializados (`ProductRepository` con métodos específicos)
- **Dispatcher personalizado:** Alternativa lightweight si no necesitas toda la potencia de PipelinR
