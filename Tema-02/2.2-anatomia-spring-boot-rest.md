## ¿Por qué `@RestController` para microservicios?

Spring ofrece dos estilos para construir APIs:

- **Controladores anotados (`@RestController`):** el estilo idiomático y más extendido. Defines los endpoints como métodos de una clase, marcados con `@GetMapping`, `@PostMapping`, etc. Conciso y directo.
- **Enrutado funcional (`RouterFunction` / `HandlerFunction`):** defines las rutas de forma programática con un DSL funcional, sin clases controladoras anotadas. Más flexible, algo menos común.

Para microservicios, **los controladores anotados son habitualmente la mejor opción**: un microservicio expone pocos endpoints muy cohesionados, y las anotaciones reducen el *boilerplate* y se leen muy bien. Usaremos `@RestController` en todo el curso. (El estilo funcional sigue siendo válido, sobre todo en aplicaciones reactivas con WebFlux.)

## El punto de entrada: la clase `@SpringBootApplication`

Una aplicación Spring Boot arranca desde un método `main` clásico, pero toda la magia la dispara `SpringApplication.run(...)`:

```java
@SpringBootApplication
public class CatalogApplication {
    public static void main(String[] args) {
        SpringApplication.run(CatalogApplication.class, args);
    }
}
```

Hay dos fases muy claras dentro de ese `run(...)`:

```
// ───────── FASE 1: ARRANQUE DEL CONTEXTO (registro) ─────────
// Spring escanea componentes, aplica la auto-configuración y
// construye el ApplicationContext: registra todos los beans
// (servicios, repositorios, clientes HTTP, EF/JPA, validadores...).

// ───────── FASE 2: SERVIDOR Y ENDPOINTS (atención a peticiones) ─────────
// Una vez listo el contexto, Spring arranca el Tomcat embebido,
// publica los endpoints de tus @RestController y se queda escuchando.
```

> **La división mental clave:** primero Spring **registra** capacidades (qué sabe hacer la app: beans, configuración, auto-config). *Después*, arranca el servidor y **atiende** las peticiones a través de controladores y filtros. La anotación `@SpringBootApplication` activa el escaneo de componentes y la auto-configuración que hacen posible la fase 1.

## Definir endpoints

Un `@RestController` agrupa endpoints cohesionados. Las anotaciones `@…Mapping` cubren los verbos HTTP:

```java
@RestController
@RequestMapping("/products")
public class ProductController {

    @GetMapping
    List<Product> getAll() { ... }                               // listar

    @GetMapping("/{id}")
    Product getById(@PathVariable int id) { ... }                // detalle

    @PostMapping
    Product create(@RequestBody Product p) { ... }               // crear

    @PutMapping("/{id}")
    Product update(@PathVariable int id, @RequestBody Product p) { ... } // actualizar

    @DeleteMapping("/{id}")
    void delete(@PathVariable int id) { ... }                    // borrar
}
```

### Binding de parámetros

Spring resuelve **de dónde sacar cada parámetro** mediante anotaciones explícitas:

```java
@GetMapping("/{id}")
Product get(
    @PathVariable int id,                    // ← de la ruta (coincide con {id})
    @RequestParam(required = false) String q // ← de la query string (?q=...)
) {
    return service.buscar(id, q);
}
```

Las reglas de *binding* más habituales:
- `@PathVariable` → viene de un `{segmento}` de la ruta.
- `@RequestParam` → de la *query string*.
- `@RequestBody` → del cuerpo de la petición (JSON), deserializado a un objeto.
- Las **dependencias** (servicios, repositorios) **no** se pasan como parámetros del método: se inyectan en el controlador por **constructor** (lo vemos en 2.3).

### Devolver respuestas con `ResponseEntity`

Para controlar el código de estado HTTP, usamos `ResponseEntity`:

```java
@GetMapping("/{id}")
ResponseEntity<Product> getById(@PathVariable int id) {
    var producto = service.buscar(id);
    return producto != null
        ? ResponseEntity.ok(producto)              // 200
        : ResponseEntity.notFound().build();        // 404
}

@PostMapping
ResponseEntity<Product> create(@RequestBody Product p) {
```

## Organizar endpoints: agrupar por recurso

Meter 30 endpoints en una sola clase no escala. Dos técnicas:

### Una ruta base por controlador

```java
@RestController
@RequestMapping("/products")   // todos los endpoints cuelgan de /products
public class ProductController {
    @GetMapping            List<Product> getAll() { ... }
    @GetMapping("/{id}")   Product getById(@PathVariable int id) { ... }
    @PostMapping           Product create(@RequestBody Product p) { ... }
}
```

### Un controlador por agregado / recurso (el patrón que usaremos)

En lugar de un único controlador gigante, creamos **un `@RestController` por recurso** (`ProductController`, `CategoryController`...). Cada uno es pequeño, cohesionado y fácil de testear. El escaneo de componentes de Spring los descubre automáticamente, sin necesidad de registrarlos a mano.

## Configuración de arranque local: `application.yml`

En Spring Boot, el puerto y el perfil activo para desarrollo local se definen en `application.yml` (no hay un archivo separado tipo `launchSettings.json`):

```yaml
server:
  port: 8080            # puerto del servidor embebido (8080 por defecto)

spring:
  application:
    name: catalog-api
  profiles:
    active: dev         # perfil local activo (carga application-dev.yml)
```

Veremos la configuración en profundidad en el punto 2.4. La idea clave: el mismo binario se comporta distinto según el **perfil** y las **variables de entorno**, sin recompilar.

## Puntos clave

- **`@RestController` con anotaciones** es el estilo idóneo para microservicios: conciso y legible.
- El arranque tiene dos fases: **registrar el contexto/beans** (auto-configuración) y luego **arrancar el servidor + atender peticiones**.
- Los parámetros de endpoint se *bindean* con `@PathVariable`, `@RequestParam` y `@RequestBody`; las dependencias se inyectan por constructor.
- Usa `ResponseEntity` para controlar el código de estado HTTP y las cabeceras.
- Organiza los endpoints con una **ruta base** (`@RequestMapping`) y **un controlador por recurso** para no inflar las clases.
