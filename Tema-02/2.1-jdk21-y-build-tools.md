# 2.1 El JDK 21 y las herramientas de build (Maven, Spring Initializr)

> **Módulo 2 — Fundamentos de Spring Boot para Microservicios**
> Duración estimada: ~35 min

---

## JDK vs. JRE

Dos cosas distintas que conviene no confundir:

- **JRE** (Java Runtime Environment): lo necesario para **ejecutar** una aplicación Java ya compilada (la JVM más las librerías base). Es lo que basta en un servidor de producción (o lo que va dentro de la imagen de contenedor).
- **JDK** (Java Development Kit): incluye el runtime **más** todas las herramientas para **desarrollar**: el compilador `javac`, el depurador, utilidades como `jar`, `jshell`, etc. Es lo que instalas en tu máquina de desarrollo.

Para el curso necesitas el **JDK 21 (LTS)**. Compruébalo:

```bash
java -version
# Debería mostrar algo como "openjdk version 21.0.x"

javac -version   # versión del compilador (solo viene en el JDK)
```

> **Nota sobre versiones:** usamos **Java 21 (LTS)** por ser el *baseline* mínimo de Spring Boot 3.x y el más extendido y documentado en el ecosistema de microservicios. Java 25 (LTS) ya está disponible; en el Módulo 7 veremos la ruta de migración. El código es prácticamente idéntico entre versiones. Para gestionar varias versiones de JDK en tu máquina, herramientas como **SDKMAN!** son muy cómodas.

## Maven: tu herramienta de build principal

Todo el ciclo de desarrollo se puede hacer desde la línea de comandos, sin depender del IDE. Esto es **fundamental** en microservicios: los pipelines de CI/CD y los Dockerfiles usan exactamente estos comandos. Usaremos **Maven** en el curso (Gradle es una alternativa igualmente válida).

### Crear proyectos: Spring Initializr

A diferencia de un solo comando, en Spring el punto de partida es **Spring Initializr** ([start.spring.io](https://start.spring.io)): eliges versión de Java, gestor de dependencias y *starters*, y te genera el esqueleto. Desde la terminal puedes usar la CLI de Spring Boot:

```bash
# Crear una Web API (la que usaremos para los servicios)
spring init --build=maven --java-version=21 --dependencies=web -n catalog-api catalog-api

# O bien, vía curl contra Spring Initializr:
curl https://start.spring.io/starter.zip \
  -d dependencies=web -d javaVersion=21 -d type=maven-project \
  -d artifactId=catalog-api -o catalog-api.zip
```

*Starters* (dependencias) que usaremos en el curso:

| Starter / artefacto | Para qué |
|---|---|
| `spring-boot-starter-web` | Nuestros microservicios REST |
| POM padre multi-módulo | Agrupar todos los servicios |
| Módulo Maven plano | Código compartido (contratos) |
| `.gitignore` (lo genera Initializr) | Ignorar `target/`, etc. |

### `mvn compile` / `mvn package` — compilar y empaquetar

```bash
mvn compile      # compila el código fuente
mvn package      # compila, pasa los tests y genera el JAR ejecutable en target/
```

### `mvn spring-boot:run` — ejecutar

```bash
mvn spring-boot:run
# Compila (si hace falta) y ejecuta. Para desarrollo local.
```

Para **recarga en caliente** al guardar cambios, añade la dependencia `spring-boot-devtools`: reinicia automáticamente la aplicación al detectar cambios en el classpath. Es tu mejor amigo en desarrollo.

```xml
<dependency>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-devtools</artifactId>
  <optional>true</optional>
</dependency>
```

### Otros comandos que usaremos

```bash
mvn dependency:resolve   # descargar/resolver dependencias (Maven lo hace solo al compilar)
mvn test                 # ejecutar tests
mvn clean                # borrar artefactos previos (target/)
mvn -P prod package      # empaquetar con el perfil de producción
java -jar target/catalog-api.jar   # ejecutar el JAR ya empaquetado (como en producción)
```

## Anatomía de un proyecto recién creado

Tras generar `catalog-api` con Spring Initializr, obtienes:

```
catalog-api/
├── pom.xml                                  ← definición del proyecto y dependencias
├── src/
│   ├── main/
│   │   ├── java/com/shopmicro/catalog/
│   │   │   └── CatalogApplication.java      ← punto de entrada (método main)
│   │   └── resources/
│   │       ├── application.yml              ← configuración
│   │       └── application-dev.yml
│   └── test/
│       └── java/com/shopmicro/catalog/      ← tests
└── target/                                  ← artefactos compilados (ignorado en git)
```

### El `pom.xml`

Es un XML que define cómo se construye el proyecto y de qué depende:

```xml
<project xmlns="http://maven.apache.org/POM/4.0.0">
  <modelVersion>4.0.0</modelVersion>

  <!-- Hereda del starter parent: gestiona versiones de dependencias y plugins -->
  <parent>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-parent</artifactId>
    <version>3.4.0</version>
  </parent>

  <groupId>com.shopmicro</groupId>
  <artifactId>catalog-api</artifactId>
  <version>0.0.1-SNAPSHOT</version>

  <properties>
    <java.version>21</java.version>
  </properties>

  <!-- Las dependencias aparecerán aquí: -->
  <dependencies>
    <dependency>
      <groupId>org.springframework.boot</groupId>
      <artifactId>spring-boot-starter-web</artifactId>
    </dependency>
  </dependencies>
</project>
```

Dos cosas que conviene conocer:
- **Los *starters*** (`spring-boot-starter-*`): agrupan un conjunto coherente de dependencias para una capacidad concreta (web, datos, seguridad...). El **`spring-boot-starter-parent`** fija además las versiones compatibles entre sí (BOM), por eso no escribimos `<version>` en cada dependencia.
- **El `spring-boot-maven-plugin`**: empaqueta un **JAR ejecutable** ("fat jar") con servidor embebido (Tomcat) incluido. Por eso un microservicio Spring Boot arranca con `java -jar` sin instalar nada más — ideal para Docker.

## El ciclo de trabajo típico

```bash
# 1. Generar el proyecto (Spring Initializr o spring init)
spring init --build=maven --java-version=21 --dependencies=web -n catalog-api catalog-api

# 2. Entrar y ejecutar (con devtools, recarga automática al guardar)
cd catalog-api
mvn spring-boot:run

# 3. (en otra terminal) probar el endpoint
curl http://localhost:8080/
```

---

## Puntos clave

- El **JDK** es para desarrollar; el **JRE/runtime** es para ejecutar (importante para Docker).
- **Maven** cubre todo el ciclo: `compile`, `package`, `spring-boot:run`, `test`, `clean`.
- **Spring Initializr** (start.spring.io) genera el esqueleto del proyecto y sus *starters*.
- `spring-boot-devtools` recarga la app al guardar — esencial en desarrollo.
- El `pom.xml` define la versión de Java y las dependencias; los *starters* y el *parent* gestionan las versiones por ti.
- El `spring-boot-maven-plugin` produce un **JAR ejecutable** con Tomcat embebido — clave para los Dockerfiles del Módulo 7.
