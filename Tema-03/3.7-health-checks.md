---

## ¿Qué es un Health Check?

Un **Health Check** es un endpoint que verifica si un servicio está **operativo** y listo para recibir tráfico.

### Casos de uso

✅ **Kubernetes:** Readiness/Liveness probes reinician pods fallidos  
✅ **Load Balancers:** Removimiento automático de instancias caídas  
✅ **Monitoreo:** Sistemas de alertas detectan problemas  
✅ **Orquestación:** Decisiones de escalado basadas en salud  
✅ **API Gateway:** Enrutamiento a instancias saludables  

---

## Estructura de una respuesta de health check

En Spring Boot, **Actuator** expone los health checks. La forma de la respuesta es:

### Respuesta simple (200 OK)

```json
{
  "status": "UP"
}
```

### Respuesta detallada (`management.endpoint.health.show-details: always`)

```json
{
  "status": "UP",
  "components": {
    "db": {
      "status": "UP",
      "details": { "database": "PostgreSQL", "validationQuery": "isValid()" }
    },
    "redis": {
      "status": "OUT_OF_SERVICE",
      "details": { "error": "Redis timeout (non-critical)" }
    },
    "externalApi": {
      "status": "DOWN",
      "details": { "error": "Payment provider unreachable" }
    }
  }
}
```

---

## Códigos de estado

| Estado (Actuator) | Significado | Código HTTP | Acción |
|-------------------|-------------|-------------|--------|
| **UP** | Servicio operativo | 200 OK | Continuar tráfico |
| **OUT_OF_SERVICE** | Funcional pero degradado | 200 OK (configurable) | Continuar, pero alertar |
| **DOWN** | No operativo | 503 Service Unavailable | Remover del servicio |

> Equivalencia con .NET: `Healthy` → `UP`, `Degraded` → `OUT_OF_SERVICE`/`UNKNOWN`,
> `Unhealthy` → `DOWN`. El mapeo de estado→código HTTP se configura en `application.yml`.

---

## Setup en Spring Boot

Basta el starter de Actuator (las dependencias de BD/Redis ya aportan sus indicadores):

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-actuator</artifactId>
</dependency>
```

Actuator detecta automáticamente indicadores para PostgreSQL (`DataSourceHealthIndicator`),
Redis, disco (`DiskSpaceHealthIndicator`), etc., cuando esas dependencias están presentes.

---

## Configuración básica

### application.yml

```yaml
management:
  endpoints:
    web:
      exposure:
        include: health, info, metrics
  endpoint:
    health:
      show-details: always
      probes:
        enabled: true          # habilita /health/liveness y /health/readiness
  health:
    livenessstate:
      enabled: true
    readinessstate:
      enabled: true
```

Esto expone:
- `GET /actuator/health` → estado global
- `GET /actuator/health/liveness` → liveness probe
- `GET /actuator/health/readiness` → readiness probe

> En .NET registrabas los endpoints con `app.MapHealthChecks(...)` en `Program.cs`. En
> Spring, los endpoints de health los publica Actuator; solo los configuras vía
> propiedades.

## Health Checks predefinidos

### Base de datos (PostgreSQL)

Automático: al tener un `DataSource` configurado, Actuator añade el componente `db` que
ejecuta `Connection.isValid()`. No requiere código.

### Cache (Redis)

Automático al añadir `spring-boot-starter-data-redis`: aparece el componente `redis`.

### HTTP (Servicios externos)

No hay indicador automático; se implementa con un `HealthIndicator` personalizado
(ver siguiente sección) usando `RestClient`/`WebClient`.

## Health Check personalizado

En Spring implementas la interfaz `HealthIndicator` (equivalente a `IHealthCheck`). El
nombre del componente se deriva del nombre del bean (`customHealthIndicator` → `custom`).

### CustomHealthIndicator.java

```java
package com.shopmicro.catalog.infrastructure.health;

import com.shopmicro.catalog.features.products.domain.ProductRepository;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import org.springframework.boot.actuate.health.Health;
import org.springframework.boot.actuate.health.HealthIndicator;
import org.springframework.stereotype.Component;

import java.time.Instant;

@Component("productRepository")
public class CustomHealthIndicator implements HealthIndicator {

    private static final Logger log = LoggerFactory.getLogger(CustomHealthIndicator.class);

    private final ProductRepository repository;

    public CustomHealthIndicator(ProductRepository repository) {
        this.repository = repository;
    }

    @Override
    public Health health() {
        try {
            log.info("Executing custom health check");
            long count = repository.count();

            return Health.up()
                .withDetail("productCount", count)
                .withDetail("timestamp", Instant.now())
                .build();
        } catch (Exception ex) {
            log.error("Custom health check failed", ex);
            return Health.down(ex).build();
        }
    }
}
```

> Devolver `Health.up()` / `Health.down()` / `Health.outOfService()` corresponde a
> `HealthCheckResult.Healthy/Unhealthy/Degraded`. Los `withDetail(...)` son el
> equivalente al `Dictionary<string, object> data` de .NET.

## Health Check con lógica condicional (degradado)

```java
package com.shopmicro.catalog.infrastructure.health;

import org.springframework.boot.actuate.health.Health;
import org.springframework.boot.actuate.health.HealthIndicator;
import org.springframework.stereotype.Component;

import java.io.File;

@Component("diskSpace")
public class DiskSpaceHealthIndicator implements HealthIndicator {

    @Override
    public Health health() {
        try {
            File root = new File("/");
            long available = root.getUsableSpace();
            long total = root.getTotalSpace();
            double percentFree = (available / (double) total) * 100;

            Health.Builder builder = (percentFree < 5) ? Health.down()
                : (percentFree < 10) ? Health.status("OUT_OF_SERVICE")
                : Health.up();

            return builder
                .withDetail("diskFreePercentage", percentFree)
                .withDetail("diskFreeGb", available / (1024.0 * 1024.0 * 1024.0))
                .build();
        } catch (Exception ex) {
            return Health.down(ex).build();
        }
    }
}
```

> Spring Boot ya trae un `DiskSpaceHealthIndicator` integrado; este ejemplo es ilustrativo
> de cómo devolver estados `UP` / `OUT_OF_SERVICE` / `DOWN` según una condición.

## Liveness vs. Readiness

En Kubernetes hay dos tipos de probes. Spring Boot los soporta de forma nativa con
**Application Availability** (`livenessState` y `readinessState`).

### Liveness Probe

Verifica si el contenedor está **vivo** (no congelado ni en deadlock). Debe ser muy
rápido y NO depender de recursos externos.

```
GET /actuator/health/liveness
```

**Configuración en Kubernetes:**

```yaml
livenessProbe:
  httpGet:
    path: /actuator/health/liveness
    port: 8080
  initialDelaySeconds: 10
  periodSeconds: 10
  timeoutSeconds: 2
  failureThreshold: 3
```

### Readiness Probe

Verifica si el servicio está **listo** para recibir tráfico (BD lista, dependencias OK).

```
GET /actuator/health/readiness
```

**Configuración en Kubernetes:**

```yaml
readinessProbe:
  httpGet:
    path: /actuator/health/readiness
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 5
  timeoutSeconds: 2
  failureThreshold: 3
```

### Asociar un indicador a la readiness

Por defecto, los `HealthIndicator` personalizados forman parte del grupo global, no de
los probes. Puedes incluirlos en el grupo de readiness con `application.yml`:

```yaml
management:
  endpoint:
    health:
      group:
        readiness:
          include: readinessState, db, productRepository
        liveness:
          include: livenessState
```

## Implementación completa en catalog-api

### application.yml

```yaml
management:
  endpoints:
    web:
      exposure:
        include: health, info, metrics, prometheus
  endpoint:
    health:
      show-details: always
      probes:
        enabled: true
      group:
        liveness:
          include: livenessState
        readiness:
          include: readinessState, db, productRepository
  health:
    livenessstate:
      enabled: true
    readinessstate:
      enabled: true

spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/catalog
    username: catalog
    password: catalog
```

> El componente `db` (PostgreSQL) lo añade Actuator automáticamente al detectar el
> `DataSource`; `productRepository` es nuestro `CustomHealthIndicator`.

## Monitoreo de health checks

### Con Prometheus

Añade Micrometer + el registro de Prometheus:

```xml
<dependency>
    <groupId>io.micrometer</groupId>
    <artifactId>micrometer-registry-prometheus</artifactId>
    <scope>runtime</scope>
</dependency>
```

Y expón el endpoint (ya incluido arriba en `exposure.include`):

```
GET /actuator/prometheus
```

> Prometheus puede *scrapear* `/actuator/prometheus`. Para alertar específicamente sobre
> salud, también se usa el exporter `up`/down por componente o la métrica de
> `health` derivada.

## Testing de health checks

```java
import com.shopmicro.catalog.features.products.domain.ProductRepository;
import com.shopmicro.catalog.infrastructure.health.CustomHealthIndicator;
import org.junit.jupiter.api.Test;
import org.springframework.boot.actuate.health.Status;

import static org.assertj.core.api.Assertions.assertThat;
import static org.mockito.Mockito.mock;
import static org.mockito.Mockito.when;

class HealthCheckTest {

    @Test
    void productRepository_shouldBeUp_whenRepositoryIsAccessible() {
        var repository = mock(ProductRepository.class);
        when(repository.count()).thenReturn(42L);
        var indicator = new CustomHealthIndicator(repository);

        var health = indicator.health();

        assertThat(health.getStatus()).isEqualTo(Status.UP);
        assertThat(health.getDetails()).containsEntry("productCount", 42L);
    }

    @Test
    void productRepository_shouldBeDown_whenRepositoryThrows() {
        var repository = mock(ProductRepository.class);
        when(repository.count()).thenThrow(new RuntimeException("DB down"));
        var indicator = new CustomHealthIndicator(repository);

        var health = indicator.health();

        assertThat(health.getStatus()).isEqualTo(Status.DOWN);
    }
}
```

> Para añadir el método `count()` a tu repositorio de dominio, declara
> `long count();` en `ProductRepository` y delega en `jpa.count()` en el adaptador JPA.

## Puntos clave

- **Liveness:** ¿El servicio está vivo? (rápido, simple — `/actuator/health/liveness`)
- **Readiness:** ¿El servicio está listo? (incluye dependencias — `/actuator/health/readiness`)
- **Códigos HTTP:** 200 para UP/OUT_OF_SERVICE, 503 para DOWN
- **Indicadores predefinidos:** PostgreSQL (`db`), Redis, disco — automáticos
- **Personalizados:** Implementa `HealthIndicator` para lógica compleja
- **Kubernetes:** Configura livenessProbe y readinessProbe apuntando a Actuator
- **Monitoreo:** Integra con Prometheus vía Micrometer
