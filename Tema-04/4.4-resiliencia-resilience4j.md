# Módulo 4 · Lección 4.4 — Resiliencia con Resilience4j: retry, timeout y circuit breaker

> **Duración estimada:** 55 min
> **Servicio del proyecto:** `ordering-service` → `catalog-service` (la HTTP Interface de 4.3, ahora resiliente)
> **Requisitos:** Lección 4.3 (HTTP Interface sobre `RestClient`)

---

## 1. Por qué una llamada de red necesita defensas

La llamada Ordering → Catalog de 4.3 funciona... cuando todo va bien. Pero la red **no** va
siempre bien: un *timeout* esporádico, un reinicio de Catalog, una microcaída. Sin defensas,
cualquiera de esos hipos transitorios tumba la creación del pedido.

**Resilience4j** añade políticas de resiliencia alrededor de la llamada HTTP. Las tres
fundamentales:

- **Retry** — reintenta ante fallos *transitorios* (puede que el segundo intento funcione).
- **TimeLimiter / Timeout** — corta una llamada que tarda demasiado (no esperes 30s a un
  servicio colgado).
- **CircuitBreaker** — si Catalog falla repetidamente, **deja de llamarlo** un tiempo, para no
  añadir leña y para fallar rápido.

---

## 2. Las tres políticas, explicadas

### 2.1 Retry: reintentar lo transitorio

Un fallo **transitorio** (timeout puntual, 503 momentáneo) puede resolverse solo. Reintentar
tiene sentido. Un fallo **permanente** (404, 400) no: reintentar es desperdiciar tiempo.

La clave es el **backoff exponencial con jitter**: si reintentas inmediatamente y mil clientes
hacen lo mismo, machacas a Catalog justo cuando se recupera (*retry storm*). Esperar 1s, 2s,
4s... con algo de aleatoriedad reparte la presión.

### 2.2 Timeout: no esperar indefinidamente

Hay dos niveles: el *timeout* por **intento** (esta llamada concreta no puede tardar más de 2s)
y el total. Sin timeout, una llamada colgada retiene un hilo y, en cadena, propaga la lentitud
aguas arriba.

### 2.3 Circuit breaker: dejar de insistir

Si Catalog ha fallado, digamos, el 50% de un volumen mínimo, asumimos que está mal y **abrimos
el circuito**: durante un periodo, las llamadas fallan *inmediatamente* sin tocar la red. Tras
ese periodo, el circuito pasa a *half-open* y deja pasar unas llamadas de prueba: si van bien,
se cierra; si no, sigue abierto. Esto **da respiro al servicio caído** y evita que Ordering
malgaste recursos esperando.

```
   cerrado ──(supera el umbral de fallos)──► abierto
      ▲                                        │
      │                                  (pasa el tiempo)
      │                                        ▼
      └──(prueba OK)──────── half-open ◄───────┘
                                │
                           (prueba KO)
                                ▼
                             abierto
```

## 3. Dos formas de aplicarlo

Resilience4j permite envolver una llamada de dos maneras. Las veremos ambas:

1. **El decorador explícito** — para que **veas el mecanismo** pieza a pieza.
2. **Las anotaciones + configuración en `application.yml`** — el atajo recomendado para
   producción con Spring Boot.

Concepto primero, atajo después.

### 3.1 El decorador explícito (para entender)

Componemos las tres políticas alrededor del `CatalogApi` de 4.3:

```java
package com.shopmicro.ordering.infrastructure.catalog;

import io.github.resilience4j.circuitbreaker.CircuitBreaker;
import io.github.resilience4j.circuitbreaker.CircuitBreakerConfig;
import io.github.resilience4j.retry.Retry;
import io.github.resilience4j.retry.RetryConfig;
import io.github.resilience4j.timelimiter.TimeLimiter;
import io.github.resilience4j.timelimiter.TimeLimiterConfig;
import io.github.resilience4j.core.IntervalFunction;

import java.time.Duration;

public class ResilientCatalogClient implements CatalogClient {

    private final CatalogApi api;
    private final Retry retry;
    private final CircuitBreaker circuitBreaker;

    public ResilientCatalogClient(CatalogApi api) {
        this.api = api;

        // 1. Retry ante fallos transitorios, con backoff exponencial + jitter.
        var retryConfig = RetryConfig.custom()
                .maxAttempts(3)
                .intervalFunction(IntervalFunction.ofExponentialRandomBackoff(
                        Duration.ofSeconds(1), 2.0))
                .build();
        this.retry = Retry.of("catalog", retryConfig);

        // 2. Circuit breaker: si falla el 50% de un volumen mínimo, abre 15s.
        var cbConfig = CircuitBreakerConfig.custom()
                .failureRateThreshold(50)              // 50 %
                .slidingWindowSize(10)                 // volumen mínimo
                .waitDurationInOpenState(Duration.ofSeconds(15))
                .build();
        this.circuitBreaker = CircuitBreaker.of("catalog", cbConfig);
    }

    @Override
    public ProductInfo getProduct(int productId) {
        // El orden de composición importa (ver nota): retry envuelve breaker.
        var decorated = Retry.decorateSupplier(retry,
                CircuitBreaker.decorateSupplier(circuitBreaker,
                        () -> api.getProduct(productId)));
        return decorated.get();
    }
}
```

> **El timeout por intento** se aplica con un `TimeLimiter` cuando la llamada es asíncrona
> (`Supplier<CompletableFuture<...>>`), o directamente como *read timeout* del `RestClient` de
> 4.2 para llamadas bloqueantes. En el decorador explícito lo más simple es fijar el timeout en
> el `RestClient`; con anotaciones (3.2) se gestiona con `@TimeLimiter`.

> **El orden importa.** Las estrategias se ejecutan como capas de cebolla, de fuera adentro.
> Retry **envuelve** al circuit breaker, que **envuelve** a la llamada (con su timeout). Así: el
> timeout corta el *intento* individual; ese fallo lo cuenta el breaker; y el retry decide si
> reintentar. Si inviertes el orden, cambias la semántica —es un punto que conviene que los
> alumnos razonen, no que memoricen.

### 3.2 El atajo de producción: anotaciones + `application.yml`

Con `resilience4j-spring-boot3` declaras las políticas en configuración y las aplicas con
anotaciones. Es lo que usarías en el día a día:

```java
package com.shopmicro.ordering.infrastructure.catalog;

import io.github.resilience4j.circuitbreaker.annotation.CircuitBreaker;
import io.github.resilience4j.retry.annotation.Retry;
import org.springframework.stereotype.Component;

@Component
public class CatalogClientAdapter implements CatalogClient {

    private final CatalogApi api;

    public CatalogClientAdapter(CatalogApi api) {
        this.api = api;
    }

    @Override
    @Retry(name = "catalog")
    @CircuitBreaker(name = "catalog", fallbackMethod = "fallback")
    public ProductInfo getProduct(int productId) {
        return api.getProduct(productId);
    }

    private ProductInfo fallback(int productId, Throwable ex) {
        // Degradación controlada cuando el circuito está abierto o se agotan los reintentos.
        throw new CatalogUnavailableException(productId, ex);
    }
}
```

```yaml
# application.yml de ordering-service
resilience4j:
  retry:
    instances:
      catalog:
        max-attempts: 3
        wait-duration: 1s
        enable-exponential-backoff: true
        exponential-backoff-multiplier: 2
        randomized-wait-factor: 0.5          # jitter
  circuitbreaker:
    instances:
      catalog:
        failure-rate-threshold: 50
        sliding-window-size: 10
        wait-duration-in-open-state: 15s
        permitted-number-of-calls-in-half-open-state: 3
  timelimiter:
    instances:
      catalog:
        timeout-duration: 2s                 # timeout por intento
```

> **Cuál enseñar en clase.** Empieza por el decorador explícito (3.1) para que los alumnos
> *vean* las tres estrategias y su orden. Cierra con las anotaciones + `application.yml` (3.2)
> como "esto es lo que de verdad pondrás en producción, y ahora entiendes qué hace por dentro".

## 4. La pieza que Resilience4j NO arregla: idempotencia

El **retry** reenvía la petición. Para una **lectura** (`GET /products/42`) eso es inofensivo:
pedir dos veces lo mismo da el mismo resultado. Pero si reintentaras una **escritura** no
idempotente (crear algo dos veces), el retry podría **duplicar** el efecto.

Por eso reintentar es seguro casi siempre en GET, y en operaciones de escritura hay que diseñar
la idempotencia (claves de idempotencia, *upserts*). Es el mismo principio que veremos con el
consumidor del Outbox en 4.8: **at-least-once exige idempotencia**. Aquí, nuestro caso es un
GET, así que reintentar es seguro de fábrica.

---

## 5. El resultado de la progresión 4.2 → 4.3 → 4.4

La misma llamada Ordering → Catalog ha pasado por tres estados:

```
4.2  RestClient        →  funciona, con boilerplate, frágil
4.3  + HTTP Interface  →  sin boilerplate (interface declarativa)
4.4  + Resilience4j    →  resiliente (retry, timeout, circuit breaker)
```

Y el handler de `CreateOrder` **sigue sin enterarse**: pide `CatalogClient` y ya está. Toda la
evolución ocurrió en la configuración y la infraestructura, nunca en la lógica de negocio.

---

## Puntos clave

- La red falla de forma **transitoria**; sin defensas, un hipo tumba la operación. Resilience4j
  pone esas defensas alrededor de la llamada.
- **Retry** (con *backoff exponencial + jitter* para evitar *retry storms*), **timeout** (no
  esperar a un servicio colgado) y **circuit breaker** (dejar de llamar a lo que está roto,
  fallar rápido, dar respiro).
- Se engancha sobre el **`RestClient` / HTTP Interface** (4.3): la resiliencia envuelve el mismo
  cliente, sin tocar al consumidor.
- **El orden de las estrategias cambia la semántica**: retry envuelve breaker envuelve timeout.
  Razonarlo, no memorizarlo.
- Empieza por el **decorador explícito** (ves el mecanismo) y cierra con **anotaciones +
  `application.yml`** (lo de producción).
- **Retry solo es seguro si la operación es idempotente.** En GET lo es; en escrituras, hay que
  diseñarlo. Mismo principio que el at-least-once del Outbox.
