# Módulo 4 · Lección 4.9 — API Gateway con Spring Cloud Gateway

> **Duración estimada:** 50 min
> **Servicio del proyecto:** `gateway` (el punto de entrada único de ShopMicro)
> **Requisitos:** Lecciones 4.1–4.5; el detalle de JWT / OIDC es del Módulo 5 (aquí se referencia)

---

## 1. El problema: ¿con quién habla el cliente?

ShopMicro tiene varios servicios: Catalog, Basket, Ordering, Identity. Sin un punto de entrada
único, el frontend tendría que conocer la URL de cada uno, gestionar CORS contra todos y lidiar
con que las direcciones internas cambien. Acoplas al cliente a la **topología interna** del
sistema.

Un **API Gateway** pone una sola puerta delante de toda la malla. El cliente habla **solo con el
Gateway**, que enruta cada petición al servicio que corresponda. La topología interna queda
**oculta** y libre para cambiar.

```
                       ┌─────────────► catalog-service
   Cliente ──► Gateway ─┼─────────────► basket-service
                       ├─────────────► ordering-service
                       └─────────────► identity-service
```

---

## 2. Qué es Spring Cloud Gateway

**Spring Cloud Gateway** es el reverse proxy / API Gateway del ecosistema Spring, construido sobre
Spring WebFlux (reactivo, no bloqueante). No es un producto aparte: es una aplicación Spring Boot
con la dependencia `spring-cloud-starter-gateway`, configurable por `application.yml` o por código,
y extensible con **filtros** (globales y por ruta).

Responsabilidades que centraliza el Gateway:

- **Enrutado** de cada ruta pública al servicio interno correcto.
- **Cross-cutting de borde**: CORS, rate limiting, logging y correlación de peticiones.
- **Aislamiento**: el cliente no conoce ni alcanza directamente los servicios internos.

Lo que el Gateway **NO** es —y esto es el corazón de la lección— se trata en el punto 5.

---

## 3. Enrutado: routes, predicates y filters

Spring Cloud Gateway se basa en **routes**: cada route tiene un **predicate** (qué petición casa:
path, host, método, cabeceras) y unos **filters** (qué transformación se aplica antes de reenviar)
y un **uri** de destino (el servicio interno).

```yaml
# application.yml del gateway
spring:
  cloud:
    gateway:
      routes:
        - id: catalog-route
          uri: http://catalog-service:8080
          predicates:
            - Path=/catalog/**
          filters:
            - StripPrefix=1          # quita el primer segmento (/catalog)
        - id: ordering-route
          uri: http://ordering-service:8080
          predicates:
            - Path=/orders/**
          filters:
            - StripPrefix=1          # quita /orders
```

Una petición a `/catalog/products/42` casa con `catalog-route`, se le **quita el prefijo**
`/catalog` (`StripPrefix=1`) y se reenvía a `http://catalog-service:8080/products/42`. El cliente
externo ve `/catalog/...`; el servicio interno recibe su ruta nativa `/products/...`.

> `catalog-service:8080` es un **nombre de servicio** (Docker Compose / Kubernetes del Módulo 7),
> no una IP ni `localhost`. Mover dónde vive Catalog es cambiar esta config, no el cliente.

### Puesta en marcha por código (alternativa al YAML)

```java
package com.shopmicro.gateway;

import org.springframework.cloud.gateway.route.RouteLocator;
import org.springframework.cloud.gateway.route.builder.RouteLocatorBuilder;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class GatewayRoutes {

    @Bean
    RouteLocator routes(RouteLocatorBuilder builder) {
        return builder.routes()
                .route("catalog-route", r -> r.path("/catalog/**")
                        .filters(f -> f.stripPrefix(1))
                        .uri("http://catalog-service:8080"))
                .route("ordering-route", r -> r.path("/orders/**")
                        .filters(f -> f.stripPrefix(1))
                        .uri("http://ordering-service:8080"))
                .build();
    }
}
```

La clase principal es una aplicación Spring Boot normal:

```java
@SpringBootApplication
public class GatewayApplication {
    public static void main(String[] args) {
        SpringApplication.run(GatewayApplication.class, args);
    }
}
```

## 4. La identidad en ShopMicro: quién valida qué

Esta es la sección que decide la arquitectura de seguridad de toda la malla. La pregunta que la
gobierna es simple: **cuando una petición llega a `ordering-service`, ¿cómo sabe ese servicio
quién la está haciendo?** La respuesta marca dos reglas que no se negocian.

Partimos de un hecho fijo: **existe un IdP** (el Identity de ShopMicro) que es la **única autoridad
de identidad**. Lo que el cliente presenta en el borde puede variar (lo vemos en 4.3), pero estas
dos reglas no cambian nunca.

### 4.1 Dos responsabilidades, dos dueños

| Responsabilidad | Dueño | Por qué |
|---|---|---|
| **Validar / resolver la credencial que trae el cliente** (cookie o token opaco) | **IdP (Identity)** | Es quien la emitió; única autoridad de identidad. Nadie más tiene un store de sesión paralelo. |
| **Validar el JWT y establecer el `principal`** | **Cada microservicio** | Solo así el micro sabe quién pide y puede autorizar. Defensa en profundidad. |

El **Gateway no valida identidad**. No es una segunda autoridad ni un perímetro confiable. Enruta
y, cuando la credencial del cliente es opaca (cookie o token de referencia), le pide al **IdP** que
la resuelva —porque el IdP es su emisor. Tener un store de sesión en el Gateway sería **duplicar la
autoridad de identidad** y partir la verdad en dos sitios.

### 4.2 El invariante interno: hacia dentro, siempre JWT contra JWKS

Sea cual sea la credencial que el cliente presenta en el borde, **de la entrada de la malla hacia
dentro la moneda es siempre un JWT efímero que cada micro valida offline contra JWKS**. Esa parte
no cambia entre escenarios:

```
                        ┌──────────────── frontera de la malla ────────────────┐
   Cliente ──cred.──► Gateway ──JWT efímero──► ordering-service ──JWT──► catalog-service
            (varía)      │                         │                       │
                         │                  valida vs JWKS           valida vs JWKS
                         │                  fija principal           fija principal
                         └── si la cred. es opaca, la resuelve el IdP
```

> **Por qué JWKS y no preguntar al IdP en cada salto.** El IdP publica su clave pública en un
> endpoint JWKS. Cada micro verifica la **firma** del JWT por su cuenta —sin llamar al IdP— y de los
> claims construye el `principal`. Validación **distribuida**, sin convertir al IdP en un cuello de
> botella síncrono en el camino crítico (el acoplamiento temporal de 4.1). El IdP interviene al
> emitir/refrescar el token y, si la credencial de borde es opaca, al resolverla; **nunca** en cada
> validación interna.

Lo único que cambia de un despliegue a otro es **qué presenta el cliente en el borde**. Hay tres
escenarios, todos legítimos, y conviene que el alumno sepa elegir.

### 4.3 Tres escenarios de credencial de cliente

Asumimos siempre un IdP detrás. La elección depende de **qué tipo de cliente** consume la API
(navegador, app nativa, servicio) y de cuánto te importan exposición y revocación.

| | **1. Cookie opaca** | **2. Token opaco (Bearer)** | **3. JWT al cliente** |
|---|---|---|---|
| **Qué sostiene el cliente** | cookie `HttpOnly` de referencia | id de referencia en `Authorization` | el JWT completo (exp ~1h) |
| **Cliente típico** | navegador / SPA con BFF | app nativa, cliente no-browser | SPA, móvil, servicio |
| **Resolución en el borde** | el Gateway la resuelve contra el IdP | el Gateway la introspecta contra el IdP | ninguna: ya es JWT |
| **Round-trip al IdP por petición** | sí (o cacheado) | sí (o cacheado) | no |
| **Exposición de la identidad** | mínima (no sale del navegador) | baja (referencia, no claims) | el cliente ve los claims |
| **Revocación** | inmediata (invalida la referencia) | inmediata | hasta `exp` (mitiga: exp corta) |
| **Refresh** | lo gestiona el borde/BFF | lo gestiona el borde | **el cliente** refresca al caducar |

**1. Cookie opaca.** La credencial es una referencia en cookie `HttpOnly`+`Secure`; el navegador
**nunca** ve el JWT (no robable por XSS). Es el patrón **BFF / Token Handler**, hoy recomendado para
SPAs y webs. El Gateway resuelve la cookie contra el IdP, que devuelve el JWT efímero que viaja
hacia dentro. Máxima protección y revocación; cuesta un round-trip de resolución en el borde.

**2. Token opaco con `Authorization`.** Igual de opaco, pero sin cookie: un *reference token* en la
cabecera, pensado para clientes **no-browser** (apps, integraciones) donde la cookie no aplica. El
Gateway lo resuelve por **introspection** (`/introspect` del IdP) y obtiene el JWT interno. Conserva
la revocación inmediata del opaco; igual coste de resolución.

**3. JWT directo al cliente (exp ~1h, refresh).** El IdP emite el **JWT al propio cliente**, que lo
manda en `Authorization: Bearer`. El Gateway no resuelve nada: ya es la moneda interna, solo
reenvía (y opcionalmente valida, 4.4). Lo más simple y sin round-trips, pero el cliente **ve los
claims** y el token **no se revoca antes de `exp`** —se mitiga con expiración corta y un **refresh
token** que el cliente usa para renovar cuando caduca.

> **El denominador común.** En los tres, hacia dentro circula un **JWT validado contra JWKS por
> cada micro** (4.2). Los escenarios solo difieren en la **cara cliente↔borde** y en quién
> sostiene/refresca la credencial. Elige por el tipo de cliente y por tu postura ante exposición
> vs. revocación; no por moda.

### 4.4 ¿Y el Gateway, valida el JWT o no? — los dos modelos

Una vez la moneda interna es el JWT (resuelto desde una credencial opaca, o presentado directamente
en el escenario 3), el Gateway puede comportarse de dos maneras. Ninguna exime al micro de validar;
cambia solo si el **borde** añade una comprobación redundante.

| | **A) Gateway valida (fail-fast)** | **B) Gateway solo reenvía** |
|---|---|---|
| **Qué hace el borde** | revalida el JWT antes de reenviar | pasa el JWT sin mirarlo |
| **Pro** | rechaza basura en el borde; no malgasta saltos internos con peticiones sin token o caducadas | Gateway más simple y delgado; una sola lógica de auth (la del micro) |
| **Contra** | duplica configuración de validación (JWKS, audiencias) también en el borde | una petición con token inválido viaja hasta el micro antes de ser rechazada |
| **Lo que NO cambia** | el micro **igual** valida y fija el principal | el micro **igual** valida y fija el principal |

> **La regla por encima de los dos modelos.** Elijas A o B, **dentro de la malla cada microservicio
> valida el JWT contra JWKS y establece su propio `principal`**. La validación del Gateway es, como
> mucho, una optimización de fail-fast en el borde — **nunca** un sustituto. Si tuvieras que quedarte
> con una sola, la del micro es la innegociable: es la única que garantiza que el servicio sabe quién
> pide aunque alguien alcance la red interna por otra vía.

### 4.5 Esbozo de configuración (el detalle es del Módulo 5)

Cada microservicio se configura como **Resource Server** de Spring Security, que valida el JWT
contra JWKS de forma offline:

```yaml
# application.yml de cada microservicio (p.ej. ordering-service) — NO solo del Gateway
spring:
  security:
    oauth2:
      resourceserver:
        jwt:
          issuer-uri: https://identity-service        # descubre el JWKS automáticamente
          # audiences: ordering
```

```java
// SecurityConfig de cada microservicio — NO solo del Gateway
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(auth -> auth.anyRequest().authenticated())
            // validación offline: firma (JWKS), issuer, audience, expiración
            .oauth2ResourceServer(oauth2 -> oauth2.jwt(Customizer.withDefaults()));
        return http.build();
    }
}
```

Lo esencial: este bloque vive en **cada servicio**, no solo en el Gateway. Eso es lo que materializa
"cada micro establece su principal". El detalle de JWT, roles y políticas es el Módulo 5.

## 5. Qué NO es el Gateway

- **No es autoridad de identidad.** No resuelve por su cuenta la credencial opaca (lo hace el IdP)
  ni es la fuente de verdad de la sesión. No tiene store de sesión propio.
- **No es perímetro confiable.** Que una petición pase el borde no autoriza a los micros a confiar a
  ciegas: cada uno revalida el JWT.
- **No es orquestador de negocio.** Enruta y aplica cross-cutting; no compone respuestas de varios
  servicios con lógica propia (eso sería un *BFF/aggregator*, otra pieza).
- **No sustituye la comunicación interna** (4.2–4.5) ni **es el broker**: lo asíncrono sigue yendo
  por RabbitMQ. El Gateway es petición/respuesta síncrona de fuera hacia dentro.

---

## Puntos clave

- El **API Gateway** es el **punto de entrada único**: oculta la topología interna; el cliente solo
  habla con él.
- **Spring Cloud Gateway** = API Gateway de Spring; **routes** + **predicates** + **filters**, por
  `application.yml` o código. Las direcciones son **nombres de servicio**, no IPs.
- **Tres escenarios cliente↔borde** (siempre con un IdP detrás): **cookie opaca** (BFF, navegador),
  **token opaco Bearer** (clientes no-browser) y **JWT directo al cliente** (exp ~1h + refresh).
  Difieren solo en la cara de borde.
- **Invariante interno:** sea cual sea el escenario, hacia dentro circula un **JWT efímero que cada
  micro valida offline contra JWKS** (Spring Security Resource Server) y con el que fija su
  `principal`.
- **Resolver la credencial opaca es del IdP** (su emisor, única autoridad). **Validar el JWT y fijar
  el `principal` es de CADA microservicio**. El Gateway no es autoridad de identidad.
- El Gateway **puede** revalidar el JWT como *fail-fast* (modelo A) o solo reenviarlo (modelo B): es
  optimización de borde, **nunca** sustituye la validación del micro.
- El Gateway **no es** autoridad de identidad, ni perímetro confiable, ni orquestador, ni broker.
