# 1.4 Conceptos clave: bounded context, acoplamiento y cohesión

> **Módulo 1 — Introducción a los Microservicios**
> Duración estimada: ~40 min

---

Estos tres conceptos son la **brújula para decidir dónde poner los límites** entre microservicios. Equivocarte aquí es el error más caro de todos, porque mover un límite mal puesto entre dos servicios es muchísimo más costoso que mover una clase de sitio dentro de un monolito.

---

## Acoplamiento y cohesión: las dos fuerzas

Son dos caras de la misma moneda y el objetivo de todo buen diseño:

> **Maximizar la cohesión, minimizar el acoplamiento.**

### Cohesión
Mide **cuánto tienen que ver entre sí las cosas que están juntas**. Alta cohesión = todo lo que cambia por la misma razón vive en el mismo sitio.

> La frase de Sam Newman: *"Reúne el código que cambia por la misma razón; separa el que cambia por razones distintas."*

Si cada vez que cambias las reglas de precios tienes que tocar tres servicios, esos tres servicios tienen baja cohesión respecto a "precios": el concepto está disperso donde debería estar junto.

### Acoplamiento
Mide **cuánto depende un componente de otro**. Bajo acoplamiento = puedes cambiar un servicio sin tener que cambiar los demás.

En microservicios distinguimos varios tipos, de mejor a peor:

| Tipo de acoplamiento | Descripción | ¿Deseable? |
|---|---|---|
| **De dominio** | A necesita una *capacidad* de B y se la pide por su API pública | ✅ Inevitable y sano |
| **De paso (pass-through)** | A pasa datos a B solo porque C los necesita | ⚠️ Sospechoso |
| **Común** | Varios servicios comparten datos mutables (¡una BD compartida!) | ❌ Evítalo |
| **De contenido** | A mete mano en las tripas internas de B (su BD, su estado) | ❌❌ El peor; rompe la autonomía |

El **acoplamiento de contenido** —que un servicio acceda a la base de datos de otro— es el pecado capital de los microservicios. Destruye la independencia: ya no puedes cambiar el esquema de B sin romper A.

## Bounded Context (contexto delimitado)

Viene del **Domain-Driven Design** (DDD) de Eric Evans, y es **la** herramienta conceptual para trazar límites de servicios.

Un *bounded context* es una frontera dentro de la cual un modelo y su lenguaje tienen un significado preciso y consistente. La idea central:

> **La misma palabra significa cosas distintas en partes distintas del negocio.**

### El ejemplo clásico: "Cliente" / "Producto"

La palabra **"Producto"** no significa lo mismo en todas partes de ShopMicro:

- En **Catalog**, un Producto es: nombre, descripción, categoría, fotos, especificaciones, precio de venta.
- En **Basket**, un Producto es: un id, una cantidad y el precio en el momento de añadirlo. Nada más.
- En **Ordering**, un Producto es: id, cantidad, precio pagado (congelado) y estado de envío.

Intentar crear **un único modelo "Producto" que sirva para todos** lleva a un objeto monstruoso, acoplado a todo y que nadie puede cambiar sin romper algo. Esto se llama, peyorativamente, *God Object* o modelo canónico.

La solución del DDD: **cada bounded context tiene su propio modelo de Producto**, ajustado a lo que ese contexto necesita. Y esos límites de contexto son justamente **buenos candidatos a límites de microservicio**.

```
   CATALOG context          BASKET context         ORDERING context
┌──────────────────┐     ┌────────────────┐     ┌──────────────────┐
│ Producto         │     │ ItemCarrito    │     │ LineaPedido      │
│  - nombre        │     │  - productoId  │     │  - productoId    │
│  - descripción   │     │  - cantidad    │     │  - cantidad      │
│  - categoría     │     │  - precioActual│     │  - precioPagado  │
│  - fotos[]       │     └────────────────┘     │  - estadoEnvío   │
│  - precio        │                            └──────────────────┘
└──────────────────┘
  Tres modelos distintos para el "mismo" concepto. Y está BIEN.
```

## El lenguaje ubicuo (ubiquitous language)

Otro concepto del DDD muy práctico: dentro de cada *bounded context*, desarrolladores y expertos de negocio usan **exactamente el mismo vocabulario**, y ese vocabulario se refleja literalmente en el código (clases, métodos, endpoints).

Si el negocio dice "una orden", el código tiene una clase `Orden`, no `TransaccionComercial`. Esto reduce la traducción mental —y los errores— entre lo que el negocio quiere y lo que el código hace.

## Heurística práctica para trazar límites

Cuando dudes dónde cortar, pregúntate:

1. **¿Cambian por la misma razón?** Si sí → probablemente el mismo servicio (cohesión).
2. **¿Pertenecen al mismo bounded context?** Si el mismo término significa lo mismo → mismo servicio.
3. **¿Cuántas llamadas de red cruzarían el límite?** Si partir A y B genera un ida y vuelta constante entre ellos, quizá deberían estar juntos (acoplamiento alto → no los separes).
4. **¿Pueden evolucionar y desplegarse de forma independiente?** Si no → el límite está mal.

> Y la recomendación de oro: **ante la duda, servicios más grandes**. Es mucho más fácil partir un servicio que resultó demasiado grande que fusionar dos que nunca debieron separarse.

---

## Puntos clave

- Objetivo de diseño: **alta cohesión, bajo acoplamiento**.
- Junta lo que cambia por la misma razón; separa lo que cambia por razones distintas.
- El **acoplamiento de contenido** (acceder a la BD de otro servicio) es el peor pecado.
- Un **bounded context** delimita dónde un modelo y su lenguaje son consistentes — y es el mejor candidato a límite de microservicio.
- El mismo concepto ("Producto") legítimamente tiene **modelos distintos** en contextos distintos.
- Ante la duda, **servicios más grandes**: partir es más barato que fusionar.
