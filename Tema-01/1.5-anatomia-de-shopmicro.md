# 1.5 Anatomía de ShopMicro (visión global del proyecto del curso)

> **Módulo 1 — Introducción a los Microservicios**
> Duración estimada: ~35 min

---

## Qué vamos a construir

**ShopMicro** es un e-commerce mínimo pero realista que iremos construyendo módulo a módulo durante todo el curso. No es un *hola mundo*: incluye persistencia real, comunicación entre servicios, seguridad, mensajería asíncrona, contenedores y observabilidad. Es deliberadamente pequeño en alcance funcional para que la complejidad la aporte la **arquitectura**, que es lo que estamos aprendiendo.

---

## Los servicios

| Servicio | Responsabilidad | Tecnología destacada | Aparece en |
|---|---|---|---|
| **catalog-api** | Catálogo de productos: listar, buscar, detalle | Spring Data JPA + PostgreSQL | Módulos 3, 6 |
| **basket-api** | Carrito de la compra del usuario | Spring Data Redis | Módulos 4, 6 |
| **ordering-api** | Crear y gestionar pedidos | Spring Data JPA + mensajería | Módulos 4, 6 |
| **identity** | Autenticación y emisión de tokens | Spring Security (JWT / OIDC) | Módulo 5 |
| **gateway** | Punto de entrada único para clientes | Spring Cloud Gateway | Módulo 4 |

Cada uno es un **bounded context** (recuerda el punto 1.4): el "Producto" de Catalog no es el mismo que el de Basket ni el de Ordering.

## Cómo se comunican (anticipo del Módulo 4)

- **Cliente → Gateway:** todo el tráfico externo entra por el Gateway (Spring Cloud Gateway), que enruta a cada servicio. El cliente nunca llama directamente a los servicios internos.
- **Ordering → Catalog (síncrono):** al crear un pedido, Ordering consulta precios actuales a Catalog. Lo haremos primero con `RestClient`/`WebClient`, luego con **OpenFeign** (cliente declarativo), y le añadiremos resiliencia con **Resilience4j**. También veremos **gRPC** como alternativa de alto rendimiento.
- **Ordering → resto (asíncrono):** cuando se crea un pedido, Ordering **publica un evento** `OrderCreated` en RabbitMQ. Otros servicios reaccionan sin que Ordering sepa quiénes son. Esto es desacoplamiento real.

## Cómo evoluciona a lo largo del curso

El proyecto **crece contigo**. Cada módulo añade una capa:

| Módulo | Qué le añadimos a ShopMicro |
|---|---|
| **1** | El diseño y los límites de los servicios (este módulo) |
| **2** | Las bases de Spring Boot: estructura, DI, configuración, logging |
| **3** | catalog-api completo: REST, validación, errores, OpenAPI, health checks |
| **4** | Comunicación: OpenFeign, Resilience4j, gRPC, RabbitMQ, el Gateway |
| **5** | Seguridad: Identity con JWT, protección de los servicios |
| **6** | Datos: PostgreSQL en Catalog, Redis en Basket, Saga y Outbox |
| **7** | Producción: Docker, Compose, observabilidad, Kubernetes |

Al final del curso tendrás un sistema que arranca entero con un `docker compose up`, donde puedes seguir una petición de punta a punta en Jaeger y que está listo (conceptualmente) para Kubernetes.

## Una nota de honestidad

Como ya dijimos en el punto 1.2: para un e-commerce real de este tamaño, un **monolito modular** sería probablemente la mejor decisión de ingeniería. Construimos ShopMicro como microservicios porque es un **curso sobre microservicios** y necesitamos un vehículo para aprender cada patrón. Mantén siempre esa perspectiva crítica: estás aprendiendo *cómo* se hace, para poder decidir con criterio *cuándo* hacerlo.

---

## Puntos clave

- ShopMicro tiene **5 servicios**: Catalog, Basket, Ordering, Identity y Gateway.
- Cada servicio es un **bounded context** con su propia base de datos.
- Combina comunicación **síncrona** (OpenFeign/gRPC) y **asíncrona** (RabbitMQ).
- El proyecto **evoluciona módulo a módulo**: cada tema añade una capa real.
- El objetivo final: un sistema completo, contenerizado, observable y desplegable.
