# 6.5 — Redis como almacén del Basket.API (en memoria)

> **Módulo 6 · Lección 5 de 9**
> **Duración estimada:** ~25 min
> **Tipo:** Código

---

## Otro servicio, otro estilo de almacén

El `Basket.API` (carrito de la compra) tiene un patrón de acceso muy distinto al de Catalog:

- Se lee y escribe **el carrito entero de un usuario**, identificado por su id.
- No hay consultas complejas ni relaciones: es un **clave → valor**.
- Los datos son **volátiles** y de vida corta (un carrito caduca).

Ese perfil es justo para lo que sirve **Redis**: un almacén clave-valor en memoria, rapidísimo.
En el curso "real" el Basket vive en Redis. **En este módulo lo sustituimos por un mapa en
memoria**, por la misma razón que usamos H2 en Catalog: queremos enseñar el **diseño**, no
aprovisionar un Redis.

Lo interesante: el **mismo principio ISP** del núcleo aplica aquí, aunque el almacén no sea JPA.

## El dominio del carrito

```java
package com.shopmicro.basket.domain;

import java.math.BigDecimal;
import java.util.ArrayList;
import java.util.Collections;
import java.util.List;
import java.util.Optional;
import java.util.UUID;

public class Basket {

    private final UUID customerId;
    private final List<BasketItem> items = new ArrayList<>();

    public Basket(UUID customerId) {
        this.customerId = customerId;
    }

    public UUID getCustomerId()        { return customerId; }
    public List<BasketItem> getItems() { return Collections.unmodifiableList(items); }

    public void addItem(UUID productId, String name, BigDecimal price, int quantity) {
        Optional<BasketItem> existing = items.stream()
            .filter(i -> i.getProductId().equals(productId))
            .findFirst();

        if (existing.isPresent()) {
            existing.get().increaseQuantity(quantity);
        } else {
            items.add(new BasketItem(productId, name, price, quantity));
        }
    }

    public BigDecimal total() {
        return items.stream()
            .map(i -> i.getPrice().multiply(BigDecimal.valueOf(i.getQuantity())))
            .reduce(BigDecimal.ZERO, BigDecimal::add);
    }
}
```

    public String getName()     { return name; }

        this.quantity += by;
    }

## El puerto: ISP también aquí

El carrito solo necesita dos operaciones: **leer** el carrito de un cliente y **guardarlo**
(crear o reemplazar). Definimos un puerto pequeño y honesto:

```java
package com.shopmicro.basket.persistence;

import com.shopmicro.basket.domain.Basket;

import java.util.UUID;

/** Lee el carrito de un cliente. Devuelve un carrito vacío si no existe. */
public interface IGetBasket {
    Basket get(UUID customerId);
}
```

```java
package com.shopmicro.basket.persistence;

import com.shopmicro.basket.domain.Basket;

/** Crea o reemplaza el carrito de un cliente (upsert). */
public interface ISaveBasket {
    void save(Basket basket);
}
```

> **Nota de diseño.** Aquí `get` **no lanza** si no existe: devuelve un carrito vacío. Es una regla
> de negocio distinta a la de Catalog (donde leer una pizza inexistente es un 404). El carrito "no
> creado todavía" es un estado **normal**, no un error. Mismo principio (puerto mínimo), distinta
> semántica según el dominio — y eso es correcto.

## La implementación en memoria

Un `ConcurrentHashMap` hace de "Redis". Implementa los dos puertos:

```java
package com.shopmicro.basket.persistence;

import com.shopmicro.basket.domain.Basket;
import org.springframework.stereotype.Component;

import java.util.UUID;
import java.util.concurrent.ConcurrentHashMap;

/**
 * Almacén del carrito en memoria. Sustituye a Redis en el aula. La interfaz pública
 * (IGetBasket, ISaveBasket) es idéntica a la que tendría una implementación Redis;
 * cambiar a Redis solo afecta a esta clase, no a quien la consume.
 */
@Component
public class InMemoryBasketStore implements IGetBasket, ISaveBasket {

    private final ConcurrentHashMap<UUID, Basket> store = new ConcurrentHashMap<>();

    @Override
    public Basket get(UUID customerId) {
        return store.getOrDefault(customerId, new Basket(customerId));   // carrito vacío si no existe
    }

    @Override
    public void save(Basket basket) {
        store.put(basket.getCustomerId(), basket);   // upsert
    }
}
```

Registro: basta `@Component` sobre `InMemoryBasketStore`. Como implementa los dos puertos, inyectar
`IGetBasket` o `ISaveBasket` resuelve al **mismo bean**.

> El bean es **singleton** (el ámbito por defecto de Spring), porque el mapa **es** el almacén: debe
> sobrevivir entre peticiones. Con Redis la conexión se gestionaría aparte, pero los **puertos** que
> ve la aplicación no cambian.

## El día que quieras Redis de verdad

Solo cambia la implementación del puerto. Esquemáticamente, con **Spring Data Redis**
(`RedisTemplate`):

```java
// Pseudocódigo: implementación Redis del MISMO puerto
@Component
public class RedisBasketStore implements IGetBasket, ISaveBasket {

    private final RedisTemplate<String, Basket> redis;   // Spring Data Redis

    public RedisBasketStore(RedisTemplate<String, Basket> redis) {
        this.redis = redis;
    }

    @Override
    public Basket get(UUID customerId) {
        Basket found = redis.opsForValue().get(customerId.toString());
        return found != null ? found : new Basket(customerId);
    }

    @Override
    public void save(Basket basket) {
        redis.opsForValue().set(basket.getCustomerId().toString(), basket);
    }
}
```

`IGetBasket` / `ISaveBasket` no cambian. Los handlers que los consumen **no se enteran**. Eso es lo
que nos da depender de **puertos pequeños** en lugar de un almacén concreto.

---

## Puntos clave

- El **mismo principio ISP** aplica fuera de JPA: el Basket expone puertos mínimos
  (`IGetBasket`, `ISaveBasket`), no un repositorio genérico.
- La **semántica del `get` depende del dominio**: en Catalog leer algo inexistente es un 404; en
  Basket es un carrito vacío (estado normal). El diseño se adapta al negocio.
- Sustituir Redis por un `ConcurrentHashMap` afecta **solo a la implementación del puerto**; los
  consumidores no cambian. Migrar a Redis real es reescribir una clase.
